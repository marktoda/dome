Yes. From first principles, I would **not** start from “core/shell,” “tools/hooks,” or even “plugins.” I would start from the irreducible thing you are trying to build:

> A second brain is a durable, human-readable source tree plus an engine that continuously turns that source tree into better structure, better recall, and better next actions.

The cleanest design I can see is:

```text
Git is the communication layer.
Markdown is the source substrate.
The engine is an adoption-and-garden runtime.
Extensions are processors that read snapshots and emit effects.
Core features are just built-in extensions.
```

This is compatible with the current “compiler over markdown vault” framing, but it makes that idea the actual operating contract rather than just the architectural vibe. The uploaded briefing already says the compiler, not any interface, is the load-bearing thing; I would take that all the way.

# 1. The clean-sheet model

The whole system can be expressed with four high-level operators:

```text
Submit
Adopt
Tend
Recall
```

Everything else is implementation detail.

## Submit

Clients create proposed changes.

```text
voice client records a capture
Claude Code edits project notes
Obsidian changes a daily note
mobile app adds a task
garden job proposes cleanup
```

The communication primitive is Git:

```text
commit
branch
pull request
merge request
ref
```

No bespoke synchronization protocol. No required write API. A client that can write markdown and push Git can participate.

## Adopt

The backend decides whether a proposed source state becomes trusted.

Adoption means:

```text
This revision has been parsed, validated, normalized, indexed, and accepted as the latest semantic state.
```

In local eventual mode, adoption advances:

```text
refs/dome/adopted/main
```

In hosted protected mode, adoption merges the PR into:

```text
main
```

Conceptually, those are the same thing. One is local/eventual; the other is hosted/linearized.

## Tend

Async garden jobs improve the vault after adoption.

They can:

```text
extract tasks
link people and projects
generate daily briefs
ask clarifying questions
summarize captures
refresh indexes
create synthesis notes
propose cleanups
```

But they do not secretly mutate state. They produce Git patches or PRs that re-enter the adoption path.

## Recall

Queries and views read the latest adopted semantic state.

```text
What is on the agenda with Danny?
What evidence do I have for Alice’s promo?
What did I decide about the scheduler design?
What are today’s open loops?
```

Recall should default to the adopted state, not the raw latest draft.

# 2. The smallest durable concepts

If I were naming the system from scratch, I would make the public mental model:

```text
Vault
Proposal
Processor
Effect
```

Everything can fit inside those.

## Vault

A Git-backed markdown repository.

It contains:

```text
markdown source
raw captures
generated regions
configuration
extension lock/config
```

It does not need to contain:

```text
graph indexes
vector indexes
LLM run logs
retry counters
external side-effect receipts
```

Those are derived/runtime state.

## Proposal

A proposed change to the vault.

In Git terms, this is:

```text
branch
commit range
pull request
merge request
garden-job branch
```

A human edit, a voice capture, and a background cleanup are all proposals.

That is the big simplification. You do not need different mutation models for humans, shells, plugins, and daemons.

## Processor

A processor is any piece of code that reads a snapshot or proposal and returns effects.

Examples:

```text
markdown linter
wikilink resolver
auto-intake
daily note generator
people agenda extractor
voice capture processor
calendar syncer
codebase doc validator
```

Core features are built-in processors.

Third-party features are installed processors.

Vault-local automations are local processors or config around processors.

## Effect

An effect is what a processor wants the system to do.

The essential effect types are:

```text
Patch
Diagnostic
Fact
Question
Job
ExternalAction
View
```

The engine, not the processor, decides how effects become durable state.

# 3. System services

At the service level, I would keep only five things.

```text
1. Git Source Service
2. Adoption Service
3. Processor Runtime
4. Projection Store
5. Query/View Service
```

## 3.1 Git Source Service

This can be GitHub, Gitea, a bare Git server, a local repo, or a custom hosted Git service.

It owns:

```text
commits
branches
pull requests
merge conflicts
history
rollback
code review
branch protections
```

For multi-client Dome, Git is the bus.

Clients submit by pushing:

```text
refs/heads/client/<device>/<topic>
```

or opening PRs:

```text
client branch → main
```

Garden jobs also submit by pushing:

```text
refs/heads/dome/job/<job-id>
```

or opening PRs.

## 3.2 Adoption Service

This is the semantic gate.

Given a proposal, it:

```text
1. Creates a candidate merge tree.
2. Runs adoption processors.
3. Applies allowed patch effects.
4. Re-runs until stable or blocked.
5. Records diagnostics.
6. Updates projections.
7. Advances the adopted ref or merges the PR.
```

The adoption service is the only thing that blesses a revision as trusted.

## 3.3 Processor Runtime

This runs processors.

There are three processor phases:

```text
adoption
  Bounded, deterministic, merge-blocking.

garden
  Async, possibly LLM-backed, can create proposals.

view
  Read-only, renders answers or UI from adopted state.
```

This avoids a split between “core features” and “plugins.” Auto-intake is just a garden processor. Markdown linting is an adoption processor. Agenda rendering is a view processor.

## 3.4 Projection Store

This is a derived cache.

Use hardened local infrastructure:

```text
SQLite
SQLite FTS
vector index if needed
object storage for large artifacts if needed
```

It stores:

```text
graph facts
search indexes
embeddings
diagnostics
run attempts
question state
schedule cursors
```

But the durable source of knowledge remains Git-backed markdown.

A projection is always keyed by:

```text
adopted commit
extension set hash
processor versions
```

If it is deleted, it can be rebuilt.

## 3.5 Query/View Service

This answers user-facing questions and renders views.

It reads:

```text
adopted Git snapshot
projection store
run/question state
```

It does not mutate source directly.

If a query wants to create durable output, it creates a proposal.

# 4. The communication layer

The write path should be Git-native.

```text
client → branch/commit/PR → adoption service → adopted state
```

The read path can be either Git-native or API-native.

```text
client fetches adopted branch
client calls query API
client renders view from projection service
```

## 4.1 Local eventual mode

For a single-user local vault:

```text
main
  latest raw source

refs/dome/adopted/main
  latest adopted semantic state
```

The source branch may be ahead:

```text
main:                    A --- B --- C
refs/dome/adopted/main:  A
```

After adoption:

```text
main:                    A --- B --- C --- D
refs/dome/adopted/main:                  D
```

Where `D` is an engine closure commit if needed.

## 4.2 Hosted protected mode

For multi-client hosted use:

```text
main
  latest adopted semantic trunk

client branches / PRs
  unadopted proposals
```

Flow:

```text
client branch
  → PR
  → adoption check
  → engine patch commits if needed
  → merge to main
```

In this mode:

```text
main == latest adopted state
```

You may still keep:

```text
refs/dome/adopted/main
```

as an internal cursor, but externally `main` is the trusted branch.

## 4.3 Garden jobs use the same path

A garden job does not get a private mutation channel.

It does this:

```text
processor runs
  → emits Patch effect
  → engine creates branch dome/job/<id>
  → opens PR or auto-merges through adoption
```

Low-risk patches can auto-merge.

High-risk patches become review PRs.

This is crucial. It means the system has one write contract.

# 5. The adoption algorithm

The adoption service should behave like a semantic merge bot.

For each proposal:

```text
1. Build candidate tree.
2. Compile candidate.
3. Run adoption processors.
4. Apply auto patches.
5. Repeat until fixed point.
6. Block, review, or accept.
```

More concretely:

```text
candidate = merge(adopted, proposal)

for iteration in 1..N:
  result = compile(candidate)
  effects = run_adoption_processors(result)

  if effects contain blocking diagnostics:
    block proposal

  patches = allowed_auto_patches(effects)

  if patches empty:
    accept candidate

  candidate = apply_patches(candidate, patches)
```

This fixed-point loop is powerful and simple.

Examples:

```text
linter formats markdown
stable ID inserter adds task IDs
generated region checksum updater refreshes checksum
index maintainer updates generated index
```

Each processor should be idempotent. After it applies its patch once, the next iteration should produce no patch.

If the loop does not converge, adoption blocks with diagnostics.

# 6. Core data types

I would keep the core types small.

## 6.1 Snapshot

An immutable Git tree.

```ts
interface Snapshot {
  commit: CommitOid;
  tree: TreeOid;
}
```

## 6.2 Proposal

A Git change that wants adoption.

```ts
interface Proposal {
  id: string;
  base: CommitOid;
  head: CommitOid;
  source: "client" | "garden" | "manual" | "import";
  metadata?: ProposalMetadata;
}
```

In hosted mode, `Proposal.id` can be the PR number.

In local mode, it can be a commit range.

## 6.3 Document

A parsed markdown file.

```ts
interface Document {
  path: string;
  blob: BlobOid;
  frontmatter: Record<string, unknown>;
  body: string;
  links: LinkSyntax[];
  regions: Region[];
  headings: Heading[];
}
```

Core should parse common markdown structure, but domain semantics should come from processors.

A checkbox is markdown syntax. Whether it is a task belongs to a processor.

## 6.4 SourceRef

A durable pointer to evidence.

```ts
interface SourceRef {
  commit: CommitOid;
  path: string;
  blob?: BlobOid;
  range?: TextRange;
  stableId?: string;
}
```

Use Git identity first. Use embedded stable IDs only where path/range is not enough:

```text
tasks
decisions
claims
people
projects
generated regions
raw captures
```

Do not put IDs on every paragraph.

## 6.5 Processor

```ts
interface Processor {
  id: string;
  version: string;
  phase: "adoption" | "garden" | "view";
  triggers: Trigger[];
  run(ctx: ProcessorContext): Promise<Effect[]>;
}
```

## 6.6 Effect

```ts
type Effect =
  | PatchEffect
  | DiagnosticEffect
  | FactEffect
  | QuestionEffect
  | JobEffect
  | ExternalActionEffect
  | ViewEffect;
```

### PatchEffect

```ts
interface PatchEffect {
  kind: "patch";
  mode: "auto" | "propose";
  patch: UnifiedDiff;
  reason: string;
  sourceRefs: SourceRef[];
}
```

### DiagnosticEffect

```ts
interface DiagnosticEffect {
  kind: "diagnostic";
  severity: "info" | "warning" | "error" | "block";
  code: string;
  message: string;
  sourceRefs: SourceRef[];
}
```

### FactEffect

```ts
interface FactEffect {
  kind: "fact";
  subject: NodeRef;
  predicate: string;
  object: NodeRef | Literal;
  assertion: "explicit" | "extracted" | "inferred" | "generated";
  sourceRefs: SourceRef[];
  confidence?: number;
}
```

### QuestionEffect

```ts
interface QuestionEffect {
  kind: "question";
  question: string;
  options?: string[];
  sourceRefs: SourceRef[];
}
```

### JobEffect

```ts
interface JobEffect {
  kind: "job";
  processorId: string;
  input: unknown;
  runAfter?: string;
  idempotencyKey: string;
}
```

### ExternalActionEffect

For calendar writes, notifications, webhooks, email, etc.

```ts
interface ExternalActionEffect {
  kind: "external";
  capability: string;
  idempotencyKey: string;
  payload: unknown;
  sourceRefs: SourceRef[];
}
```

External effects should go through an outbox ledger, not direct fire-and-forget.

# 7. Operation logging

Use Git for source history. Use a runtime ledger for processor attempts.

## 7.1 Source mutations

Every source mutation is a commit.

Human commit:

```text
alice: add notes from Danny meeting
```

Engine commit:

```text
dome(intake): extract tasks from voice capture

Dome-Processor: dome.intake.extract
Dome-Run: run_01J...
Dome-Input-Commit: abc123
Dome-Source: captures/voice/2026-05-27/foo.md
```

Linter commit:

```text
dome(linter): normalize markdown

Dome-Processor: acme.markdown_linter
Dome-Run: run_01J...
```

Git gives you:

```text
history
diff
rollback
blame
review
merge conflict detection
```

## 7.2 Processor runs

The run ledger records attempts.

```ts
interface RunRecord {
  id: string;
  processorId: string;
  processorVersion: string;
  phase: "adoption" | "garden" | "view";
  inputCommit: CommitOid;
  outputCommit?: CommitOid;
  proposalId?: string;
  status: "queued" | "running" | "succeeded" | "failed" | "skipped";
  effectHashes: string[];
  capabilityUses: CapabilityUse[];
  cost?: CostRecord;
  error?: string;
  startedAt: string;
  finishedAt?: string;
}
```

In hosted mode, this can be backed by:

```text
database rows
CI/check run logs
PR comments
artifacts
```

In local mode, SQLite is enough.

## 7.3 External side effects

External side effects must be idempotent and auditable.

Use an outbox table:

```ts
interface OutboxRecord {
  id: string;
  capability: string;
  idempotencyKey: string;
  payloadHash: string;
  status: "pending" | "sent" | "failed";
  externalId?: string;
  runId: string;
}
```

No extension should directly “send a calendar invite” without the engine recording the attempt.

# 8. Trigger model

Triggers should be derived from adopted source and runtime schedules.

## 8.1 Signals

The compiler produces signals from Git diffs and markdown parsing.

Examples:

```text
file.created
file.modified
document.changed
capture.created
region.changed
task.syntax.added
link.added
frontmatter.changed
```

Processors subscribe to signals.

```ts
interface Trigger {
  signal?: string;
  path?: string;
  schedule?: string;
  command?: string;
}
```

## 8.2 No custom source event queue

For source changes, Git is the queue.

The engine can always ask:

```text
What changed between adopted commit X and candidate commit Y?
```

That produces the signals.

## 8.3 Runtime queue

A runtime queue is still useful for:

```text
scheduled jobs
LLM jobs
external side effects
retry/backoff
long-running async garden work
```

But this queue should not duplicate Git’s role as source history.

# 9. Generated content

Generated content should be treated carefully.

There are two kinds.

## 9.1 Ephemeral views

Most generated content should be ephemeral.

Examples:

```text
agenda with Danny
promo packet draft
project health dashboard
voice review queue
```

These are rendered by view processors from adopted state and projections. They do not need to be committed.

## 9.2 Committed generated artifacts

Commit generated content only when it is useful as shared source.

Examples:

```text
daily note morning region
project index page
human-readable task rollup
curated synthesis note
```

Use owned regions:

```md
<!-- dome:region id="dome.daily.morning_brief" owner="dome.daily" checksum="sha256:..." -->

## Morning brief

- Ask Danny about vendor approval.
- Follow up with Alice about launch metrics.
<!-- /dome:region -->
```

The engine rule:

```text
A processor may repeatedly overwrite only regions or paths it owns.
```

But broad patch permission is separate from ownership.

A linter may patch all markdown files and own nothing.

A daily extension may own `dome.daily.*` generated regions.

# 10. Extension system

The cleanest extension system is code-first and effect-mediated.

## 10.1 What core owns

The core runtime owns:

```text
Git operations
adoption loop
markdown parsing
processor scheduling
effect validation
patch application
capability enforcement
projection store
run ledger
```

It does **not** own:

```text
daily notes
people memory
task semantics
voice intake
calendar semantics
promo packet generation
ideation
codebase docs
```

Those are extensions.

## 10.2 What extensions own

Extensions own domain behavior.

They register processors:

```ts
export default defineExtension({
  id: "dome.intake",
  version: "1.0.0",

  activate(ctx) {
    ctx.processor("detect-capture", {
      phase: "adoption",
      triggers: [{ signal: "file.created", path: "captures/**" }],
      run: detectCapture,
    });

    ctx.processor("extract-capture", {
      phase: "garden",
      triggers: [{ signal: "capture.created" }],
      run: extractCapture,
    });

    ctx.processor("review-queue", {
      phase: "view",
      triggers: [{ command: "dome.intake.review" }],
      run: renderReviewQueue,
    });
  },
});
```

Core features are built the same way.

```text
dome.intake
dome.links
dome.tasks
dome.people
dome.daily
dome.voice
dome.calendar
dome.search
```

There is no special “core feature” mutation path.

## 10.3 Minimal manifest

The manifest is not a behavior DSL.

It should only describe identity, entrypoint, requested capabilities, and assets.

```yaml
id: dome.intake
version: 1.0.0
entry: ./dist/index.js

capabilities:
  read:
    - captures/**
    - inbox/**
    - wiki/**
    - notes/**

  patch:
    auto:
      - wiki/**
      - inbox/processed/**
    propose:
      - "**"

  owns:
    regions:
      - dome.intake.*
    paths:
      - wiki/generated/intake/**

  model:
    invoke: true
    max_daily_cost_usd: 5.00

  graph:
    write:
      - dome.intake.*
      - dome.tasks.*
      - dome.people.*

  network:
    fetch: false
```

The vault or server grants capabilities:

```yaml
extensions:
  dome.intake:
    enabled: true
    grant:
      model.invoke: true
      patch.auto:
        - wiki/**
        - inbox/processed/**
    config:
      confidence_threshold: 0.82
```

## 10.4 Capability tiers

Capabilities should be about effect power, not arbitrary trust labels.

```text
read
  Can inspect source paths.

patch.propose
  Can create reviewable patches.

patch.auto
  Can create patches that may auto-apply.

owns.region / owns.path
  Can repeatedly regenerate owned generated artifacts.

graph.write
  Can emit projection facts in a namespace.

model.invoke
  Can call LLMs through engine budget/policy.

network/calendar/notify
  Can use external services through audited capabilities.

unsafe
  Can use local filesystem/process escape hatches.
```

A trusted linter can have:

```yaml
patch.auto:
  - "**/*.md"
```

That is fine. It still emits a patch. The engine still validates, commits, and can revert it.

## 10.5 Extension robustness rule

Extensions never mutate adopted state directly.

They receive immutable snapshots and return effects.

```text
snapshot in
effects out
```

The engine validates effects.

That is the line that protects the core.

A bad extension can:

```text
fail
produce diagnostics
propose a bad patch
get quarantined
open a bad PR
```

But it should not be able to silently corrupt adopted state.

# 11. How core features become extensions

## 11.1 Auto-intake

Auto-intake is not special.

It is a built-in garden extension.

### Trigger

```text
capture.created
file.created under inbox/**
```

### Inputs

```text
raw capture markdown
recent graph context
user config
```

### Effects

```text
PatchEffect:
  create atomic notes
  move capture to processed
  add tasks
  add backlinks

FactEffect:
  capture mentions person
  capture relates to project
  task derived from capture

QuestionEffect:
  "When you said launch doc, did you mean Project Atlas launch doc?"

DiagnosticEffect:
  capture too ambiguous for auto-commit
```

High-confidence outputs can auto-merge. Low-confidence outputs become a review PR or question.

## 11.2 Daily notes

Built-in extension:

```text
dome.daily
```

Processors:

```text
adoption:
  detect daily-note structure
  validate generated regions

garden:
  morning brief
  evening summary
  task rollover

view:
  today
  week review
```

Daily note generated regions are owned by `dome.daily`.

## 11.3 People memory

Built-in extension:

```text
dome.people
```

Processors:

```text
adoption:
  extract explicit person links
  detect aliases
  emit alias collision diagnostics

garden:
  infer agenda candidates
  summarize relationship context

view:
  agenda with person
  promo evidence packet
```

No special core entity system is needed. The people extension emits facts.

## 11.4 Markdown/link maintenance

Built-in extension:

```text
dome.markdown
```

Processors:

```text
adoption:
  parse wikilinks
  validate broken links
  normalize frontmatter if allowed
  generate link facts
```

This is a good example of a “core-feeling” feature that still fits the extension system.

## 11.5 Search/indexing

Built-in extension:

```text
dome.search
```

Processors:

```text
adoption:
  update text projection
  emit indexable document records

garden:
  refresh embeddings

view:
  semantic search
```

The projection store is core infrastructure. The search behavior is an extension.

# 12. Query and recall

Queries should default to adopted state.

```ts
interface QueryRequest {
  revision?: CommitOid; // default adopted
  text?: string;
  entities?: string[];
  relations?: string[];
  timeRange?: { start?: string; end?: string };
  requireEvidence?: boolean;
  includeSourceSnippets?: boolean;
}
```

A query can return:

```text
source snippets
facts
rendered views
diagnostics
questions
suggested next actions
```

Important rule:

```text
No evidence, no durable claim.
```

Generated or inferred facts must cite source refs or be explicitly marked speculative.

This is what makes the system safe for questions like:

```text
What did Alice do this year?
What is on the agenda with Danny?
What did I decide last month?
```

# 13. Multi-client workflow examples

## 13.1 Voice capture from phone

```text
1. Phone records audio.
2. Phone transcribes locally or remotely.
3. Phone commits captures/voice/2026-05-27/<id>.md.
4. Phone pushes branch and opens PR.
5. Adoption service validates raw capture shape.
6. PR merges quickly.
7. Garden processor extracts tasks/notes/questions.
8. Engine opens follow-up PR or auto-merges high-confidence patches.
9. Other clients fetch main/adopted and see results.
```

## 13.2 Claude Code edits project notes

```text
1. Claude Code edits markdown directly.
2. It commits and pushes branch.
3. Adoption service runs linter/link/task processors.
4. Engine pushes formatting/ID patches onto the PR if needed.
5. PR becomes green.
6. PR merges.
7. Query service answers from new adopted state.
```

Claude Code does not need bespoke write tools.

## 13.3 Garden job produces daily brief

```text
1. Scheduler triggers dome.daily.morning.
2. Processor reads adopted state and graph.
3. Processor emits PatchEffect replacing owned daily region.
4. Engine creates dome/job/daily-morning branch.
5. If policy allows, it auto-merges through adoption.
6. Otherwise it opens a review PR.
```

## 13.4 Linter touches the whole vault

```text
1. User enables acme.markdown_linter.
2. Extension requests patch.auto: **/*.md.
3. Adoption processor emits large patch.
4. Engine applies in candidate tree.
5. If fixed point converges, merge continues.
6. If not, adoption blocks with diagnostics.
```

Broad write power is okay because it is still patch-mediated.

# 14. What not to build

To keep the system elegant, I would explicitly avoid these:

```text
A custom source synchronization protocol
  Git already solves this.

A giant YAML behavior DSL
  Behavior belongs in processors; YAML is metadata/config.

A separate app database as source of truth
  Markdown/Git should remain source.

Direct extension writes to adopted state
  Extensions emit effects; engine applies them.

A hardcoded ontology in core
  Relations and entities belong to extensions.

A special path for built-in features
  Built-ins use the same processor/effect system.

Generated markdown for everything
  Prefer ephemeral views unless committing the artifact helps humans.

A separate event log for source changes
  Git diffs are the source event log.
```

# 15. The core robustness story

This design is robust because every important mutation has the same shape:

```text
immutable input snapshot
  → processor
  → declared effects
  → policy validation
  → Git commit or projection update
```

The engine can always answer:

```text
What input commit caused this?
Which processor produced it?
What patch did it apply?
What source evidence supports it?
Can I revert it?
Can I rebuild it?
```

Failure modes are contained:

```text
bad client commit
  PR blocks

bad adoption processor
  adoption blocks; disable/quarantine processor

bad garden processor
  bad PR or failed run, adopted state remains safe

bad projection
  rebuild from adopted commit

bad generated content
  revert engine commit

external side effect failure
  retry or inspect outbox ledger
```

# 16. The clean final architecture

If I had to compress the whole design into one page:

```text
Dome is a Git-native semantic adoption system for markdown vaults.

Clients communicate by Git:
  commit, branch, PR, fetch, merge.

The backend runs adoption:
  candidate Git tree
    → processors
    → effects
    → patches/diagnostics/facts/questions/jobs
    → accepted revision

The latest trusted state is:
  main in hosted protected mode
  refs/dome/adopted/main in local eventual mode

Core owns:
  Git adoption
  markdown parsing
  processor runtime
  effect validation
  patch application
  projection storage
  query/view execution

Extensions own:
  domain semantics
  intake
  tasks
  people
  daily notes
  voice
  calendar
  ideation
  code docs
  search behavior

Extensions are powerful because they can:
  read snapshots
  emit facts
  emit diagnostics
  emit questions
  invoke models if granted
  propose or auto-apply patches
  own generated regions
  register views and commands

Extensions are safe because they cannot:
  mutate adopted state directly
  bypass Git history
  skip policy validation
  write projections without source refs
  perform external side effects without audited capabilities
```

The most elegant part is that “core” features do not need to be special. Auto-intake, daily notes, link maintenance, agenda prep, and voice extraction can all be built as first-party extensions over the same processor/effect contract that third-party extensions use.

The smallest deep contract is:

```text
Git snapshot in.
Effects out.
Engine decides what becomes state.
```

That is the system I would build.
