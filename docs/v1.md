According to the uploaded architect briefing, Dome is already best understood as “a compiler over a markdown vault,” where the compiler is load-bearing and the writing surface is interchangeable. That framing is the anchor for the revised spec below.

# Dome v1 Spec: Git-Native Semantic Adoption Engine

## 0. Design thesis

Dome is not primarily a file mutation API, a hook runner, or a plugin framework.

Dome is:

> **A Git-native semantic adoption engine for markdown vaults.**

The core loop is:

```text
Clients write markdown and create Git revisions.
Dome compiles those revisions.
Dome adopts the latest fully understood semantic state.
Shells read, query, and render that adopted state.
Extensions add domain-specific semantics through code, patches, and views.
```

The central v1 design move is to separate:

```text
source branch
  The latest user/client-written state.

adopted ref
  The latest Dome-compiled, semantically coherent state.
```

For example:

```text
refs/heads/main
  Raw source branch. Humans, agents, editors, and clients may write here.

refs/dome/adopted/main
  Latest fully adopted semantic state. Dome advances this ref only after compilation succeeds.
```

This gives Dome local eventual consistency now and a natural path to hosted merge-queue semantics later.

# 1. What this spec simplifies

Earlier versions of the design introduced too many custom concepts: custom revisions, change batches, manifest-defined generation types, a large YAML contribution DSL, and extension-declared page/generation/workflow structures.

This spec collapses those.

## 1.1 Git owns revision identity

Use Git concepts directly:

```ts
type CommitOid = string;
type TreeOid = string;

interface GitRevision {
  commit: CommitOid;
  tree: TreeOid;
}

interface GitRange {
  base: CommitOid;
  head: CommitOid;
}
```

No separate `VaultRevision`, `ChangeBatch`, or custom event queue is needed for v1.

The source queue is:

```text
refs/dome/adopted/main .. refs/heads/main
```

## 1.2 Git patches are the extension effect system

Extensions do not normally write files directly. They produce patches.

```text
Extension code
  → proposed patch
  → engine validates permission
  → engine applies patch in staging
  → engine commits patch
  → adopted ref advances
```

This preserves broad extension power without losing rollback, reviewability, idempotency, or provenance.

## 1.3 Manifest is not the extension language

The extension manifest should not be a big YAML DSL.

It should be an install contract:

```text
identity
entrypoint
requested permissions
static assets
trust requirements
default config
```

Behavior belongs in code.

## 1.4 Ownership is separate from write permission

This is the most important extension-surface correction.

An auto-linter may need broad patch permissions:

```yaml
patch.auto:
  - "**/*.md"
```

But that does not mean it owns those files.

Ownership is only for things an extension may regenerate repeatedly, such as generated regions or generated paths.

```text
patch permission:
  "I may produce edits to this path."

ownership:
  "This region/path is my generated artifact and I may replace it later."
```

This distinction makes the system much less rigid.

# 3. Git branch and ref model

## 3.1 Local v1 mode

Local v1 optimizes for single-client use.

```text
refs/heads/main
  User/client source branch.

refs/dome/adopted/main
  Latest fully adopted semantic state.
```

The source branch may run ahead:

```text
main:                  A --- B --- C
refs/dome/adopted/main A
```

After Dome sync:

```text
main:                  A --- B --- C --- D
refs/dome/adopted/main                 D
```

Where:

```text
B, C = user/client commits
D    = Dome engine closure commit, if needed
```

If no closure commit is needed, adopted can point directly at `C`.

```text
main:                  A --- B --- C
refs/dome/adopted/main             C
```

## 3.2 Meaning of the adopted ref

The adopted ref means:

> Dome has compiled this Git revision, applied required engine closures, updated semantic projections, recorded diagnostics, and considers this revision safe for trusted semantic queries.

Queries default to adopted state.

```text
"What is on the agenda with Danny?"
  → query refs/dome/adopted/main
```

A shell may explicitly request source-head or working-tree mode, but that is provisional.

## 3.3 Hosted merge queue as future mode

Hosted merge queue is not required for v1.

It is an admission-control layer over the same adoption machinery.

Local mode:

```text
main may be ahead of adopted.
adopted is trusted.
```

Hosted protected-trunk mode:

```text
clients submit branches
Dome compiles candidate branch
Dome adds closure commit
main advances only after adoption
```

In hosted protected mode:

```text
main == adopted semantic trunk
```

Same compiler. Same extension model. Stricter write admission.

# 4. Adoption state machine

The core state transition is:

```text
source range
  → staging
  → compile
  → diagnose
  → close
  → adopt
```

## 4.1 Source range

Dome identifies:

```text
base = refs/dome/adopted/main
head = refs/heads/main
range = base..head
```

## 4.2 Staging

Dome creates a staging branch or worktree.

```text
refs/dome/staging/<run-id>
```

The engine performs all adoption work there.

The adopted ref is not moved until adoption succeeds.

## 4.3 Compile

Core compiles the Git range.

```ts
interface CompileResult {
  base: CommitOid;
  head: CommitOid;
  tree: TreeOid;

  observations: Observation[];
  diagnostics: Diagnostic[];
  graphFacts: GraphFact[];
  patchRequests: PatchRequest[];
}
```

## 4.4 Diagnose

Blocking diagnostics stop adoption.

Examples:

```text
generated region manually edited
duplicate stable document ID
raw immutable capture modified
invalid frontmatter schema
patch would escape granted permissions
extension adoption pass failed
```

If adoption blocks:

```text
source branch remains ahead
adopted ref remains unchanged
diagnostics are recorded
optional review branch is created
```

## 4.5 Close

The engine applies required patches.

Examples:

```text
refresh generated region
insert missing task ID
normalize frontmatter
update generated index
apply linter patch
write extracted voice task
```

Every engine-applied patch becomes a visible Git commit.

Example commit:

```text
dome(linter): normalize markdown formatting

Dome-Extension: acme.markdown_linter
Dome-Run: run_01J...
Dome-Base: 41a98c2
Dome-Source-Head: 9c1e002
```

## 4.6 Adopt

Finally, Dome atomically updates:

```text
refs/dome/adopted/main → final staging commit
```

If the process crashes before this ref update, the previous adopted state remains trusted.

If it crashes after the ref update, adoption succeeded.

```ts
interface ParsedDocument {
```

# 6. Engine API

```ts
interface DomeEngine {
  status(): Promise<EngineStatus>;

  sync(input?: SyncInput): Promise<SyncResult>;

  query(input: KnowledgeQuery): Promise<KnowledgeResult>;

  renderView(input: RenderViewInput): Promise<RenderedView>;

  run(input: ManualRunInput): Promise<Run>;

  diagnostics(input?: DiagnosticQuery): Promise<Diagnostic[]>;

  questions(input?: QuestionQuery): Promise<Question[]>;

  close(): Promise<void>;
}
```

## 6.1 Sync modes

```ts
type SyncMode =
  | "provisional-working-tree"
  | "adopt-commits"
  | "adopt-working-tree";
```

### `provisional-working-tree`

Fast feedback.

```text
compile dirty working tree
do not create commits
do not advance adopted ref
do not run expensive handlers
```

### `adopt-commits`

Robust default.

```text
compile adopted..HEAD
apply required closure patches
commit engine patches
advance adopted ref
```

### `adopt-working-tree`

Convenience mode.

```text
create checkpoint commit from working tree
compile/adopt checkpoint
advance adopted ref
```

## 6.2 Engine state

Git stores source state.

SQLite/local DB stores runtime and projection state.

```text
.git/
  commits, branches, refs, diffs, rollback

.dome/state/engine.sqlite
  runs
  diagnostics
  extension grants
  schedule cursors
  questions
  capability usage

.dome/index/graph.sqlite
  rebuildable graph projection

.dome/index/search.sqlite
  rebuildable search/vector projection
```

Do not store knowledge only in SQLite. Knowledge must be recoverable from Git-backed markdown or cited generated artifacts.

# 7. Extension model

## 7.1 Definition

An extension is:

```text
manifest + code entrypoint + optional assets
```

The manifest is not the behavior. The code is the behavior.

```text
Manifest:
  install contract

Code:
  analyzers, handlers, views, commands
```

## 7.2 Extension contribution types

Only four contribution types are needed.

```text
Analyzer
  Runs during compilation or adoption.
  Reads source and emits diagnostics, observations, graph facts, or patches.

Handler
  Durable runtime job triggered by event, schedule, or command.
  May use capabilities.
  May emit patches, diagnostics, questions, graph facts, or external effects.

View
  Read-only rendering over adopted source, graph, diagnostics, and runs.

Command
  Shell-exposed semantic operation.
```

Everything else is a specialization.

```text
hook         = subscription that starts a handler
workflow     = handler that may use a model/prompt
prompt       = handler asset
tool         = command projected into a shell
generated region = owned patch target
```

## 7.3 Extension activation API

```ts
export default defineExtension({
  id: "dome.daily",
  version: "1.0.0",

  activate(ctx) {
    ctx.analyzer("extract_tasks", {
      gate: "adoption",
      run: extractTasks,
    });

    ctx.handler("morning_brief", {
      trigger: { schedule: "0 7 * * *" },
      run: morningBrief,
    });

    ctx.view("today", {
      render: renderToday,
    });

    ctx.command("prepare_agenda", {
      input: PrepareAgendaInput,
      output: PrepareAgendaOutput,
      run: prepareAgenda,
    });
  },
});
```

The extension receives registration APIs at activation time.

At runtime, handlers receive capabilities.

```ts
interface RunContext {
  run: Run;
  adopted: GitRevision;

  source: SourceReadCapability;
  patches: PatchCapability;
  graph: GraphCapability;
  diagnostics: DiagnosticCapability;
  questions: QuestionCapability;

  model?: ModelCapability;
  calendar?: CalendarCapability;
  notify?: NotifyCapability;
  network?: NetworkCapability;
}
```

No raw filesystem by default. No raw database. No raw Git mutation.

# 8. Minimal extension manifest

The manifest should be small.

```yaml
id: dome.daily
version: 1.0.0
entry: ./dist/index.js

permissions:
  source:
    read:
      - wiki/**
      - notes/**
      - captures/**

  patch:
    propose:
      - wiki/**
      - notes/**
    auto:
      - wiki/daily/**
      - wiki/tasks/**

  own:
    regions:
      - dome.daily.*
    paths:
      - wiki/generated/daily/**

  graph:
    read: true
    write:
      - dome.daily.*
      - dome.tasks.*

  model:
    invoke: optional
    max_daily_cost_usd: 2.00

  network:
    fetch: false

  calendar:
    read: optional
    write: false

  notify:
    send: false
```

The manifest says what the extension requests.

The vault config says what the user grants.

```yaml
extensions:
  dome.daily:
    enabled: true
    grant:
      patch.auto:
        - wiki/daily/**
      model.invoke: true
    config:
      morning_brief_time: "07:00"
      include_calendar: true
```

This gives the vault owner control.

# 9. Permissions model

## 9.1 Source read permission

```yaml
source:
  read:
    - wiki/**
    - notes/**
```

The extension may inspect matching files through engine APIs.

## 9.2 Patch proposal permission

```yaml
patch:
  propose:
    - "**/*.md"
```

The extension may produce reviewable patches against matching paths.

## 9.3 Patch auto-apply permission

```yaml
patch:
  auto:
    - wiki/daily/**
```

The extension may produce patches the engine is allowed to apply automatically.

The engine still validates:

```text
patch applies cleanly
patch touches only granted paths
patch does not violate generated-region ownership
patch does not introduce blocking diagnostics
```

## 9.4 Ownership permission

```yaml
own:
  regions:
    - dome.daily.*
  paths:
    - wiki/generated/daily/**
```

Ownership means:

```text
The extension may repeatedly overwrite these generated regions or generated paths.
```

Ownership is stronger than patch permission, but narrower.

An extension may have broad patch permission without owning anything.

## 9.5 Graph permission

```yaml
graph:
  read: true
  write:
    - dome.people.*
    - dome.tasks.*
```

Graph writes are writes to a projection, not writes to source truth.

Every graph fact must include source refs unless explicitly marked speculative.

## 9.6 Model/network/calendar/notify permissions

These are runtime capabilities.

```yaml
model:
  invoke: optional
  max_daily_cost_usd: 2.00

network:
  fetch:
    - api.example.com

calendar:
  read: true
  write: false

notify:
  send: false
```

External side effects must occur inside a `Run` and be recorded.

## 9.7 Unsafe permissions

Some local extensions need process or filesystem escape hatches.

Make this explicit.

```yaml
unsafe:
  fs:
    read:
      - /Users/me/Documents/**
    write:
      - .dome/cache/acme/**
  shell:
    exec:
      - prettier
      - rg
```

Unsafe permissions should be:

```text
disabled by default
local-only unless explicitly trusted
visible in dome status/doctor
never silently granted to remote packages
```

# 10. Trust tiers

```text
builtin
  Shipped with Dome. May be adoption-gated.

local
  Stored in the vault or local workspace. Can request broad permissions.

package
  Installed from a package manager. Needs explicit grants for broad writes,
  unsafe capabilities, model access, or adoption-gated execution.
```

Default policy:

```text
third-party package extensions cannot be adoption-blocking by default
third-party package extensions cannot auto-patch broad paths by default
LLM/model use is opt-in
network/calendar/notify are opt-in
unsafe capabilities are opt-in and loud
```

# 12. Gates

Keep gates simple.

```text
adoption
  Must finish before adopted ref advances.

post-adoption
  Runs after adopted ref advances and may create later engine commits.

manual/scheduled
  Runs only from user command or schedule.
```

## 12.1 Adoption gate

Allowed by default:

```text
deterministic analyzers
schema validation
link validation
generated-region ownership checks
cheap linting
safe patch generation
```

Not allowed by default:

```text
LLM calls
network calls
calendar writes
notifications
long-running jobs
non-idempotent effects
```

An auto-linter may be adoption-gated if it is deterministic and bounded.

## 12.2 Post-adoption gate

Good for:

```text
task extraction
voice extraction
summary refresh
agenda index update
graph enrichment
embedding jobs
```

## 12.3 Manual/scheduled gate

Good for:

```text
daily morning brief
weekly review
research synthesis
promo packet generation
large vault migration
```

# 13. Patch effect system

## 13.1 Patch request

```ts
interface PatchRequest {
  id: string;
  owner: ExtensionId | "engine";
  reason: string;
  sourceRefs: SourceRef[];
  patch: UnifiedDiff;
  mode: "propose" | "auto";
}
```

## 13.2 Patch application

The engine applies patches only in staging.

```text
patch request
  → permission check
  → apply in staging worktree
  → recompile/validate if needed
  → commit
  → advance adopted ref
```

## 13.3 Why not raw writes?

Trusted extensions may request broad patch permissions, but normal extensions should not mutate files directly because direct mutation breaks:

```text
atomic adoption
rollback
review
idempotency
run-to-commit provenance
index consistency
crash recovery
```

Patch-mediated writes preserve the Git-native state machine while allowing powerful changes.

# 14. Generated regions

Generated regions are just owned patch targets.

Syntax:

```md
<!-- dome:region id="dome.daily.morning_brief" owner="dome.daily" checksum="sha256:..." -->

## Morning brief

...

<!-- /dome:region -->
```

Rules:

```text
An extension may replace a generated region only if it owns that region namespace.

Manual edits inside generated regions produce diagnostics.

User prose outside generated regions is never rewritten by generated-region logic.

Generated regions are discovered from markdown; they do not need to be predeclared one by one in YAML.
```

This is the fix for the earlier “manifest specifies generation types” awkwardness.

Manifest should say:

```yaml
own:
  regions:
    - dome.daily.*
```

Not:

```yaml
generated_regions:
  - id: dome.daily.morning_brief
  - id: dome.daily.task_rollover
  - id: dome.daily.agenda
```

The markdown and extension code define actual regions.

The manifest only grants the namespace.

# 16. Runs and idempotency

Git owns source history.

The run ledger owns runtime history.

```ts
interface Run {
  id: RunId;
  extensionId: ExtensionId;
  contributionId: string;

  trigger:
    | { kind: "adoption"; sourceHead: CommitOid }
    | { kind: "observation"; observationHash: string; adopted: CommitOid }
    | { kind: "schedule"; scheduledFor: string; adopted: CommitOid }
    | { kind: "manual"; command: string; adopted: CommitOid };

  idempotencyKey: string;

  status:
    | "queued"
    | "running"
    | "succeeded"
    | "failed"
    | "skipped"
    | "quarantined";

  inputRefs: SourceRef[];
  outputPatches: PatchRequest[];
  capabilityUses: CapabilityUse[];
  cost?: CostRecord;
  error?: SerializedError;

  createdAt: string;
  startedAt?: string;
  finishedAt?: string;
}
```

Idempotency keys should include:

```text
extension id
contribution id
extension version
source/adopted commit
relevant source refs
prompt/model version when applicable
```

This turns Git into the anchor for replay safety.

# 18. Shell surface

Shells should consume an engine surface.

```ts
interface EngineSurface {
  commands: CommandRegistry;
  views: ViewRegistry;
  diagnostics: DiagnosticAPI;
  questions: QuestionAPI;
  instructions: InstructionBundle;
}
```

## 18.1 Shell command examples

```text
dome status
dome sync
dome sync --provisional
dome sync --adopt-working-tree
dome query "agenda with Danny"
dome view dome.daily.today
dome diagnostics
dome questions
```

## 18.2 Claude Code workflow

```text
1. Claude Code edits markdown directly.
2. Claude Code commits changes or runs dome sync --adopt-working-tree.
3. Dome compiles and adopts.
4. Claude Code queries adopted state.
```

No need to force Claude Code through `writeDocument`.

## 18.3 Obsidian workflow

```text
1. User edits markdown freely.
2. Local daemon gives provisional diagnostics.
3. Periodic checkpoint or manual commit creates source revision.
```

## 19.1 Auto-linter extension

### Manifest

```yaml
id: acme.markdown_linter
version: 1.0.0
entry: ./dist/index.js

permissions:
  source:
    read:
      - "**/*.md"

  patch:
    auto:
      - "**/*.md"

  model:
    invoke: false

  network:
    fetch: false
```

### Code

```ts
export default defineExtension({
  id: "acme.markdown_linter",
  version: "1.0.0",

  activate(ctx) {
    ctx.analyzer("lint_all_markdown", {
      gate: "adoption",

      async run(runCtx) {
        const files = await runCtx.source.glob("**/*.md");

        const patch = await createMarkdownLintPatch(files, {
          ruleset: runCtx.config.ruleset ?? "default",
        });

        if (patch.isEmpty()) return;

        return runCtx.patches.auto({
          reason: "Apply markdown lint fixes",
          patch,
        });
      },
    });
  },
});
```

### Adoption behavior

```text
1. User commits messy markdown.
2. Dome stages adoption.
3. Linter runs during adoption.
4. Linter emits patch touching many files.
5. Engine validates patch.auto permission for **/*.md.
6. Engine applies patch in staging.
7. Engine commits linter fix.
8. Engine advances adopted ref.
```

The linter has broad write power, but the write is still reviewable and revertible.

# 21. Invariants

These are the v1 invariants I would use.

```text
GIT_IS_REVISION_SUBSTRATE
Commit OIDs, tree OIDs, ranges, diffs, and refs define source ordering.

MARKDOWN_IS_SOURCE_OF_KNOWLEDGE
Durable knowledge must be recoverable from markdown, captures, or cited generated artifacts.

ADOPTED_REF_IS_SEMANTIC_CURSOR
refs/dome/adopted/<branch> points to the latest fully compiled and closed revision.

CORE_IS_DETERMINISTIC
Core compilation has no shell, LLM, network, calendar, notification, or runtime side effects.

ENGINE_OWNS_ADOPTION
Only the engine advances adopted refs and applies engine patches.

EXTENSIONS_ARE_CODE_FIRST
Manifests declare identity and requested capabilities; behavior lives in extension code.

EXTENSIONS_DO_NOT_WRITE_DIRECTLY_BY_DEFAULT
Extensions mutate source by emitting patches that the engine validates and commits.

PATCH_PERMISSION_IS_NOT_OWNERSHIP
An extension may patch paths without owning them.

GENERATED_REGIONS_ARE_OWNED
Automation may repeatedly overwrite only owned generated regions or owned generated paths.

CAPABILITIES_ARE_EXPLICIT
Model, network, calendar, notification, shell, and unsafe filesystem access require explicit grants.

RUNS_ARE_IDEMPOTENT_BY_GIT_INPUT
Run idempotency keys include Git revision/tree, extension version, and relevant source refs.

GRAPH_IS_A_PROJECTION
The graph accelerates recall but is rebuildable from adopted source and enabled extensions.

QUERIES_DEFAULT_TO_ADOPTED
Semantic queries read the adopted ref unless explicitly asked for provisional source or working-tree state.

SHELLS_ARE_REPLACEABLE
No domain semantics depend on a particular shell.

HOSTED_MQ_IS_ADMISSION_POLICY
A future merge queue changes how commits enter trunk, not how Dome compiles or adopts them.
```

# 24. Final compressed spec

```text
Dome v1 is a Git-native semantic adoption engine for markdown vaults.

Git owns revision identity, ordering, diffs, branch state, rollback, and source history.

Markdown owns durable human-readable knowledge.

Core is a deterministic compiler:
  Git range + config + compiler contributions
    → observations
    → diagnostics
    → graph facts
    → patch requests

Engine is a runtime:
  stages adoption
  runs Core
  validates diagnostics
  applies extension patches
  commits closure changes
  advances refs/dome/adopted/<branch>
  maintains run ledger, diagnostics, indexes, questions, and views

Shells are replaceable:
  edit markdown
  create commits
  call engine commands
  render views
  display diagnostics/questions

Extensions are code-first:
  manifest declares identity and requested capabilities
  code registers analyzers, handlers, views, and commands
  extensions mutate source by emitting Git patches
  broad patch permissions are allowed
  ownership is reserved for generated regions/paths
  unsafe raw access is explicit and local-trust only by default

Local v1:
  source branch can run ahead
  adopted ref is the latest trusted semantic state

Hosted future:
  merge queue protects trunk
  same adoption engine decides what can merge
```

The key rule is:

> **Extensions can be powerful, even broadly write-capable, but their normal effect on source is a Git patch that the engine validates, commits, and adopts.**

That removes the overbuilt YAML DSL while preserving the clean state transition that makes Dome robust.
