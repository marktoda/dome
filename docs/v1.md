# Dome v1 design and plan

This is the V1 plan of record as of 2026-06-01.

V1 is for Mark's Git-backed Obsidian work vault. The implementation should be
narrow, but the architecture should preserve the broader V2 shape:

> Dome is a source-preserving, convergent maintenance-loop runtime for personal
> memory.

Put in product language:

> Dome keeps your second brain in a useful state.

V1 is not a note-taking app, a task manager, a daily-note generator, or a vault
linter. Those are surfaces or individual maintenance concerns. Dome is the
background compiler and garden engine that keeps a knowledge environment useful
over time.

## Product thesis

The work vault naturally drifts:

- raw captures pile up,
- tasks and commitments get stranded,
- duplicate concepts appear,
- links break or point at the wrong thing,
- project pages go stale,
- daily notes lose the thread,
- foreground agents search the wrong files,
- important uncertainty disappears.

Dome notices that drift and restores useful state.

The user keeps writing, capturing, and working in normal markdown. Foreground
agents keep doing point-in-time work with the user. Dome handles the background
maintenance that should not require a fresh manual prompt every time.

The V1 value proposition is:

1. The vault stays source-first and inspectable.
2. Dome safely adopts committed markdown.
3. Background maintenance loops improve the vault over time.
4. The daily note and context packets surface what matters now.
5. Foreground agents get better read-first context.
6. Low-risk uncertainty is resolved by agents or models when evidence is
   sufficient.
7. Owner attention is reserved for decisions that actually need Mark.

## Compiler model

Dome's current engine model remains valid:

```txt
native markdown/git write
  -> Proposal
  -> adoption fixed-point loop
  -> adopted semantic state
  -> garden, schedule, answer, and job dispatch
  -> projections and surfaces
```

The low-level core is still:

```txt
Vault / Proposal / Processor / Effect
```

Maintenance loops do not replace those primitives. They sit one level above
them as the product and extension design language.

```txt
Processor = executable compiler pass
Effect    = compiler output
Proposal  = compilation unit
Projection = rebuilt memory/index
Surface   = rendered or written interface
Loop      = named desired-state objective maintained by processors
```

A processor answers: "What code runs, in what phase, with what triggers and
capabilities?"

A maintenance loop answers: "What useful condition should remain true over
time, and how does Dome know it has settled?"

That distinction keeps the engine simple. V1 should not introduce a new
`DomeLoop.run()` workflow primitive. Processors remain the only executable
extension unit, and effects remain the only output.

## Product model

The V1/V2 product model is:

```txt
Sources -> Memory -> Maintenance Loops -> Surfaces
```

### Sources

Sources are the raw material of memory.

For V1, sources are mostly markdown files in a Git-backed Obsidian vault:

- daily notes,
- meeting notes,
- project notes,
- people pages,
- raw captures,
- generated intake pages,
- commits and source refs.

For V2, sources can expand to voice captures, calendar events, emails, chats,
documents, web clips, transcripts, and mobile captures.

Rule:

> Dome preserves raw evidence and provenance.

Dome may improve human-facing markdown, but source material and the evidence
behind generated claims must remain inspectable and recoverable.

### Memory

Memory is Dome's rebuilt understanding of sources.

It includes projections such as:

- links,
- tags,
- entities,
- topics,
- tasks,
- open loops,
- commitments,
- decisions,
- questions,
- search documents,
- summaries,
- recency,
- relationships,
- diagnostics.

These should not all become heavy first-class product objects. Most are
projection facts, search rows, diagnostics, or generated markdown derived from
sources.

```txt
Source = durable human memory
Memory = Dome's working interpretation
```

### Maintenance Loops

Maintenance loops are the V1 automation design unit.

A loop maintains one desired condition:

```txt
Raw captures should get digested.
Open loops should not disappear.
Links should resolve or remain intentionally uncertain.
Foreground agents should have good context packets.
Important questions should remain alive until resolved.
Today's work surface should be useful.
```

A loop is not a scheduler, not a workflow engine, and not a processor subtype.
It is a named maintenance objective implemented by one or more processors.

### Surfaces

Surfaces are where memory becomes useful.

V1 surfaces include:

- daily notes,
- generated intake pages,
- synthesis pages,
- project and person pages,
- `dome today`,
- `dome prep`,
- `dome agenda`,
- `dome query`,
- `dome export-context`,
- `dome check`,
- `dome status`.

V2 surfaces may include a mobile app, voice capture, meeting briefs, weekly
reviews, email digests, browser extension views, and API or MCP consumers.

Rule:

> Loops maintain memory. Surfaces display or receive the results.

The daily note is important in V1, but it is a surface, not the architecture.

## Governing laws

### 1. Source preservation

Dome may make source-preserving improvements automatically.

When a change would destroy information, choose between ambiguous meanings, or
assert something not grounded in the user's sources, Dome should preserve the
uncertainty instead.

Practical implications:

- raw captures are immutable,
- generated assertions need source refs,
- duplicate consolidation should preserve source material,
- ambiguous link fixes should become questions or proposals,
- destructive edits require especially strong evidence and git-visible
  provenance.

### 2. Provenance everywhere

Every generated claim, patch, question, or surfaced item should be able to
answer:

```txt
Why do you think that?
Where did it come from?
When did that become true?
What source would I inspect?
What processor and run produced it?
```

The current run ledger, source refs, projections, and Dome commit trailers are
the right foundation. V1 should make this easier to inspect.

### 3. Convergence

Repeated Dome runs with no new source changes should eventually no-op.

```txt
Run Dome once: the vault gets better.
Run it again: maybe a little better.
Run it repeatedly with no new source changes: eventually no changes.
```

Every maintenance loop must define:

- what input changed,
- what stable output identity should exist,
- how the processor detects that output already exists,
- what makes the next run a no-op.

This is the main safety rule for LLM-backed processors.

### 4. Uncertainty is durable

Dome should not silently collapse ambiguity.

If evidence is insufficient, the system should preserve uncertainty as a
source-backed question, diagnostic, or candidate proposal. Agents and models may
resolve low-risk uncertainty when policy allows, but the resolution path should
leave an audit trail.

### 5. Small CLI, powerful compiler

V1 should not add a new top-level command for every maintenance behavior.

Primary operator commands stay consolidated:

- `serve` / `sync` run the compiler and background work,
- `status` / `check` explain what remains,
- `resolve` records decisions,
- `query` / `export-context` support foreground agents,
- `today` / `prep` / `agenda` render work surfaces,
- `inspect` / `doctor` remain advanced/debug surfaces.

There should be no separate `dome tend` command in V1. Tending is what
`serve` and `sync` already do after adoption.

## Current shipped foundation

The current codebase already ships the substrate needed for V1:

- Git-backed markdown source of truth.
- `refs/dome/adopted/<branch>` semantic cursor.
- Fixed-point adoption loop.
- Processor/effect/capability boundary.
- Adoption, garden, view, schedule, answer, and job dispatch.
- Garden-emitted patches routed as sub-Proposals.
- Run ledger, projection store, questions, answers, outbox, and recovery paths.
- CLI compiler loop: `serve`, `sync`, `status`, `check`, `resolve`.
- Recall and work surfaces: `query`, `export-context`, `today`, `prep`,
  `agenda`, `lint`, `inspect`, `doctor`.
- First-party bundles for markdown, graph, health, daily, intake, lint, and
  search.
- Optional model-invoke support for garden processors.

V1 should build on this. It does not need an engine rewrite.

## Loop contract

Every V1 maintenance loop should have a small registry-level definition. V1
keeps this as an adjacent first-party registry rather than embedding loop
metadata in each bundle manifest; that keeps loops clearly descriptive while
future bundle-local loop metadata remains possible.

This metadata is not an executable workflow. It is a design, status, and test
surface over existing processors.

Conceptual shape:

```yaml
loops:
  - id: dome.capture.digest
    goal: "New raw captures have a disposition."
    evidence:
      - ".dome/config.yaml"
      - ".dome/model-provider.ts"
      - "inbox/raw/*.md"
      - "inbox/processed/*.md"
      - "wiki/generated/intake/*.md"
      - "wiki/syntheses/intake-*.md"
    processors:
      - dome.intake.extract-capture
      - dome.intake.capture-index
      - dome.intake.synthesize-capture
      - dome.intake.synthesize-rollup
    surfaces:
      - "wiki/generated/intake/*.md"
      - "inbox/processed/*.md"
      - "wiki/syntheses/intake-rollup.md"
      - "export-context"
    settlement:
      key: "raw path + raw content hash + processor version"
      noOpWhen: "generated digests, archives, and syntheses carry current source/input hashes"
```

Required loop fields:

- `id` - stable namespaced identifier.
- `goal` - the desired condition in plain language.
- `evidence` - source paths, projections, or operational rows the loop reads.
- `processors` - required processor ids that implement the loop.
- `optionalProcessors` - optional processor ids from opt-in bundles that
  contribute evidence or answers when enabled without making the loop partial
  when disabled.
- `surfaces` - markdown files, view commands, or context exports the loop feeds.
- `settlement` - stable identity and no-op condition.
- `risks` - known failure modes, especially LLM churn or destructive edits.

Loop-level status should eventually answer:

```txt
loop id
current state
last successful run
last source change considered
pending questions
pending owner decisions
pending agent-safe work
recent patches
whether the loop appears settled
```

## V1 maintenance loops

V1 should focus on five first-party loops and one composed daily surface.

### 1. Capture digestion loop

Desired state:

```txt
New raw captures have a disposition.
```

Evidence:

- `inbox/raw/*.md`,
- `inbox/processed/*.md`,
- generated intake pages,
- capture-index facts,
- low-confidence questions.

Corrections:

- extract source-backed observations,
- produce generated capture pages,
- archive processed raw files without mutating their content,
- identify candidate tasks, commitments, entities, decisions, and questions,
- update synthesis and rollup pages.

Surfaces:

- generated intake pages,
- intake rollup,
- `query`,
- `export-context`,
- daily open-loop surface.

Settlement:

- stable key is raw path plus raw content hash plus processor version,
- no-op when the current raw capture hash has a matching generated digest,
  processed archive, and disposition,
- low-confidence candidates are persisted in generated capture frontmatter so
  questions can be rebuilt without the raw inbox file or another model call,
- synthesis updates only when the generated capture set changes.

Completion criteria:

- A new raw capture becomes a generated source-backed digest without manual
  prompting.
- Re-running `sync` without source changes produces no capture churn.
- Low-confidence items become durable questions with source refs and rebuild
  from generated markdown.
- At least one real work-vault week of captures can be processed without lost
  raw material.

### 2. Open-loop continuity loop

Desired state:

```txt
Important unresolved work remains visible until resolved, dismissed, or made
irrelevant by newer sources.
```

This is broader than checkbox tasks. It includes commitments, waiting-on items,
follow-ups, unresolved project next steps, and important loose ends.

Evidence:

- daily notes,
- meeting notes,
- project pages,
- generated intake pages,
- task-index facts,
- question rows,
- recent commits.

Corrections:

- extract source-backed open-loop facts,
- dedupe repeated mentions of the same loop,
- carry unresolved work forward to the right surface,
- mark loops stale, resolved, or blocked when evidence supports it,
- preserve ambiguity as questions.

Surfaces:

- today's daily note,
- `dome today`,
- `dome prep`,
- project pages,
- `export-context`.

Settlement:

- stable key is source ref plus normalized open-loop text plus optional entity
  or project,
- no-op when the open loop is already represented in projection memory and the
  current daily/project surface has the correct source-backed entry,
- resolved loops stop resurfacing only when a source-backed resolution exists.

Completion criteria:

- Stranded todos from meeting notes and captures reliably appear in the daily
  work surface.
- The same open loop does not duplicate every day.
- Resolved or dismissed loops stop resurfacing.
- `status` can distinguish owner-needed decisions from agent-safe open-loop
  cleanup.

### 3. Link and concept coherence loop

Desired state:

```txt
Links and concepts are navigable, intentionally unresolved, or preserved as
source-backed uncertainty.
```

This loop covers broken wikilinks, obvious aliases, duplicate concept pages,
stub creation, and index simplification.

Evidence:

- wikilinks,
- page titles,
- aliases,
- frontmatter,
- graph facts,
- search results,
- duplicate-detection diagnostics,
- recent page activity.

Corrections:

- auto-fix obvious title or slug misses,
- suggest aliases,
- create stubs when evidence is strong,
- ask questions for ambiguous links,
- propose source-preserving duplicate consolidation reviews,
- preserve redirects or source refs when merging,
- update small indexes when the correct grouping is obvious.

Surfaces:

- patched markdown links,
- alias/frontmatter updates,
- candidate stub pages,
- consolidation proposals,
- simplified index pages,
- `check` grouped by repair path.

Settlement:

- stable key for a broken link is file path plus occurrence context plus link
  target,
- stable key for a duplicate candidate is the unordered page pair plus content
  hash,
- no-op when the link resolves, is intentionally unresolved, or has an open
  source-backed question.

Completion criteria:

- Obvious broken links are fixed automatically or proposed with high precision.
- Ambiguous broken links do not create duplicate stub spam.
- Explicit missing `wiki/concepts/...` and `wiki/entities/...` links with no
  plausible existing target create source-backed stubs; bare links and
  ambiguous candidates stay diagnostics/questions.
- Duplicate consolidation starts source-preserving: aliases, redirects,
  syntheses, or proposals before deletion. In the current shipped slice,
  answering a duplicate question with `merge` creates a deterministic
  `wiki/syntheses/duplicate-review-*.md` page while preserving both sources.
- Index changes are stable across repeated runs.

### 4. Context packet loop

Desired state:

```txt
For active work, foreground agents can quickly receive a good read-first
context packet.
```

This is one of the highest-value V1 loops. Dome should make Claude Code,
Codex, ChatGPT, and future agents better without replacing them.

Evidence:

- recent commits,
- daily notes,
- project pages,
- person pages,
- graph facts,
- search documents,
- open loops,
- questions,
- recent decisions.

Corrections:

- maintain project/day/person/meeting context packets,
- rank relevant pages by recency, graph proximity, and open-loop relevance,
- include unresolved questions and recent decisions,
- produce concise source-backed summaries,
- expose read sets for foreground agents.

Surfaces:

- `dome export-context`,
- `dome query`,
- `dome prep`,
- project context markdown if useful,
- future MCP/API responses.

Settlement:

- stable key is packet target plus adopted commit range plus processor version,
- no-op when the packet was generated from the same relevant source set,
- query-time packets may be ephemeral, but persisted packets must carry their
  source set.

Completion criteria:

- A foreground agent can ask for a project/day/person packet and get a useful
  read-first set without manual file hunting.
- Packets include source refs, open loops, decisions, and unresolved questions.
- Packet generation does not rewrite markdown unless the source set changes.
- Real work-vault questions are answered faster or with fewer irrelevant reads.

### 5. Question continuity loop

Desired state:

```txt
Important uncertainty remains alive until answered, obsoleted, or explicitly
dismissed.
```

Evidence:

- `QuestionEffect` rows,
- generated questions in markdown,
- diagnostics,
- low-confidence model outputs,
- answer history,
- newer source evidence.

Corrections:

- dedupe repeated questions,
- auto-answer questions when source evidence is enough and policy allows,
- mark obsolete questions when newer sources supersede them,
- route owner-needed questions to `status` / `check`,
- route agent-safe questions to foreground-agent or model-backed resolution.

Surfaces:

- `dome check`,
- `dome status`,
- `dome resolve`,
- daily work surface,
- context packets.

Settlement:

- stable key is question idempotency key plus source refs,
- no-op when a question is answered, obsoleted, or represented exactly once as
  unresolved,
- auto-resolution requires source refs and recorded confidence.

Completion criteria:

- Repeated processor runs do not create duplicate questions.
- Agent-safe questions can be resolved without Mark manually answering every
  one.
- Owner-needed questions remain visible but do not block unrelated sync.
- Answer handlers produce source-preserving patches through normal adoption.

## Daily note as composed surface

The daily note is the main V1 work surface, not the aircraft.

Desired state:

```txt
Today has a useful work surface.
```

It should aggregate from the five loops:

- capture digestion,
- open-loop continuity,
- link and concept coherence where relevant,
- context packets,
- question continuity.

Recommended daily shape:

```txt
# YYYY-MM-DD

## Start Here

## Meetings

## Open Loops

## Notes

## Decisions

## Done

## Story of the Day
```

The daily note should stay collaborative markdown. Foreground agents may edit
it. Dome should avoid brittle ownership regions unless a specific generated
section truly needs stable identity.

If generated sections are needed, they should be small, source-backed, and
stable. The daily note should not become a rigid app database.
In the shipped V1 surface, Dome uses small generated blocks for:

- `## Start Here` context derived from yesterday's daily note,
- `## Open Loops` entries raised from source-backed open loops.

Settling a generated source-backed open-loop row should count as markdown
evidence: `[x]` means resolved/completed, and `[-]` means dismissed. Dome may
preserve those rows under `### Resolved Today` or `### Dismissed Today` and
suppress the same source/body identity and equivalent repeated surface loops
later, without mutating the original source note or requiring hidden state.

Completion criteria:

- A useful daily note exists before the workday starts.
- It includes relevant open loops, meeting prep links, unresolved questions,
  and current context.
- Mark and foreground agents can edit it naturally.
- Dome does not duplicate the same raised task every day.
- End-of-day or next-morning context can summarize what changed yesterday.

## Agent operating contract

V1 explicitly supports vault-aware foreground agents.

Agents may:

- edit markdown,
- create git commits,
- run `status`, `check`, `sync`, `today`, `prep`, `agenda`, `query`, and
  `export-context`,
- resolve agent-safe questions when policy and source evidence allow,
- fix safe diagnostics directly in markdown.

Agents must:

- avoid direct `.dome/state` edits,
- prefer source fixes over suppressing diagnostics,
- commit normal markdown changes before asking Dome to adopt them,
- preserve source refs and uncertainty,
- escalate only owner-needed decisions.

The key distinction:

```txt
Foreground agent = point-in-time collaborator.
Dome = background compiler and memory homeostasis engine.
```

V1 should make foreground agents more effective by giving them context, not by
trying to replace them.

loops.

### M1 - Loop substrate and status vocabulary

Goal:

Make maintenance loops visible without making them new executable primitives.

Work:

- Add loop metadata to first-party bundle manifests or an adjacent first-party
  loop registry.
- Document the maintenance-loop contract in the processor or SDK surface specs.
- Add loop-level status grouping to `status --json` or `check --json`.
- Add tests that loop metadata references real processor ids and valid
  surfaces.
- Add a small `inspect` subject if needed for debugging loop state.

Done when:

- Every V1 loop has an id, goal, processors, surfaces, and settlement rule.
- `status` or `check` can explain unresolved work by loop.
- Missing or stale loop metadata fails tests.
- Runtime dispatch is still processor-based.

### M2 - Work-vault dogfood baseline

Goal:

Run the existing engine in Mark's work vault as the baseline for all later
automation.

Work:

- Enable the intended command model provider for allowed LLM processors.
  `dome init --with-model-provider anthropic` should scaffold the vault-local
  command adapter and config stanza without enabling `dome.intake`
  automatically.
- Run `dome serve` during real work sessions. The long-running host should
  pick up `.dome/config.yaml`, `.dome/model-provider.ts`, and vault-local
  `.dome/extensions/**` changes without requiring a restart.
- Configure first-party surfaces to match the real work vault shape, especially
  `dome.daily.config.daily_path` for Obsidian daily notes under `notes/`.
- Route top-level `inbox/raw/*.md` captures to a status next action when the
  capture digestion loop is inactive or not model-ready.
- Let foreground agents fix existing `dome check` content diagnostics.
- Classify remaining diagnostics as auto-fixable, agent-fixable,
  owner-needed, or noise.
- Tighten AGENTS/CLAUDE guidance around the foreground-agent contract.

Done when:

- The work vault can run `sync`, `status`, `check`, `today`, `prep`, `query`,
  and `export-context` predictably.
- Existing diagnostics have a known disposition.
- No manual `.dome/state` edits are needed.
- Runtime config and vault-local extension changes take effect in `dome serve`
  before the changed commit range is adopted.
- The model provider path works for at least one real capture.
- Pending raw captures do not silently sit in `inbox/raw/` when `dome.intake`
  is disabled or not model-ready.

### M3 - Capture digestion loop

Goal:

Make messy input become useful source-backed memory.

Work:

- Harden `dome.intake` LLM processors against real work-vault captures.
- Add stable source-hash based no-op detection where missing.
- Improve generated capture pages for provenance and disposition.
- Route extracted open loops and questions into projection memory.
- Improve capture synthesis and rollup stability.

Done when:

- Real raw captures are digested without manual prompting.
- Raw captures that arrived while `dome.intake` was disabled or model-unready
  are picked up when the loop becomes active and provider-ready.
- Re-running Dome without source changes creates no capture churn.
- Generated capture facts have source refs.
- Low-confidence findings become durable questions instead of silent claims.
- Raw capture content remains preserved.

### M4 - Open-loop continuity and daily surface

Goal:

Make the daily note a useful work cockpit fed by source-backed open loops.

Work:

- Redesign the daily template.
- Add stable identities for task/open-loop facts where needed.
- Add or improve a garden processor that raises source-backed open loops into
  today's daily surface.
- Improve `today` and `prep` around the new daily shape, including ranking that
  keeps due, priority, and recently changed source-backed work near the top.
- Make bounded daily/action views sample both the target daily note and the
  wider vault backlog so `--limit` does not hide one source group entirely.
- Treat generated source-backed `## Open Loops` rows as daily-surface work in
  `today` / `prep` without indexing them as duplicate task facts.
- Fold repeated open-loop mentions at the surface layer while preserving every
  backing source reference.
- Add resolution/dismissal handling so completed or dismissed loops stop
  resurfacing.

Done when:

- Today starts with a useful work surface.
- Open tasks, commitments, waiting-on items, and follow-ups from recent sources
  surface reliably.
- Bounded `today` / `prep` views still expose both daily-note and wider-backlog
  work when both exist.
- Daily note entries are source-backed.
- Duplicate resurfacing is controlled.
- Foreground agents can update the daily naturally.

### M5 - Link and concept coherence

Goal:

Turn broken links and duplicate concepts into navigability improvements instead
of chores.

Work:

- Add repair candidates for broken wikilinks.
- Auto-fix only obvious path/title/alias misses.
- Add source-preserving stub creation for high-confidence missing concepts.
- Add duplicate-concept consolidation proposals.
- Add index simplification for small, obvious indexes.

Done when:

- Obvious broken links are fixed or proposed with high precision.
- Ambiguous links preserve uncertainty instead of generating duplicate stubs.
- Duplicate consolidation never destroys source material.
- Re-running the same link/concept loop no-ops when nothing changed.
- `check` groups remaining link/concept issues by repair path.

### M6 - Context packet loop

Goal:

Make Dome materially improve foreground-agent work.

Work:

- Improve `export-context` packet structure.
- Rank context from an expanded FTS candidate set using source-backed graph,
  page type, open-loop, decision, question, and diagnostic signals. Add recency
  only if work-vault dogfood shows it improves packets without churn.
- Recall exact-path documents from topic-matched projection memory so open
  loops, decisions, questions, and diagnostics can pull in relevant pages even
  when the page body itself misses FTS.
- Include unresolved questions and source-backed summaries.
- Add provenance/explain support for generated context.
- Dogfood packets with Claude Code/Codex on real vault tasks.

Done when:

- Foreground agents can start work from a Dome packet instead of manual file
  hunting.
- Packets are concise enough to use and complete enough to avoid obvious
  misses.
- Every packet item has a source path or source ref.
- Mark can point to real cases where packet quality improved an agent session.

### M7 - Question continuity and agent-safe resolution

Goal:

Make uncertainty durable without turning Mark into a question-answering clerk.

Work:

- Add question metadata: risk, confidence, recommended answer, automation
  policy, and owner-needed reason.
- Prefer `dome resolve` in docs, tests, and agent instructions.
- Add agent-safe resolution guidance for foreground agents.
- Add opt-in background auto-resolution for low-risk questions with
  source-grounded recommended answers.
- Keep answer handlers routed through normal garden patches and adoption.

Done when:

- Repeated runs do not duplicate questions.
- Low-risk questions can be resolved by a vault-aware agent or model with an
  audit trail.
- Owner-needed questions are clearly separated from agent work.
- Open questions do not block unrelated sync or garden progress.

### M8 - Recall and search quality

Goal:

Make the vault easier to search and reason over than plain markdown.

Work:

- Improve query ranking with graph, page type, open-loop facts, decisions,
  unresolved questions, and diagnostics; treat recency as a follow-on signal if
  work-vault examples justify it.
- Share projection-signal recall between `query` and `export-context` so
  topic-matched open loops, decisions, questions, and diagnostics can pull in
  exact-path documents even when page text misses FTS.
- Improve `query` answers and `export-context` summaries with explainable
  ranking metadata.
- Add explain/provenance views for generated facts (`inspect facts`) and
  generated markdown changes (`inspect patches`).
- Decide whether embeddings are V1 or V1.1 based on work-vault failures.

Done when:

- `query` and `export-context` find relevant work-vault material with fewer
  irrelevant files than raw text search.
- Generated answers cite source paths.
- There is an inspection path from generated memory back to processor run and
  source evidence.
- Embeddings are either shipped behind policy or explicitly deferred with
  evidence.

### M9 - Extension hardening

Goal:

Make first-party and future third-party automation safer to extend.

Work:

- Tighten manifest validation for missing processors, missing bundles, and
  invalid loop metadata.
- Clarify bundle dependencies and degradation behavior.
- Finish authoring docs for adding processors and maintenance loops.
- Reduce first-party metadata duplication.
- Add grant-scope lockstep tests where current coverage is too broad.

Done when:

- Adding a new V1 maintenance loop has a documented path.
- Invalid loop/bundle metadata fails early.
- Broad grants are visible and intentional.
- The extension surface still feels like processors and effects, not a hidden
  workflow engine.

### M10 - V1 release soak

Goal:

Prove Dome is useful in the work vault for day-to-day use.

Work:

- Run Dome continuously or near-continuously for at least two real work weeks.
- Track daily note usefulness, capture digestion quality, open-loop surfacing,
  context packet quality, and question burden.
- Fix high-friction issues found during dogfood.
- Keep tests, typecheck, docs, and substrate links green.

Done when:

- No lost or overwritten human markdown edits.
- No manual `.dome/state` edits.
- Repeated no-source-change runs settle.
- Daily notes are useful enough to start the day from.
- Foreground agents use Dome context packets or query surfaces successfully.
- Low-risk questions are usually handled without Mark.
- Remaining diagnostics are understood backlog, not mysterious failure.
- `bun test`, `bun run typecheck`, and `git diff --check` pass.

## Overall V1 done criteria

V1 is done when Mark's work vault is meaningfully better with Dome running than
with plain markdown plus ad hoc Claude Code maintenance.

Concrete gates:

- **Safety:** Git history, source refs, and raw preservation make every Dome
  change inspectable and reversible.
- **Convergence:** The five V1 loops no-op when their source evidence has not
  changed.
- **Usefulness:** Daily notes, context packets, and search surfaces are used in
  real work without feeling like chores.
- **Automation:** Safe maintenance happens without Mark manually prompting for
  every link fix, capture digest, task raise, or question resolution.
- **Attention:** `status` and `check` distinguish engine failures, content
  hygiene, agent-safe work, and owner-needed decisions.
- **Extensibility:** A new automation can be added as a maintenance loop backed
  by processors, without adding a new CLI command or engine primitive.
- **Testing:** Unit, integration, harness, and convergence tests cover the core
  loops and the full suite passes.

## Non-goals for V1

- A polished mobile app.
- Hosted multi-client sync.
- Public plugin marketplace.
- Replacing Obsidian.
- Replacing Claude Code, Codex, or other foreground agents.
- Turning markdown into a rigid app database.
- Making every warning disappear.
- Building a generic workflow engine.

## Open decisions

1. How much daily-note generated structure is enough without making the note
   feel owned by Dome?
2. What exact question metadata is required for agent-safe resolution?
3. How aggressive should duplicate consolidation be before V1?
4. Are embeddings necessary for V1, or should graph/recency/search ranking carry
   the first release?
5. Should persisted context packets be markdown files, projection rows, or
   query-time views?
6. Which link repairs are safe enough to auto-apply in Mark's work vault?

## Design summary

The simplest V1 framing:

```txt
Dome watches source-first markdown.
It adopts changes through a compiler loop.
Maintenance loops detect drift and emit grounded corrections.
Memory projections and markdown surfaces make the result useful.
All loops must preserve provenance and converge.
```

That gives V1 a narrow dogfood target while keeping the V2 architecture open:

```txt
V1: Dome runs over Mark's Git-backed markdown work vault.
V2: Dome runs over a person's connected knowledge environment.
```
