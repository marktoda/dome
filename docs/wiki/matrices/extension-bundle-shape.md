# Extension bundle shape matrix

The canonical map of "what an extension bundle contributes to a vault." Rows are bundles (first-party shipped + community-authored shapes anticipated); columns are the five registration kinds bundles can contribute. Each cell names which file inside the bundle directory provides the contribution, or marks the kind unused by this bundle.

**Lockstep status:** documentary in v0.5; lockstep-checked in v0.5.1. The v0.5.1 extension to `tests/integration/matrix-coverage.test.ts` parses this matrix and asserts each cell that names a specific filename corresponds to an actual file in the bundle's `assets/extensions/<bundle>/` shipped layout (for first-party bundles) or in the loader's fixture (for hello-world). The v0.5 matrix is the contract pin; the v0.5.1 lockstep is the structural fence.

An **extension bundle** is a directory under `<vault>/.dome/extensions/<bundle-name>/` (or shipped from the SDK at `assets/extensions/<bundle-name>/`, copied into the vault during install) containing a `manifest.yaml` plus any combination of the five registration kinds. The bundle mechanism is a packaging convention over the existing 5-kind registration surface documented in [[wiki/specs/sdk-surface]] §"Registration"; bundles do not introduce a new primitive.

## Five contribution kinds

- **Page type** — `<bundle>/page-types.yaml` adds entries to the vault's `PageTypesConfig.extensions` list during bundle load. Each entry declares the type name, optional `frontmatter_extras` schema, and (implicitly) creates `wiki/<plural>/` as a recognized directory per [[wiki/invariants/PAGE_TYPE_BY_DIRECTORY]].

- **Preamble fragment** — `<bundle>/preamble.md` contributes a markdown subsection to AGENTS.md's templated `## Extension conventions` section, regenerated by `dome doctor --repair`. The fragment teaches the agent the bundle's conventions (page-type shape, hook behavior, workflow names).

- **Workflow** — `<bundle>/workflows/<name>.md` adds a prompt to the vault's `PromptLoader` per [[wiki/specs/prompts-and-workflows]] §"Prompt loading lifecycle". Frontmatter `tools:` and `triggers:` make a plain prompt into a workflow.

- **Hook** — `<bundle>/hooks/<name>.yaml` (declarative form) or `<bundle>/hooks/<name>.ts` (programmatic form, v0.5.1+) registers a hook against an event pattern. The bundle's name namespaces the hook ID — `<bundle>:<filename>` — preventing cross-bundle hook-ID collision per [[wiki/gotchas/extension-bundle-load-order]].

- **CLI command** — `<bundle>/cli/<command>.ts` adds a `dome <command>` invocation, mirroring the vault-local `<vault>/.dome/cli/*.ts` registration kind in [[wiki/specs/sdk-surface]] §"Registration". Bundle-contributed CLI commands appear in `dome --help` after the SDK's shipped 9 commands; the `dome install-extension` workflow (v0.5.1+) re-runs the bundle loader on `--help` invocation.

- **Tool** — `<bundle>/tools/<name>.ts` adds a Tool to the registry (v0.5.1+). v0.5 bundles do not contribute Tools because the programmatic-load surface for Tools ships in v0.5.1; first-party bundles in v0.5 use only the four kinds above.

## Matrix

| Bundle ↓ \ Contribution kind → | Page type (`page-types.yaml`) | Preamble fragment (`preamble.md`) | Workflows (`workflows/*.md`) | Hooks (`hooks/*.yaml`) | CLI commands (`cli/*.ts`) |
|---|---|---|---|---|---|
| **`dailies`** *(first-party, Phase 1 shipped)* | `daily` (and `weekly`) | `preamble.md` | `create-daily.md`, `create-weekly.md` | `create-daily.yaml` (`schedule: "0 6 * * *"`), `create-weekly.yaml` (`schedule: "0 6 * * 1"`) | `migrate-dailies.ts` |
| **`aggregation`** *(first-party, Phase 3 anticipated)* | — *(reads dailies/weeklies; produces `wiki/syntheses/`)* | `preamble.md` | `weekly-rollup.md`, `monthly-retro.md`, `surface-stale-threads.md` | `weekly-rollup.yaml` (`schedule: "0 18 * * 0"`), `monthly-retro.yaml` (`schedule: "0 18 1 * *"`), `stale-threads.yaml` (`schedule: "0 18 * * 0"`) | — |
| **`recall`** *(first-party, Phase 4 anticipated)* | — *(reads everything; produces stdout)* | `preamble.md` | `plan-today.md`, `follow-up-with.md`, `retro-recall.md` | — *(workflow-only; no hook scheduling)* | `plan-today.ts`, `follow-up-with.ts`, `retro-recall.ts` |
| **`hello-world`** *(test fixture)* | `hello` | `preamble.md` | `say-hello.md` | `say-hello.yaml` (`schedule: "* * * * *"`) | — |

## Reading the matrix

- **A bundle whose row has only `—` cells in `Page type` and `Hook`** contributes only workflows and CLI commands; it adds capability the agent can invoke but does not extend the page-type system or react to vault writes. The `recall` bundle is the canonical example — it's a query surface, not a maintenance surface.

- **A bundle's `Page type` cell names which wiki/ subdirectories become recognized.** When `dailies` is loaded, `wiki/dailies/` is a valid `writeDocument` target with the `daily` type validation. Without the bundle, writes to `wiki/dailies/` are rejected by [[wiki/invariants/PAGE_TYPE_BY_DIRECTORY]].

- **A bundle's `Hook` cell with `schedule:` declarations** depends on the v0.5 `schedule:` field on declarative hooks (per [[wiki/specs/hooks]] §"Declarative — `.dome/hooks/*.yaml`"). The per-vault scheduler in `wireDispatcher` reads these and wakes on intervals, synthesizing `clock.tick.<schedule-id>` events the hook handler matches. See [[wiki/gotchas/scheduled-hook-idempotency]] for the catch-up semantics.

- **A bundle's `CLI commands` cell** introduces commands that are *bundle-conditional* — they exist only when the bundle is loaded. `dome migrate-dailies` is not in the SDK's shipped command set; it appears when `<vault>/.dome/extensions/dailies/` is present.

- **Cells marked `—`** are intentionally unused. A bundle that doesn't need a page type doesn't declare one; the loader treats the kind as not-contributed.

## Why the matrix exists

The brainstorm's pressure-test (per `docs/cohesive/brainstorms/2026-05-26-dome-hooks-v1-roadmap.md` §"G3 — substrate extension") established that the hook layer alone is insufficient for v1 features that want to bundle a page type + preamble + workflows + hooks + CLI as one coherent unit. The matrix is the substrate-shape pin that names what bundles can do; it makes the bundle mechanism legible to a new bundle author by enumeration.

The matrix is also the input the bundle-shape lockstep test asserts against (planned v0.5.1): every first-party bundle the SDK ships must have a row in this matrix, and every cell that names a filename must correspond to a real file in the bundle's `assets/extensions/<bundle>/` layout. Drift between this matrix and the shipped bundles is a structural violation.

Future bundles (Phase 2's `compile-on-write` keystone, calendar integration, voice transcription pipeline) will add rows. The five contribution kinds are sealed for v0.5 + v1; a sixth kind would be a v1.x-or-later substrate change touching the [[wiki/specs/sdk-surface]] §"Registration" 5-kind matrix.
