# Harnesses

This spec is normative for how external agentic environments interact with Dome. A *harness* is a runtime that hosts an agent loop and operates against a Dome-managed markdown vault. The harness is the SDK's user, not part of it.

Dome's relationship to harnesses is asymmetric. Native Dome surfaces (mobile, desktop, web, voice — see [[VISION]] §"Two surface patterns") embed the SDK and ship designed UX. **Agentic harnesses** (Claude Code, Cursor, OpenCode, Codex, future agents) are brought by the user; they're general-purpose and use whatever tools they have natively. The spec below governs the latter class: how *agentic harnesses* connect to a Dome vault.

## The compiler-boundary contract

Dome doesn't dictate which tools an agentic harness reaches for. Claude Code with full filesystem access uses `Read` / `Grep` / `Write` / `Edit` natively; a hypothetical sandboxed harness might prefer MCP-routed reads. Dome's contract with every harness is **the compiler boundary** — four surfaces an agentic harness interacts with, regardless of its native tool choice:

1. **`AGENTS.md` at the vault root** — auto-loaded by Claude Code (via `CLAUDE.md` shim) and any harness that supports the `AGENTS.md` convention. Teaches the agent the vault's conventions, page types, named processor commands, and invariants at session start. Generated by `dome init`; templated sections refreshed by the `mergeAgentsMd` mechanism (today: `dome init` re-runs; v1.x: the reserved `dome doctor --repair` verb plus the planned `dome.health.agents-md-template-drift` garden-phase processor per [[wiki/specs/cli]] §"dome doctor"); user-prose sections preserved via delimiter convention. See [[wiki/invariants/AGENTS_MD_IS_ORIENTATION_SURFACE]].
2. **The CLI** — `dome sync`, `dome status`, `dome inspect <subject>`, `dome query`, `dome lint`, `dome lint --apply <id>`, `dome stats`, `dome rebuild`, `dome export-context <topic>`, plus `dome doctor` and `dome answer`. Invokable from any shell. The agent calls these via whatever shell-execution tool the harness exposes (`Bash` in Claude Code). See [[wiki/specs/cli]].
3. **The compiler host** — `dome serve` is the first host mode and can run foreground like an LSP/watch process or as a background service (launchd / systemd). The host catches committed branch movement and turns it into Proposals; the engine adopts; garden-phase processors fire on their schedules. Future mobile/web/server hosts reuse the same runtime boundary rather than inventing another compiler.
4. **`dome sync`** — explicit adoption catch-up. `dome sync` constructs an internal Proposal from `refs/dome/adopted/<branch>..refs/heads/<branch>` when the compiler host was off or the agent wants to block on adoption. It runs the same engine adoption loop per [[wiki/specs/adoption]].

These four surfaces are the *load-bearing* contract. The MCP server ([[wiki/specs/mcp-surface]]) is preserved as a fifth, non-primary surface — available for future harnesses that benefit from explicitly-typed MCP-routed operations, but not load-bearing for v1 value delivery.

## How a harness writes into the vault

An agentic harness can write to the vault in three ways. Each goes through the engine's adoption loop:

### 1. Native write + watcher pickup (the canonical path)

The agent uses its harness-native write tool (`Write` in Claude Code, `:w` in vim, the OS file API in Obsidian), then commits through ordinary git. The engine constructs a Proposal automatically when `dome serve` observes the branch ref moving (or on the next `dome sync` when the compiler host is not running). Adoption proceeds.

This is the path most agentic harnesses use. Pinned by [[wiki/invariants/ALL_MUTATION_GOES_THROUGH_ADOPTION]].

### 2. Explicit `dome sync` invocation

The agent invokes `dome sync` via `Bash` (or its harness equivalent). Use this when the agent wants to *block on adoption* — running `dome sync` synchronously returns the adoption result, including any blocking diagnostics. The agent reads the result and adjusts its next action.

```bash
# Agent's Bash invocation
cd ~/vaults/work && dome sync
# Reads output: "adopted main: ..." or "blocked main: ... 3 diagnostics"
# Agent reacts: write a fix, re-submit
# Agent reacts: write a fix, commit it, re-sync
```

### 3. MCP read/query tools (optional, when MCP is mounted)

When a harness mounts the MCP server, read/query tools are available as typed protocol operations. v1.0 does not expose a Proposal-submission MCP tool; adoption catch-up remains Git + CLI-native. See [[wiki/specs/mcp-surface]].

Both write paths produce a Proposal internally, run the same adoption loop, and produce the same adoption result. There is no "trusted write" path that bypasses adoption.

## How a harness reads from the vault

Two ways, both grounded in adopted state:

### 1. Direct filesystem read

The agent uses its native read tool (`Read`, `cat`, `grep`, etc.) against the working tree. This reads *whatever state the working tree is in* — possibly drafts ahead of adopted. Agents that care about adopted state run `dome sync` first, then read.

For agentic harnesses (which usually operate against the user's current working tree), reading the working tree is the default. Adopted state is only meaningfully different when the user has uncommitted drafts; in that case, the agent should either submit the drafts or read the adopted ref explicitly.

### 2. `dome query` or MCP `dome.query`

Structured queries against adopted state + projection store. Returns results with SourceRefs into adopted commits. The agent prefers this when it wants evidence-backed answers ("what did I decide about X last quarter") rather than raw page contents.

```bash
cd ~/vaults/work && dome query "what's on the platform-team thread"
```

## Supported v1 harnesses

### Claude Code

The default v1 agentic harness. Configured by:

1. The vault carries `AGENTS.md` (generated by `dome init`); Claude Code's `CLAUDE.md` shim points at it (also written by `dome init`). Every session in the vault directory arrives oriented.
2. The user runs a compiler host for continuous compilation: initially `dome serve --vault <path>` in a foreground terminal or under launchd / systemd. Out-of-band writes the agent makes via `Write` / `Edit` are committed normally; the host observes branch movement; the engine adopts; the projection store updates; the log grows.
3. The agent invokes named operations via `Bash` when explicit structured ops are wanted: `dome lint --apply H1`, `dome export-context "platform ownership"`, `dome stats`, `dome query "..."`.

**MCP mounting is optional.** A user who wants the MCP surface available (e.g., to test it, or in anticipation of a future workflow that benefits from MCP-typed routing) adds the Dome MCP server to their Claude Code MCP config:

```json
{
  "mcpServers": {
    "dome": {
      "command": "bun",
      "args": ["x", "@dome/sdk", "serve", "--vault", "${VAULT_PATH}"]
    }
  }
}
```

Once configured, Dome's MCP read/query tools and prompts are available — but the agent is not expected to reach for them when native tools suffice. See [[wiki/specs/mcp-surface]] for the technical contract when MCP *is* mounted.

### Headless SDK processor invocation (for scheduled-cron, intake, view-phase commands)

Dome runs garden-phase and view-phase processors in a headless context:

- Scheduled processors (e.g., daily `dome.daily.create-daily` cron via `dome serve`'s scheduled-trigger dispatch).
- Intake processors (when a `signal:` trigger fires on a write to `inbox/<bucket>/`).
- View-phase command processors invoked via the CLI (`dome lint`, `dome query`, `dome export-context`).

Garden-phase processors that hold the `model.invoke` capability call through the engine's provider-neutral model-invoke shim. Core records provider-reported cost in the run ledger and enforces the effective model allowlist; concrete adapters such as an Anthropic or Vercel AI SDK provider are outside the `@dome/sdk` root import graph. Daily `maxDailyCostUsd` enforcement is planned on top of the existing cost ledger. The headless context is the engine running outside an interactive harness — it loads bundles, runs the processor, ledgers the run, returns.

The end-to-end harness can inject a test `ModelProvider` through `HarnessOpts.modelProvider`, and its `tick()` path now mirrors `dome sync`: after adoption, and even when HEAD is already in sync, it drains due schedule, queued-job, and outbox work against the adopted state. Scenarios can call `drainOperationalWork()` directly when they need to isolate that operational pump from drift detection.

### Other MCP-capable harnesses (Cursor, OpenCode, Codex CLI, future)

Any harness that supports `AGENTS.md` (or an equivalent orientation file) and a shell-execution tool can interact with Dome via the compiler-boundary contract above — no MCP mount required. Per-harness setup notes (where to put `AGENTS.md` if the harness's auto-load path differs from Claude Code's `CLAUDE.md` shim convention, how the harness exposes shell-execution to the agent, MCP-mounting examples for harnesses that benefit) land at `docs/wiki/sources/<harness>-setup.md` as each harness is added — see [[wiki/entities/claude-code]] for the reference example.

A harness that *also* mounts the Dome MCP server gains the MCP tool / prompt / resource surface on top of the compiler-boundary contract. The MCP surface is additive, not exclusive.

## Future-harness pressure (v2+, non-normative integration points)

These are aspirational integration points the v1 SDK design accommodates without committing to:

- **Native mobile app** — imports from `@dome/sdk` core (no LLM, no MCP) for typed markdown read, local draft writes, and adoption catch-up through the same Git-native Proposal construction path. Voice or research-style intake routes through `@dome/sdk/workflows` only when the user invokes an LLM-driven processor. Mobile is a *native Dome surface* per [[VISION]] §"Two surface patterns" — opinionated UX flows over the SDK.
- **Native desktop app** — same shape as mobile; likely an Electron / Tauri / Wails shell. Holds a long-running Vault via `openVault` → use → `vault.close()` per [[wiki/specs/sdk-surface]] §"Vault lifecycle".
- **Voice client** — captures speech via OS-native dictation, writes to `inbox/voice/*`, commits the capture through the native Git-backed surface, and lets the `dome.intake` garden-phase processor run async ingest. The ingest processor runs server-side in `@dome/sdk/workflows`.
- **Web app** — Bun's server capabilities let the SDK serve as an HTTP backend the web client speaks to. An `@dome/sdk/http` companion entrypoint adds HTTP-specific protocol adapters when the use case justifies a split (mirroring `@dome/sdk/mcp`).

**Future-harness shells construct against `AbstractSurface`.** Per [[wiki/specs/sdk-surface]] §"Consumer surfaces", core exposes a protocol-agnostic `AbstractSurface` (`submit`, `query`, `read`, `commands`, `instructions`) built by `buildAbstractSurface(vault)`; per-protocol renderers (`renderMcp` today; future `renderHttp`, `renderVoice`) project from it. A v2+ HTTP shell ships `renderHttp(surface)` in `@dome/sdk/http`, consuming the same abstract surface the MCP server consumes — only the wire format changes.

None of these change the SDK contract; they change the harness shape above it. The [[wiki/matrices/protocol-adapter]] matrix names which entrypoint each future shell reaches its symbols through; the [[wiki/invariants/ENGINE_HAS_NO_LLM_OR_MCP_DEPENDENCY]] axiom + [[wiki/gotchas/transitive-llm-dependency]] gotcha together guarantee a v2+ mobile/voice/web shell pays for only the deps it actually uses.

## What's NOT a harness

- Obsidian. Obsidian is a *browser* over the vault — it reads markdown directly. Obsidian-side edits are *native writes*; the watcher catches them and turns them into Proposals — see [[wiki/gotchas/out-of-band-vault-edits]] for the canonical-path framing.
- `git`, `vim`, the filesystem. Same as Obsidian — these are vault-readers and vault-editors that produce native writes the engine catches up via the watcher.

## What an agentic harness needs

Three things, plus one optional:

1. **`AGENTS.md` in the vault** (auto-generated by `dome init`; refreshed via the templated-section merge — today on `dome init` re-runs, v1.x via the reserved `dome doctor --repair` verb plus the planned `dome.health.agents-md-template-drift` garden-phase processor) — the orientation surface.
2. **A shell-execution capability** — so the agent can invoke `dome sync`, `dome lint`, `dome stats`, `dome query`, etc. Claude Code's `Bash` is the reference implementation.
3. **A vault path** in environment or working directory.

Optional fourth: an MCP server mount, for harnesses that benefit from explicitly-typed structured operations beyond the CLI surface.

## Why this design

The interface-agnostic principle — Dome works with *any* harness that can read a convention file and invoke shell commands — is what makes Dome durable across the agentic-tool churn. When a better harness ships (Cursor's growth, OpenCode's adoption, Claude Desktop's arrival, whatever ships in 2028), the user adopts it. Dome's value doesn't drop — the vault, the SDK, the CLI, and `AGENTS.md` are durable; only the harness changes.

The structural payoffs:

- **No lock-in to one vendor.** Dome doesn't ship its own chat surface in v1. It augments wherever the user already lives.
- **Multi-harness usage is natural.** Same vault, Claude Code on one machine and Cursor on another. Both read `AGENTS.md`; both can shell out to the CLI. Their MCP mounts (if any) are independent.
- **The SDK is the contract internally; the compiler boundary is the contract externally.** Internal Dome flows (garden-phase processors compiling raw captures, scheduled processors running cron-driven jobs) go through the engine's adoption loop with full capability enforcement. External writes from any consumer shell are turned into Proposals by the watcher and adopted through the same loop. One pattern, one set of guarantees.

System prompts and orientation are markdown files, not harness-specific code. `AGENTS.md` is the orientation surface; every harness that honors the `AGENTS.md` convention picks it up the same way. Per-harness setup is documentation, not SDK code — the SDK doesn't auto-configure anything; that would create harness-version coupling.

For v2+, this design lets native mobile / desktop / web / voice clients sit on the same SDK contract via direct import. Cross-AI handoff (`dome export-context`) extends the same principle to "send context to a different agent product entirely."
