# Harnesses

This spec is normative for how external agentic environments interact with Dome. A *harness* is a runtime that hosts an agent loop and operates against a Dome-managed markdown vault. The harness is the SDK's user, not part of it.

Dome's relationship to harnesses is asymmetric. Native Dome surfaces (mobile, desktop, web, voice — see [[VISION]] §"Two surface patterns") embed the SDK and ship designed UX. **Agentic harnesses** (Claude Code, Cursor, OpenCode, Codex, future agents) are brought by the user; they're general-purpose and use whatever tools they have natively. The spec below governs the latter class: how *agentic harnesses* connect to a Dome vault.

## The compiler-boundary contract

Dome doesn't dictate which tools an agentic harness reaches for. Claude Code with full filesystem access uses `Read` / `Grep` / `Write` / `Edit` natively; a hypothetical sandboxed harness might prefer MCP-routed reads. Dome's contract with every harness is **the compiler boundary** — four surfaces an agentic harness interacts with, regardless of its native tool choice:

1. **`AGENTS.md` at the vault root** — auto-loaded by Claude Code (via `CLAUDE.md` shim) and any harness that supports the `AGENTS.md` convention. Teaches the agent the vault's conventions, page types, named workflows, and invariants at session start. Generated by `dome init`; templated sections refreshed by `dome doctor --repair`; user-prose sections preserved via delimiter convention. See [[wiki/invariants/AGENTS_MD_IS_ORIENTATION_SURFACE]].
2. **The CLI** — `dome lint`, `dome lint --apply <id>`, `dome stats`, `dome doctor`, `dome export-context <topic>`, `dome migrate`, `dome reconcile`. Invokable from any shell. The agent calls these via whatever shell-execution tool the harness exposes (`Bash` in Claude Code). See [[wiki/specs/cli]].
3. **The compiler daemon** — `dome serve` runs as a background process (launchd / systemd / sometimes ad-hoc); the watcher catches native writes, hooks fire on each event, scheduled hooks fire on their intervals. The agent doesn't interact with the daemon directly; the daemon interacts with the vault.
4. **`dome reconcile`** — explicit catch-up after the daemon was off or after an out-of-band sync (`git pull`). The agent (or the user) invokes this when needed.

These four surfaces are the *load-bearing* contract. The MCP server ([[wiki/specs/mcp-surface]]) is preserved as a fifth, non-primary surface — available for future harnesses that benefit from explicitly-typed MCP-routed operations, but not load-bearing for v0.5 value delivery.

## Supported v0.5 harnesses

### Claude Code

The default v0.5 agentic harness. Configured by:

1. The vault carries `AGENTS.md` (generated by `dome init`); Claude Code's `CLAUDE.md` shim points at it (also written by `dome init`). Every session in the vault directory arrives oriented.
2. The user runs `dome serve --vault <path>` as a launchd / systemd service for continuous compilation. Out-of-band writes the agent makes via `Write` / `Edit` get caught by the watcher; hooks fire; the index updates; the log grows.
3. The agent invokes named operations via `Bash` when explicit structured ops are wanted: `dome lint --apply H1`, `dome export-context "platform ownership"`, `dome stats`.

**MCP mounting is optional.** A user who wants the MCP surface available (e.g., to test it, or in anticipation of a future workflow that benefits from MCP-typed routing) adds the Dome MCP server to their Claude Code MCP config:

```json
{
  "mcpServers": {
    "dome": {
      "command": "bun",
      "args": ["x", "@dome/sdk", "serve", "--vault", "${VAULT_PATH}"]
    }
  }
}
```

Once configured, Dome's MCP tools and prompts are available — but the agent is not expected to reach for them when native tools suffice. See [[wiki/specs/mcp-surface]] for the technical contract when MCP *is* mounted.

### Headless SDK loop (for `dome lint`, `dome export-context`, intake hooks)

Dome ships a minimal headless agent loop in `@dome/sdk` for non-interactive contexts:

- Scheduled workflows (e.g., daily `lint` via cron or `dome serve`'s scheduled-hook dispatch).
- Intake hook handlers (when a declarative hook invokes a workflow against a dropped file).
- CLI commands that map to workflows (`dome lint`, `dome export-context <topic>`).

This loop uses `@ai-sdk/anthropic` directly. It loads the named workflow prompt, binds the listed Tools, runs the loop until the LLM produces a stop turn, then returns. It is intentionally minimal — not a chat surface; not interactive; one workflow per invocation.

### Other MCP-capable harnesses (Cursor, OpenCode, Codex CLI, future)

Any harness that supports `AGENTS.md` (or an equivalent orientation file) and a shell-execution tool can interact with Dome via the compiler-boundary contract above — no MCP mount required. Per-harness setup notes (where to put `AGENTS.md` if the harness's auto-load path differs from Claude Code's `CLAUDE.md` shim convention, how the harness exposes shell-execution to the agent, MCP-mounting examples for harnesses that benefit) land at `docs/wiki/sources/<harness>-setup.md` as each harness is added — see [[wiki/entities/claude-code]] for the reference example. The setup notes are vault-author-contributed sources, not normative SDK output; the SDK doesn't ship harness-specific configuration.

A harness that *also* mounts the Dome MCP server gains the MCP tool / prompt / resource surface on top of the compiler-boundary contract. The MCP surface is additive, not exclusive.

## Future-harness pressure (v1+, non-normative integration points)

These are aspirational integration points the v0.5 SDK design accommodates without committing to:

- **Native mobile app** — imports from `@dome/sdk` core (no LLM, no MCP) for typed markdown read/write. Voice or research-style intake routes through `@dome/sdk/workflows` only when the user invokes an LLM-driven workflow. Mobile is a *native Dome surface* per [[VISION]] §"Two surface patterns" — opinionated UX flows over the SDK.
- **Native desktop app** — same shape as mobile; likely an Electron / Tauri / Wails shell. Holds a long-running Vault via `openVault` → use → `vault.close()` per [[wiki/specs/sdk-surface]] §"Vault lifecycle".
- **Voice client** — captures speech via OS-native dictation, writes to `inbox/voice/*`, lets the intake hook run async ingest. The capture shell needs only `@dome/sdk` core (`writeDocument`); the ingest workflow runs server-side in `@dome/sdk/workflows`.
- **Web app** — Bun's server capabilities let the SDK serve as an HTTP backend the web client speaks to. v1+ decision; an `@dome/sdk/http` companion entrypoint may add HTTP-specific protocol adapters when the use case justifies a split (mirroring `@dome/sdk/mcp`).

**Future-harness shells construct against `AbstractSurface`.** Per [[wiki/specs/sdk-surface]] §"Consumer surfaces", core exposes a protocol-agnostic `AbstractSurface` (`tools`, `prompts`, `resources`, `instructions`) built by `buildAbstractSurface(vault)`; per-protocol renderers (`renderMcp` today; future `renderHttp`, `renderVoice`) project from it. A v1+ HTTP shell ships `renderHttp(surface)` in `@dome/sdk/http`, consuming the same abstract surface the MCP server consumes — only the wire format changes.

None of these change the SDK contract; they change the harness shape above it. The [[wiki/matrices/consumer-surface]] matrix names which entrypoint each future shell reaches its symbols through; the [[wiki/invariants/CORE_HAS_NO_LLM_OR_MCP_DEPENDENCY]] axiom + [[wiki/gotchas/transitive-llm-dependency]] gotcha together guarantee a v1+ mobile/voice/web shell pays for only the deps it actually uses.

## What's NOT a harness

- Obsidian. Obsidian is a *browser* over the vault — it reads markdown directly, without going through Dome's Tools. This is by design (`MARKDOWN_IS_SOURCE_OF_TRUTH`). Obsidian-side edits are *out-of-band* writes; the watcher catches them and `dome reconcile` catches up anything missed — see [[wiki/gotchas/out-of-band-vault-edits]] for the canonical-path framing.
- `git`, `vim`, the filesystem. Same as Obsidian — these are vault-readers and vault-editors that bypass Tools by design. The compiler reconciles whatever they write.

## What an agentic harness needs

Three things, plus one optional:

1. **`AGENTS.md` in the vault** (auto-generated by `dome init`; refreshed by `dome doctor --repair`) — the orientation surface.
2. **A shell-execution capability** — so the agent can invoke `dome lint`, `dome stats`, `dome export-context`, etc. Claude Code's `Bash` is the reference implementation.
3. **A vault path** in environment or working directory.

Optional fourth: an MCP server mount, for harnesses that benefit from explicitly-typed structured operations beyond the CLI surface.

## Why this design

The interface-agnostic principle — Dome works with *any* harness that can read a convention file and invoke shell commands — is what makes Dome durable across the agentic-tool churn. When a better harness ships (Cursor's growth, OpenCode's adoption, Claude Desktop's arrival, whatever ships in 2027), the user adopts it. Dome's value doesn't drop — the vault, the SDK, the CLI, and `AGENTS.md` are durable; only the harness changes.

The structural payoffs:

- **No lock-in to one vendor.** Dome doesn't ship its own chat surface in v0.5. It augments wherever the user already lives.
- **Multi-harness usage is natural.** Same vault, Claude Code on one machine and Cursor on another. Both read `AGENTS.md`; both can shell out to the CLI. Their MCP mounts (if any) are independent.
- **The SDK is the contract internally; the compiler boundary is the contract externally.** Even when the user invokes a workflow from inside Dome's own headless agent loop (for `dome lint` or `dome export-context`), the internal path goes through Dome's Tools and invariants. External writes from any consumer shell flow through the watcher + reconcile + hooks loop. Two patterns, clearly scoped (see [[VISION]] §"Principles" #3 "Invariants are enforced two ways, by scope").

System prompts and orientation are markdown files, not harness-specific code. `AGENTS.md` is the orientation surface; every harness that honors the `AGENTS.md` convention picks it up the same way. Per-harness setup is documentation, not SDK code — the SDK doesn't auto-configure anything; that would create harness-version coupling.

For v1+, this design lets native mobile / desktop / web / voice clients sit on the same SDK contract via direct import. Cross-AI handoff (`dome export-context`) extends the same principle to "send context to a different agent product entirely."
