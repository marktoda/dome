# v0.5 → v1 → v2+ build plan

The phasing for Dome over time. v0.5 was the demo; v1 is the engine model (the four-concept core unified around Vault / Proposal / Processor / Effect); v2+ is the product; long-term is what the product becomes when it works.

## v0.5 — Demo (shipped, archival)

**Goal:** Prove the original four-concept core (Vault / Document / Tool / Hook) compiles in real use. The author was the first user.

**Shipped:**

- TypeScript SDK on Bun. Single npm package with entrypoint splits (`@dome/sdk` core, `@dome/sdk/workflows`, `@dome/sdk/mcp`, `@dome/sdk/cli`).
- The original four concepts: Vault, Document, Tool, Hook.
- Seven Tools (read/write/append/search/wikilink/move/delete).
- Named invariants across axiom / shipped-default / opt-in tiers; the canonical list lived at `docs/wiki/invariants/`.
- Hook system: programmatic + declarative forms; async by default; cycle detection; drop-zone intake pattern.
- Prompt library: shipped workflow prompts.
- MCP server: one MCP tool per SDK tool.
- CLI: the shipped command surface.
- Eval suite shape (fixtures + expected effects).
- The `dome.dailies` extension bundle as the proof of concept that extensions can package page-types + preambles + workflows + hooks + CLI commands as one coherent unit.
- Adopted-ref + Dome-* trailers (v0.5+phase1+phase3): first-class git refs replace the per-machine cursor file; engine commits carry four trailers in their message bodies.

**Validation:** The author used Dome for `~/vaults/work` and `~/vaults/personal` for several months. Real PMing happened against real workflow friction.

## v1 — The engine model (current)

**Goal:** Unify the substrate around a smaller core. Make the design implementable from documentation alone; make the trust boundary structural rather than reviewer-attention-based.

**The shape change:** The original four concepts (Vault, Document, Tool, Hook) reduce to a new four (Vault, Proposal, Processor, Effect). Tools, Hooks, Workflows dissolve into Processors that emit Effects. The fixed-point adoption loop replaces the three-phase reconcile; a Bun.sqlite-backed projection store, run ledger, and outbox land as engine infrastructure; a capability broker enforces effect emission at one chokepoint. Core features become first-party `dome.*` extension bundles registered through the same path third-party bundles use.

**Ships:**

- The four-concept core: Vault, Proposal, Processor, Effect (per [[wiki/specs/sdk-surface]]).
- The seven-kind Effect taxonomy: Patch / Diagnostic / Fact / Question / Job / ExternalAction / View (per [[wiki/specs/effects]]).
- The three processor phases: adoption / garden / view (per [[wiki/specs/processors]]).
- The fixed-point adoption loop with MAX_ITER cap, divergence diagnostic, atomic closure commit (per [[wiki/specs/adoption]]).
- Eight capability tiers enforced at one engine chokepoint (per [[wiki/specs/capabilities]]).
- Bun.sqlite-backed projection store, run ledger, outbox (per [[wiki/specs/projection-store]], [[wiki/specs/run-ledger]]).
- First-party extension catalog: `dome.markdown`, `dome.index`, `dome.log`, `dome.links`, `dome.intake`, `dome.daily`, `dome.lint`, `dome.search`, `dome.migrate` (per [[wiki/matrices/built-in-extensions-x-phase]]).
- CLI surface (canonical list at [[wiki/specs/cli]] §"The CLI surface"): `dome sync`, `dome status`, `dome query`, `dome inspect <subject>`, `dome lint`, `dome rebuild`, `dome stats`, `dome serve`, `dome init`, `dome migrate`, `dome run-processor`, `dome export-context`, plus `dome doctor` and `dome answer` reserved for v1.x. (`dome submit` was retired in Phase 11a.)
- MCP server as one Recall+Submit protocol adapter over `AbstractSurface` (per [[wiki/specs/mcp-surface]]).
- Hosted-protected mode designed-for in [[wiki/specs/adoption]] §"Hosted-protected mode" — deferred to v1.5.

**Harnesses:**

- Claude Code (primary).
- Other MCP-capable agents (Cursor, OpenCode, Codex, future) work via `AGENTS.md` + shell-execution capability; MCP mounting optional.

**Browser:** Obsidian. The vault is plain markdown.

## v1.5 — Hosted multi-client

**Goal:** Make the engine work in hosted mode where PRs against `main` are Proposals.

**Ships:**

- Engine `--hosted` mode: webhook-driven Proposal construction from PR events; engine adoption runs in CI; closure commits land on the PR; PR auto-merges or routes to review based on capability policy.
- Multi-client coordination (concurrent Proposals from voice / desktop / mobile clients merge through the same adoption loop).

**Audience:** Small teams or solo users with multiple devices. The hosted vault lives on Gitea / GitHub; clients push branches.

## v2 — Product

**Goal:** Anyone can use Dome to extend their brain.

**Ships:**

- Native mobile app (iOS first; voice-first capture; structured browse; prep mode; inbox review).
- Native desktop app (electron / tauri / wails shell; same surfaces as mobile + power-user TUI option).
- Voice client (AirPods → inbox/voice → garden-LLM processor).
- Web app (`@dome/sdk/http` adapter over `AbstractSurface`).
- Onboarding flow that meets users where they are: blank vault, Obsidian vault, Apple Notes, Google Docs.
- Polished `dome migrate` for cross-version vault upgrades.
- Third-party plugin marketplace / discovery surface.
- Refined garden-LLM prompts based on v1 dogfooding + user-research-driven additions.

**Audience:** The wedge expands from the author to high-context operators (managers, founders, researchers, consultants), then broadens. The principles ([[wiki/invariants/MARKDOWN_IS_SOURCE_OF_TRUTH]], [[wiki/concepts/brain-companion]]) are structural; they make Dome work for anyone.

## Long-term — Always-with-you

**Goal:** Dome is what people use the way they use their phones.

The vision (from [[VISION]] §"Long term"): ambient, always there, low-friction. You think out loud; Dome remembers. You ask; Dome answers from your own thinking. The garden grows over years because the gardening is automatic. The cognitive surface of an individual — what they pay attention to, what they decide, what they change their mind about — becomes a durable, exportable, queryable asset that compounds for a lifetime.

What this requires beyond v2:

- Cross-device sync that handles concurrent Proposals well at scale ([[wiki/gotchas/concurrent-harness-write]] generalized).
- Temporal awareness — "what was I thinking 6 weeks ago"; a `wiki/claims/` page type if it earns its keep.
- Native voice capture at the OS level (iOS / macOS dictation deeply integrated).
- An ecosystem of third-party `dome.*` bundles for different domains (writers, researchers, family historians, founders).
- The market trusting that Dome's local-first / markdown-canonical promise will hold over years.

The long-term vision is a north star, not a 2027 product spec. It guides v2+ feature decisions: anything that adds proprietary lock-in or breaks the markdown-canonical contract is out of bounds, even if it would ship faster.
