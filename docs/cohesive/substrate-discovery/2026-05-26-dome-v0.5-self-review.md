### Behavior matrices

- **Existing:** `tool-invariant-enforcement`, `event-types-and-payloads`, `intent-prompt-tools`.
- **Missing but likely needed:** a **harness-surface × consumer-need** matrix that names, for each anticipated v1+ consumer (mobile/desktop/voice/web), what part of `@dome/sdk` they require (Vault, raw Tools, AI-SDK ToolSet, MCP adapters, workflow runner, prompt registry). ### Known gotchas / scars

- **`concurrent-harness-write`** — caller-supplied optimistic locking via `expected_mtime` (commit ca12413, fully implemented in `tools/guards.ts`).
- **`hook-cycle`** — per-(handler, target) repetition check + depth safety net at `hook-dispatcher.ts:119`. AsyncLocalStorage carries causation across re-entry (commit 6c830a4).
- **`agent-prompt-regression`** — eval suite at `src/eval/replay.ts` + fixture vaults at `src/eval/fixture-vault.ts`; marked `@internal` per `src/index.ts:71-73`.

- **`dirty-git-state-at-reconcile`** — `reconcile.ts:isDirtyGitState` refuses MERGE_HEAD / rebase-merge / rebase-apply / CHERRY_PICK_HEAD.

### Locality boundaries

- **Subsystem A seams:**
  - `@dome/sdk` (core) vs `@dome/sdk/cli` (shell) — `package.json` `exports` enforces the bundle split; recent commit 9d7ca1f.
  - `src/tools/registry.ts` is the **canonical Tool registry**; every downstream catalog derives from it (`BoundToolSurface`, `aiTools`, `toolParsers`, `MCP_TOOL_NAMES`, MCP adapter list, workflow-frontmatter Zod enum). Adding an 8th Tool used to touch 9 files; now it's two (new `src/tools/<name>.ts` + one registry entry). Recent commits 36c5e1b, b50b9dc, 463aae8 all converge on this single-source-of-truth pattern.
  - `src/shipped-defaults.ts` is the canonical source for `SHIPPED_VAULT_CONFIG` and `SHIPPED_PAGE_TYPES` (commit 463aae8 collapsed 4 copies into one).
  - `src/privileged-writer.ts` is private (deliberately not exported from `src/index.ts:29-37`); the public seam is `vault.rebuildIndex()` at `vault.ts:203`.
- **Suspected boundary tensions (v1 pressure):**
  - **Workflows + LLM are exported from core.** `src/index.ts:54-69` re-exports `runWorkflow`, `buildAiSdkTools`, `DEFAULT_MODEL`, `WorkflowRegistry`, `PromptLoader` directly from the core surface. A consumer shell that doesn't want LLM-driven workflows (a deterministic-only mobile-side Vault writer) cannot opt out without bundling `@anthropic-ai/sdk` and `ai`.
  - **MCP is exported from core.** `src/index.ts:86-93` similarly re-exports `DomeMcpServer`, `buildToolAdapters`, `buildPromptAdapters`, `ResourceAdapter`. A consumer that doesn't speak MCP (a direct-SDK web client over HTTP) bundles MCP regardless.
  - **The `vault.aiTools` field is on every Vault.** `Vault` interface at `vault.ts:51` carries `aiTools: import("ai").ToolSet` and `toolParsers` unconditionally — wired in via `bindTools(vault, writer, wrapMutation)` at `tools/registry.ts:264`. Every consumer pays for the AI-SDK Tool shape, even those that only invoke `vault.tools.*` directly.
- **Suspected premature centralization:**
  - None obvious. The single-source-of-truth Tool registry, `shipped-defaults`, and the unified `BoundToolSurface` are recent simplifications that already collapsed prior duplications.
- **Suspected duplication-that-wants-abstraction:**
  - The CLI's three workflow-driven commands (`lint`, `migrate`, `export-context`) all reduce to `runWorkflowAtPath` (commit e841e3e) — recently consolidated.
  - The MCP server stitches together `tools` + `prompts` + `resources` + `instructions` in `mcp/server.ts:18-37`. The "surface" pattern (a thing that aggregates the four kinds the SDK exposes — Tools, prompts/workflows, resources, instructions — for one consumer) does not yet have a name. v1+ consumers will repeat this aggregation.

### Missing memory

Ranked by leverage — what would prevent the most predictable future bug or scope-creep mistake when a v1 contributor (or the agent itself, six months from now) makes a change.

- **No named invariant for the core/shell boundary.** `src/index.ts:96-99` describes the split in a comment ("The CLI shell surface ... lives at a separate entrypoint"), and `package.json` `exports` enforces it. But there is no `CORE_HAS_NO_LLM_DEPENDENCY` or `CONSUMER_SHELL_BOUNDARY_IS_NAMED` invariant document, no matrix that lists "which import surfaces are core vs. shell," and no `dome doctor` check that would catch a future PR that re-exports `runWorkflow` from `cli/index.ts` or pulls a CLI-only symbol back into `src/index.ts`. The substrate that would close this: a named invariant in `docs/wiki/invariants/CORE_AND_SHELL_BOUNDARIES.md` declaring (a) `@dome/sdk` may not depend on Commander or any CLI-only library, (b) `@dome/sdk/cli` is the only consumer of pre-flight error kinds beyond `ToolError`, (c) future consumer shells (mobile/desktop/voice/web) follow the same shape; plus a `dome doctor` check that enumerates exports per entrypoint and asserts the disjoint sets.

- **No matrix for "what each v1 consumer needs from the SDK."** `wiki/specs/harnesses.md` §"Future-harness pressure" lists four consumers (mobile, desktop, voice, web) at lines 62-69; `wiki/syntheses/v0.5-build-plan.md` §"v1+ — Product" repeats this. Neither says which of the SDK's exported symbols each consumer reaches for, and which it doesn't. A `wiki/matrices/consumer-surface.md` with rows = consumer shells, columns = exported symbol families (Vault, Tools, Hooks, MCP, AI-SDK ToolSet, Workflows, Prompts, Eval, Privileged-writer) would (a) ratify that the core surface is small enough to embed in a mobile binary, (b) flag now (not at v1) whether `runWorkflow` belongs in core or in a separate `@dome/sdk/workflows` entrypoint.

- **Invariant count drift.** Delta ledger at line 189 (`docs/cohesive/delta-ledgers/2026-05-25-dome-v0.5-foundation.md`) states `Invariants ... — split 5 axiom + 3 default + 2 opt-in`. Live state: 6 axiom + 4 shipped-default + 2 opt-in = 12 total (per `src/types.ts:68-81` `INVARIANTS` const and `tests/integration/public-surface.test.ts` line ~30 which asserts "12 named invariants are enumerated in INVARIANTS const"). The artifact that would close this: a sync pass on the delta ledger §"Updated substrate counts" table, OR a normative-doc-vs-code count check in `dome doctor` (the gotcha here is **substrate-vs-code count drift**, a generalization of the gotcha `dome doctor` would catch if it surfaced per-invariant `INVARIANTS` membership against `docs/wiki/invariants/*.md` filenames).

- **`v0.5-build-plan.md` says "CLI: 5 commands" but lists 6, while live state has 7.** `docs/wiki/syntheses/v0.5-build-plan.md:27` reads `CLI: 5 commands (init, migrate, serve, lint, doctor, export-context)` — six items, claimed as five, missing `reconcile` (which was added in the durability pass per the delta ledger and is in `wiki/specs/cli.md` and `src/cli/cli.ts`). The artifact that would close this: edit the build plan to read `CLI: 7 commands (init, migrate, serve, reconcile, lint, doctor, export-context)`, OR (better) excise the inline count and link to `wiki/specs/cli.md` §"Implementation note" as the canonical table. The pattern of inline counts diverging from the canonical surface is exactly the **substrate-wide drift-resistance** concern the pass-2 review introduced; this is its first concrete violation.

- **No gotcha for "consumer pulls Anthropic + MCP unintentionally."** `src/index.ts:43-93` re-exports `runWorkflow`, `WorkflowRegistry`, `DEFAULT_MODEL`, `DomeMcpServer`, etc. A v1 web consumer that wants only the read-side (search, link-resolve, read) on a thin HTTP wrapper will, today, transitively pull `@anthropic-ai/sdk` and `@modelcontextprotocol/sdk` into its bundle because `import { openVault } from "@dome/sdk"` reaches `vault.ts` which imports the AI SDK via `bindTools`. The gotcha that would close this: `docs/wiki/gotchas/transitive-llm-dependency.md` naming the symptom (every consumer's bundle carries Anthropic), root cause (`Vault.aiTools` is always built), and the mitigation paths (lazy-init via getter; OR split `@dome/sdk` into `@dome/sdk/core` and `@dome/sdk/llm` like the CLI shell was split).

- **No spec for the `Vault` lifecycle.** `wiki/specs/sdk-surface.md` §"Vault" lines 15-37 describes `openVault` but says nothing about close/teardown. `domeServe` at `cli/commands/serve.ts:60-65` returns `{ stop: async () => { await watcher.stop(); } }` — the watcher is stopped but the Vault itself is not. There is no `vault.close()` and no spec saying whether `drainHooks` is idempotent (`hook-dispatcher.ts:152` shows it just waits for `queue.onIdle()`, so re-calling is benign — but this is implementation, not spec). The gotcha that would close this: a `Vault lifecycle` section in `sdk-surface.md` declaring open / drain / close semantics, especially relevant for v1 long-running mobile/desktop processes.

- **No semantic linter despite the spec carrying the surface.** `wiki/specs/sdk-surface.md` §"Why this design" + the delta ledger §"Deferred" both call out that semantic linters are deferred. `dome lint` (the LLM workflow) is shipped. But the substrate-prescribed semantic-linter rules (e.g., `dome-lint-workflow-tool-list-against-catalog`, `dome-lint-stale-inbox`, the count-drift check above) have no formal rule docs. The artifact that would close this: a single `docs/wiki/specs/semantic-linters.md` declaring (a) the rule format (name, severity, what it checks, what closes it), (b) the canonical rule list, (c) how `dome doctor` and `dome lint` differ in their semantic-rule surfaces.

- **`PROMPTS_ARE_CONTRACT` retirement is in the ledger but only obliquely cited from the live substrate.** Delta ledger line 17 records the retirement; `wiki/specs/sdk-surface.md` §"Why this design" line 314 mentions "prompts-as-contract" as a principle. There is no live mention of `PROMPTS_ARE_CONTRACT` as a retired name anywhere a reader of the wiki alone (without reading the cohesive substrate) could discover. The artifact that would close this: a one-line note in `wiki/specs/sdk-surface.md` §"Why this design" — "`PROMPTS_ARE_CONTRACT` was an early candidate invariant; it is now expressed as the architectural principle below, not as a per-vault enforceable rule" — so the substrate itself records the retirement.
