## Approved direction (verbatim from the brainstorm)

Dome v0.5 is a TypeScript SDK (Bun runtime) with a four-concept core (Vault, Document, Tool, Hook) and four extensibility channels (hooks with both programmatic and declarative registration forms — drop-zone intakes being the most common declarative use case; custom page types; per-vault overrides; plugin packages). Tools enforce 10 named invariants at call site; hooks run async via event queue. MCP server makes Claude Code (and any MCP-capable harness) Dome-aware. Small CLI (5 commands) for side-doors. No proprietary UI in v0.5; the mass-market mobile/web product (v1+) ships proprietary clients over the same SDK contract. The project's own `docs/` is restructured as a Dome vault — dogfood from day one, proof of generality.

## Per-file changes

### Specs (8 added)

- **`docs/wiki/specs/sdk-surface.md`** — Normative spec for the public SDK API. Defines Vault, Document, Tool, Hook; catalogs 10 default Tools with invariant-enforcement assignments; defines the layered registration mechanism (3 sources × 5 registration kinds); enumerates the anti-concept list (no separate Workflow / Agent / Event / Plugin types).
- **`docs/wiki/specs/vault-layout.md`** — Normative spec for on-disk vault shape. Defines directory tree, category derivation from path, ownership rules per category, multi-vault model (vault-agnostic SDK).
- **`docs/wiki/specs/page-schema.md`** — Normative spec for frontmatter contract. Universal frontmatter for all wiki pages + type-specific extensions for entity / concept / source / synthesis + the extension type declaration mechanism.
- **`docs/wiki/specs/hooks.md`** — Normative spec for the Hook mechanism. Two registration forms (programmatic .ts + declarative .yaml). Event taxonomy derived from Tool Effects. Async-by-default execution with sync opt-in. Cycle prevention via depth limit. Folds in intake-as-hook pattern; folds in event taxonomy that pre-collapse had its own spec.
- **`docs/wiki/specs/prompts-and-workflows.md`** — Normative spec for the prompt library. A prompt with workflow frontmatter IS a workflow (the post-collapse design). Lists shipped workflows: ingest, voice-ingest, query, lint, research, sensitivity-classify, export-context. Override layering: SDK → plugins → vault-local. Eval suite as the prompt-regression mitigation.
- **`docs/wiki/specs/harnesses.md`** — Normative spec for how agentic harnesses mount Dome. Claude Code as the primary v0.5 harness; the headless SDK loop for batch / scheduled workflows; other MCP-capable harnesses (Cursor, OpenCode, Codex CLI) supported via the same MCP surface. What's not a harness (Obsidian, vim, git).
- **`docs/wiki/specs/mcp-surface.md`** — Normative spec for the MCP server. One MCP tool per SDK tool. MCP prompts expose workflow prompts. MCP resources expose vault content (read-only via MCP; writes go through Tools). stdio default + future HTTP/SSE.
- **`docs/wiki/specs/cli.md`** — Normative spec for the 5-command CLI: `dome init`, `dome migrate`, `dome serve`, `dome lint`, `dome doctor`, `dome export-context`. Explicit list of what's NOT a CLI command (capture, chat, browse, query) with the reasoning.

### Invariants (10 added)

Each invariant doc states: invariant name, normative statement, why it's load-bearing, structural enforcement mechanism (which Tool / linter), counter-example, test guarantee, related.

- **`docs/wiki/invariants/RAW_IS_IMMUTABLE.md`** — `raw/` files immutable after creation; `appendRawLinkedPage` is the single documented exception.
- **`docs/wiki/invariants/MARKDOWN_IS_SOURCE_OF_TRUTH.md`** — All derived state rebuildable from markdown; no proprietary database.
- **`docs/wiki/invariants/LOG_IS_APPEND_ONLY.md`** — `log.md` mutated only by `appendLog`.
- **`docs/wiki/invariants/EVERY_WRITE_IS_LOGGED.md`** — Every mutating Tool emits an appendLog Effect.
- **`docs/wiki/invariants/PAGE_TYPE_BY_DIRECTORY.md`** — Page type derived from immediate `wiki/` subdirectory; `.dome/page-types.yaml` declares allowed set; frontmatter and directory are redundant by design.
- **`docs/wiki/invariants/SENSITIVE_GOES_TO_INBOX.md`** — Sensitive-classified content routes via `routeSensitiveToInbox` to `inbox/review/`, never direct `writePage`.
- **`docs/wiki/invariants/WIKILINKS_ARE_FULLPATH.md`** — `[[wiki/entities/x]]` not `[[x]]`; `writePage` rejects short-form; `wikilinkResolve` returns null on short-form.
- **`docs/wiki/invariants/PROMPTS_ARE_CONTRACT.md`** — Behavior changes go through prompt edits, not code edits; Tools stay content-agnostic.
- **`docs/wiki/invariants/PAGE_CREATION_REQUIRES_RECURRENCE.md`** — New-page creates require explicit `reason: recurring | named_explicitly | structural`; prevents page explosion.
- **`docs/wiki/invariants/HOOKS_CANNOT_BYPASS_TOOLS.md`** — Hooks observe events and call Tools; they never mutate the vault directly; type system enforces at compile time; out-of-band hook writes detected at runtime.

### Matrices (3 added)

- **`docs/wiki/matrices/tool-invariant-enforcement.md`** — 10-tool × 10-invariant matrix with enforcement mechanism per cell. Every bolded cell maps to a `tests/invariants/<INVARIANT>.test.ts` test.
- **`docs/wiki/matrices/intent-prompt-tools.md`** — User intent × workflow prompt × bound tool subset × expected effects. Documents the intent-routing logic that lives in `system-base.md`.
- **`docs/wiki/matrices/event-types-and-payloads.md`** — Canonical event taxonomy. Effect-derived events + internal lifecycle events. Pattern-matching rules. Why events are derived not fired.

### Gotchas (6 added)

Each gotcha doc names the symptom, root cause, structural mitigation, specific scenarios, operational notes, related substrate.

- **`docs/wiki/gotchas/agent-prompt-regression.md`** — High severity. Model upgrades or prompt edits can change behavior silently. Mitigation: eval suite with fixtures + expected-effect assertions.
- **`docs/wiki/gotchas/multi-page-partial-write.md`** — Medium. Ingest that updates 7 pages fails after 3. Mitigation: collect-and-apply with git rollback.
- **`docs/wiki/gotchas/out-of-band-vault-edits.md`** — Low. Obsidian / vim edits don't go through Tools. By-design tolerance; `dome doctor` detects drift.
- **`docs/wiki/gotchas/concurrent-harness-write.md`** — Medium. Two harness sessions race on the same page. Mitigation: timestamp-based optimistic locking.
- **`docs/wiki/gotchas/hook-cycle.md`** — Medium. Hook A → Tool → event → Hook A again. Mitigation: causation-chain depth limit (configurable, default 5).
- **`docs/wiki/gotchas/async-read-after-write-staleness.md`** — Low. Read immediately after write may not see hook follow-on. Mitigation: sync-mode opt-in for hooks that must complete first.

### Vault seeds (18 added)

7 entities + 8 concepts + 1 source + 2 syntheses. Each is a small, structured stub establishing the page exists with the right type, frontmatter, and cross-references — content grows over time as the vault is used.

### Vault config (4 added)

- **`docs/.dome/page-types.yaml`** — Declares 4 default types + 4 extension types (spec, invariant, matrix, gotcha) for this vault.
- **`docs/.dome/config.yaml`** — Project-specific config. Disables `PAGE_CREATION_REQUIRES_RECURRENCE` (specs authored explicitly) and `SENSITIVE_GOES_TO_INBOX` (project vault, no sensitive content). Declares async hook mode default.
- **`docs/index.md`** — Catalog of all wiki pages, grouped by type, sorted alphabetically.
- **`docs/log.md`** — Append-only operation history. Bootstrapped with one entry recording the vault creation.

## Positions retired from the original architecture

The seed at `docs/raw/original-architecture.md` (preserved verbatim) carried positions that the brainstorm revised:

| Original position | Revised position | Reason |
|---|---|---|
| `PAGES_USE_ONE_SCHEMA` — all wiki pages use one generic format | `PAGE_TYPE_BY_DIRECTORY` — four typed pages by directory | The user's actual `~/vaults/work/` vault evidenced four types in practice (137 entities / 90 concepts / 57 sources / 12 syntheses). The "one schema" idea was aspirational; reality is four. |
| `ATOMIC_IDEAS_ARE_INTERMEDIATE` (with deferred "yet" in ARCHITECTURE.md `:232`) | Skip atom materialization in v0.5; agent extracts atoms internally; can add `wiki/claims/` later | The brainstorm decided v0.5 doesn't need persisted atoms; the `wiki/claims/` extension stays available for v1+ if "what have I changed my mind about" queries demand it. |
| `/wiki/inbox.md` as a single destination for sensitive content | `inbox/review/` directory (a drop-zone intake) routed via `route_sensitive_to_inbox` Tool | The collapse of intakes-into-hooks made `wiki/inbox.md` a special case of the generalized inbox/<bucket>/ pattern. The Tool is the structural enforcement; the directory is one of many intake buckets. |
| 5 extensibility channels (intakes + hooks + page-types + per-vault overrides + plugins) | 1 hook mechanism (with programmatic + declarative forms) + 3 other registration kinds + 3 sources | Six structural collapses ratified mid-brainstorm: intake→hook, workflow→prompt-w-frontmatter, plugin→packaging, agent→harness, event→derived-from-Effect, document.category→derived-from-path. Reduces conceptual surface without losing capability. |
| Python as the implementation language | TypeScript on Bun | User pushback: better familiarity, richer type system for Tool/Effect contracts, same-language alignment with future web/mobile, Anthropic + MCP first-class TS support. The TUI-via-Textual argument collapses because v0.5 ships no proprietary TUI. |

These revisions are documented in `docs/cohesive/brainstorms/2026-05-25-dome-vision.md` decision dialog. The original seed is preserved unchanged so the lineage stays auditable.

## Durability + git-as-axiom pass (mid-rewrite, ratified by user 2026-05-25)

After the simplification pass, the user pressure-tested the durability story for hooks and asked for an end-to-end design. Key insight: log.md-as-event-source mixes audit with execution state and contradicts MARKDOWN_IS_SOURCE_OF_TRUTH in spirit. State-based reconciliation (filesystem + git diff) is cleaner.

Then user locked **git as a hard requirement** (not just recommended), which removed the hash-cache fallback path and collapsed the reconciliation implementation to a thin layer over `isomorphic-git`.

**Six concrete changes:**

1. **State-based reconciliation.** No log-as-event-source. Reconciliation reads filesystem (inbox files = pending; git status = changed) + lockfile directory (`.dome/in-flight/`) + scheduled-state file. log.md remains audit-only.
2. **`VAULT_IS_GIT_REPO` as axiom.** Joins the four axioms; five total. Every vault is a git repository.
3. **`INBOX_IS_EPHEMERAL` as shipped-default.** Intake hooks must move/delete inbox files on completion. Filesystem-as-pending-signal.
4. **`dome reconcile` command.** 4 phases: in-flight recovery, inbox processing, git-diff replay, scheduled catch-up. Runs at `dome serve` startup automatically. Refuses during mid-merge.
5. **`isomorphic-git` as v0.5 dependency.** Pure-JS git engine; works in Bun without native binary. The only git path; no hash-cache fallback.
6. **`inbox/raw/` default-on.** `dome init` creates the bucket AND ships `intake-raw.yaml` as a shipped-default hook. Quick-capture works out of the box; other intakes opt-in.
