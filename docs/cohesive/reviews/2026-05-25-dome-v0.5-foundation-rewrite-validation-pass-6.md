## Architectural reflection

**How it feels now:** Six passes have produced a substrate that genuinely teaches itself. A future contributor reading sdk-surface → vault-layout → hooks → the invariant docs in order can reconstruct Dome's behavior without ever opening implementation code. The seven-Tool seal, the six-axiom set, the per-(handler, target) cycle check, the state-based reconciliation, and the dispatcher-owned-paths boundary are each load-bearing and each fully described where they live. ## Important issues

### I1. MCP-exposed `sensitivity_classify` is a sub-workflow, not a top-level invocable

- **Severity:** Medium
- **Category:** Spec drift
- **Why it matters:** `docs/wiki/specs/mcp-surface.md:55` exposes `dome.workflow.sensitivity_classify` as a freestanding MCP prompt that a harness can offer to its user, marked opt-in. But `docs/wiki/specs/prompts-and-workflows.md:66` and `docs/wiki/specs/hooks.md:110` both pin sensitivity-classify as a sub-workflow invoked *inside* `ingest`, gated by the SENSITIVE_GOES_TO_INBOX invariant. A harness that invokes `dome.workflow.sensitivity_classify` standalone has no defined behavior — there's no inbox file to gate, no downstream `writeDocument` to refuse. The contract a plugin author or harness integrator would read is ambiguous: invocable or not?
- **Evidence:** `mcp-surface.md:55` lists it as a standalone MCP prompt; `prompts-and-workflows.md:66` row reads "sub-workflow inside `ingest` when `SENSITIVE_GOES_TO_INBOX` is enabled"; `hooks.md:110` "The classification must run *before* `writeDocument` is called".
- **Recommended fix:** Either (a) remove `dome.workflow.sensitivity_classify` from the MCP prompt table and note in mcp-surface.md that sub-workflows aren't exposed standalone; or (b) add a paragraph in mcp-surface.md explicitly describing what the sub-workflow does when invoked standalone (likely: classify text, return a result, no vault mutation). The current state implies (b) but doesn't deliver it.
- **Substrate artifact to add or update:** spec (mcp-surface.md §"Prompts exposed")

### I2. Dispatcher's privileged-API Effect emission is unstated

- **Severity:** Medium
- **Category:** Domain model
- **Why it matters:** `INDEX_AND_LOG_ARE_DISPATCHER_OWNED.md` defines `dispatcher.writeIndex` and `dispatcher.appendLogEntry` as the only writers to `index.md` and `log.md`. `event-types-and-payloads.md:19` claims `document.written.index` exists as an event derivable from a `wrote-document{path: 'index.md', diff}` Effect, and `hooks.md:72` notes auto-update-index's own write "doesn't match `document.written.wiki.*`" — implying the index write *does* produce an event (just an unsubscribed one). But no spec states that the privileged dispatcher API emits Effects. If it doesn't, `document.written.index` and `log.appended` events have no actual emitter and the event taxonomy is partly unreachable. If it does, the emission rule needs to be named so plugin Tool authors know what shape to mirror.
- **Evidence:** `INDEX_AND_LOG_ARE_DISPATCHER_OWNED.md` defines the privileged API without describing Effect emission; `event-types-and-payloads.md:19` posits `document.written.index` as derivable from a write Effect; `hooks.md:72` implies the dispatcher write produces an event.
- **Recommended fix:** Add a sentence to `INDEX_AND_LOG_ARE_DISPATCHER_OWNED.md` §"Structural enforcement" stating that `dispatcher.writeIndex` and `dispatcher.appendLogEntry` produce `wrote-document` and `appended-log` Effects through the same Effect pipeline as public Tools, so the existing event taxonomy works uniformly. Or, if the design choice is to suppress events on dispatcher writes, name that and explain how `auto-update-index` recomputes idempotently without seeing its own writes.
- **Substrate artifact to add or update:** named invariant (`INDEX_AND_LOG_ARE_DISPATCHER_OWNED.md`)

### I3. `inbox/review/` framing as an opt-in intake bucket

- **Severity:** Low
- **Category:** Spec drift
- **Why it matters:** `docs/wiki/specs/vault-layout.md:32` lists `inbox/`'s subdirectories as "shipped-default capture bucket (created by dome init)" and "voice/, research/, clip/, review/ — created when the user activates the corresponding intake." But `hooks.md:113-118` is explicit: `inbox/review/` is NOT an intake — no workflow runs on writes to it — it's the destination for the `sensitivity-classify` sub-workflow when SENSITIVE_GOES_TO_INBOX is enabled. Putting `review/` in the same enumeration as `voice/`, `research/`, `clip/` (which are real intakes) reads as if there's a missing `intake-review.yaml` template the user would copy. There isn't.
- **Evidence:** `vault-layout.md:32` enumerates `review/` alongside intake buckets; `hooks.md:113` "It is NOT an intake (no workflow runs on writes to it). It is the user's manual-review queue."
- **Recommended fix:** Split the `vault-layout.md` enumeration into two clauses: "voice/, research/, clip/ — opt-in intake buckets activated via hook template; review/ — opt-in destination created when SENSITIVE_GOES_TO_INBOX is enabled." One sentence; surfaces the asymmetry without reorganizing the doc.
- **Substrate artifact to add or update:** spec (`vault-layout.md` directory tree)

## Locality concerns

The substrate is well-localized. The seven-Tool seal is enforced at sdk-surface.md (the canonical home) and the matrix reflects it. The dispatcher's privileged API is a real shared seam between hooks (auto-update-index calls it) and INDEX_AND_LOG_ARE_DISPATCHER_OWNED (defines it); the locality is appropriate because the seam is small (2 functions) and the access pattern is structural (HookContext.dispatcher field). The "concept folder 8 → 2" collapse documented in pass 1 looks right in retrospect — the 6 deleted concept docs were code-shape similarity, not real contract similarity.

## Future-fit concerns

VISION.md §"Shape over time" frames v1+ pressures (native mobile, voice, sync) as non-normative, and the specs honor that — `concurrent-harness-write.md` §"v1+ sync notes" and `mcp-surface.md` §"Authentication" both mark v1+ deferral explicitly. No v1+ pressure is smuggled into v0.5 normative sections.

## Enforcement concerns

The 12 named invariants split cleanly: 9 enforced at the Tool call site or the hook-type-system boundary (matrix-covered, test-named), 2 enforced at vault-open (`VAULT_IS_GIT_REPO`) or workflow-prompt boundary (`INBOX_IS_EPHEMERAL`) with the off-matrix carve-out documented, and 1 at the dispatcher-rejects-public-Tool layer (`INDEX_AND_LOG_ARE_DISPATCHER_OWNED`). Every invariant carries a `**Test guarantee:**` line naming a concrete test file. `INBOX_IS_EPHEMERAL` is the weakest enforcement (prompt-only), explicitly flagged as such in the matrix off-matrix subsection and the deferred-substrate list — that's correctly visible. `HOOKS_CANNOT_BYPASS_TOOLS.md`'s "runtime watcher" check for plugin authors importing `node:fs` directly is the right pragmatic enforcement boundary; the type-level enforcement is what matters.

## Behavior knowable outside implementation?

Yes. Each Tool carries a canonical TypeScript signature in `sdk-surface.md` §"Tool signatures"; the matrix gives per-cell enforcement; the event taxonomy maps Effects to events and names typical hook handlers; the cycle-prevention semantics (per-(handler, target) + depth net) are described identically in `hooks.md:126` and `hook-cycle.md`. A future contributor implementing `writeDocument` against this substrate would have the signature, the eight invariants it touches, the error vocabulary (`'invariant-violated'`, `'dispatcher-owned-path'`, `'concurrent-write-conflict'`, `'frontmatter-mismatch'`, `'wikilink-not-fullpath'`, `'sensitive-must-route-to-inbox'`, `'page-creation-requires-reason'`), the validate-before-commit batching pattern from `multi-page-partial-write.md:28`, and the test guarantees. The implementation is reproducible from the docs.

None at the Medium+ threshold. Spot-checks for "should," "may," "could," "TODO," "TBD" in normative sections turn up only the deliberate "MAY" / "MUST" in `INBOX_IS_EPHEMERAL.md:13` (which is correctly RFC-style normative). The pass-5 `hooks.md:280` hedge is closed.
