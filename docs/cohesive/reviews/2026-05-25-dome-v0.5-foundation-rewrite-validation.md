## Executive judgment

The rewrite is ambitious and largely coherent: the four-concept core, the tiered feature model, the git-as-axiom durability story, and the prompts-as-contract principle are all present in end-state language and tightly cross-referenced. A future contributor reading sdk-surface + hooks + the invariants can reproduce most of v0.5. But the four refinement passes left counts and ghost surfaces unreconciled across files. Most damaging: the **11th invariant** (`INBOX_IS_EPHEMERAL`) is silently missing from the tool-invariant matrix, the index headline, and the ledger preamble — exactly the kind of "did this rule survive the rewrite?" question the matrix exists to answer. Repair is mechanical, not architectural.

## Blocking issues

### B1. Delta-at-a-glance preamble non-conformant and inconsistent with body
- **Severity:** Blocker
- **Category:** Spec drift
- **Why it matters:** The preamble is what `validate-rewrite` quotes verbatim into the validation review at decision time, and what `implement-cohesively` Step 1 enumerates into the implementation intent paragraph. The current preamble (a) does not follow the 8-category count-or-name list shape mandated by `references/templates/design-delta-ledger.md` §"Authoring rules", and (b) lists "10 named invariants" + "6 gotchas" + matches the *pre-pass-3* state — the durability and per-workflow-commits passes were never folded into the preamble. The reader of the validation review sees pre-pass-3 substrate counts even though the body and disk reflect post-pass-4.
- **Evidence:** `docs/cohesive/delta-ledgers/2026-05-25-dome-v0.5-foundation.md:9-41` (preamble); `:167-213` (passes 3-4 changing counts); ledger §"Updated substrate counts" :196-213 explicitly says "10 — split 5 axiom + 3 default + 2 opt-in" but on-disk inventory has 11.
- **Recommended fix:** Rewrite the preamble against the canonical 8-bullet authoring rule. State counts that match disk: 8 specs / **11** invariants (5 axiom + **4** default + 2 opt-in) / 3 matrices / **8** gotchas / 2 concepts / 9 entities / 2 sources / 2 syntheses. Add explicit Conceptual changes / Named invariants / Behavior matrices / Gotchas / Semantic linters / Tests proposed / Deferred bullets. The "5+3+2=10" arithmetic in the ledger's substrate-counts table is mis-summed — INBOX_IS_EPHEMERAL is `tier: shipped-default` on disk.
- **Substrate artifact to add or update:** ledger preamble.

### B2. `INBOX_IS_EPHEMERAL` is invisible to the enforcement matrix
- **Severity:** Blocker
- **Category:** Invariant
- **Why it matters:** The tool-invariant matrix is the single artifact that answers "is this rule actually enforced?" for every named invariant. INBOX_IS_EPHEMERAL has a normative statement, a counter-example, and a tier marker — but the matrix says it covers "9 Tool-boundary-enforced invariants" and silently omits this one. Worse, INBOX_IS_EPHEMERAL.md states "the contract is enforced by the intake-workflow prompts themselves" — i.e., it is enforced by prompt prose, which the rewrite's own `agent-prompt-regression.md` flags as the highest-severity drift surface. A future contributor reading the matrix would not learn that this invariant has weaker-than-structural enforcement.
- **Evidence:** `docs/wiki/matrices/tool-invariant-enforcement.md:10` ("Exhaustive over the 7 SDK Tools × 9 Tool-boundary-enforced invariants"); `:12` names only `VAULT_IS_GIT_REPO` as the off-matrix case. `docs/wiki/invariants/INBOX_IS_EPHEMERAL.md:19-22`. `docs/index.md:18` claims "10 total" but lists 11.
- **Recommended fix:** Either (a) add an INBOX_IS_EPHEMERAL row to the matrix with explicit "enforced by workflow prompts; `dome doctor` reports stale files; no Tool-call-site refusal" cells, and update the header to "10 Tool-boundary + workflow-enforced invariants," or (b) carve a second short matrix for prompt-enforced invariants and link from the main matrix's header note. Update `docs/index.md` invariant count to 11 with the correct 5+4+2 split.
- **Substrate artifact to add or update:** `docs/wiki/matrices/tool-invariant-enforcement.md`; `docs/index.md`.

### B3. `.dome/in-flight/` ghost in `cli.md` contradicts the no-lockfile design
- **Severity:** Blocker
- **Category:** Spec drift
- **Why it matters:** Pass 4 explicitly removed lockfiles and reduced reconciliation 4→3 phases. SDK spec and hooks spec are consistent with this. CLI spec still tells `dome init` to create `.dome/{prompts,hooks,in-flight,state}/`. A contributor implementing `dome init` from cli.md would produce a directory the rest of the system insists must not exist.
- **Evidence:** `docs/wiki/specs/cli.md:24`; contradicted by `docs/wiki/specs/sdk-surface.md:183-184` and `docs/wiki/specs/hooks.md:135-147`.
- **Recommended fix:** Strike `in-flight` from cli.md line 24. Audit other directory enumeration for the same ghost.
- **Substrate artifact to add or update:** `docs/wiki/specs/cli.md`.

### B4. Auto-commit default contradicts `VAULT_IS_GIT_REPO` operational note
- **Severity:** Blocker
- **Category:** Invariant
- **Why it matters:** SDK and hooks specs commit to per-workflow auto-commit as the v0.5 default. The named-invariant doc for `VAULT_IS_GIT_REPO` says the opposite: "The user is encouraged to commit periodically; Dome does not auto-commit." Per-workflow commits are load-bearing for the no-lockfile recovery story, the log.md / git log alignment, and the universal-undo claim. A future contributor reading the invariant doc as canonical would build the wrong runtime.
- **Evidence:** `docs/wiki/specs/sdk-surface.md:185-191`; `docs/wiki/specs/hooks.md:220-235`; contradicted by `docs/wiki/invariants/VAULT_IS_GIT_REPO.md:30-32`.
- **Recommended fix:** Rewrite VAULT_IS_GIT_REPO.md "Operational notes" first bullet to: "Workflows auto-commit by default per `git.auto_commit_workflows: true` (see `wiki/specs/hooks.md` §"Commit policy"); user out-of-band edits remain uncommitted unless the user explicitly commits."
- **Substrate artifact to add or update:** `docs/wiki/invariants/VAULT_IS_GIT_REPO.md`.

### I2. Hooks spec references retired `dome init --kind <profile>` flag
- **Severity:** High
- **Category:** Spec drift

### I3. `eval` command claimed by prompts-and-workflows.md is absent from CLI
- **Severity:** High
- **Category:** Spec drift
- **Why it matters:** prompts-and-workflows.md :111 states "the SDK ships an `eval` command." cli.md explicitly says "**Seven commands**" — `eval` is not among them. agent-prompt-regression.md is load-bearing on the eval suite as the structural mitigation. A contributor cannot tell whether `eval` is a CLI command, a `bun test` target, a workflow, or vapor.
- **Evidence:** `docs/wiki/specs/prompts-and-workflows.md:111-118`; `docs/wiki/specs/cli.md:12`, `:167-177`.
- **Recommended fix:** Either (a) add `dome eval` to CLI spec as an 8th command and update the count, or (b) reframe in prompts-and-workflows.md as "the SDK ships a `bun run eval` test target" and point at where it lives. Pick one; update agent-prompt-regression.md's mitigation reference to match.
- **Substrate artifact to add or update:** `docs/wiki/specs/cli.md` and/or `docs/wiki/specs/prompts-and-workflows.md`.

### I4. `writeDocument` Tool description omits optimistic-locking mechanism
- **Severity:** Medium
- **Category:** Spec drift / Locality
- **Why it matters:** The concurrent-harness-write gotcha names timestamp-based optimistic locking as the structural mitigation, declares `writeDocument` returns `Result.err({ kind: 'concurrent-write-conflict' })`, and treats this as load-bearing for the v0.5 multi-harness story. The SDK spec's writeDocument catalog row lists six enforcements; concurrent-write detection is absent.
- **Evidence:** `docs/wiki/gotchas/concurrent-harness-write.md:16-21`; `docs/wiki/specs/sdk-surface.md:79`, `:62-66`.
- **Recommended fix:** Add a §"Concurrency" note to writeDocument's catalog row or to the Tool catalog preamble, naming the `concurrent-write-conflict` error kind and pointing at the gotcha.
- **Substrate artifact to add or update:** `docs/wiki/specs/sdk-surface.md`.
