## Substrate-alignment findings

### F1 (Medium). `_writer` reaches plugin code via Vault interface

- **Severity:** Medium
- **Category:** Invariant enforcement
- **Why it matters:** `INDEX_AND_LOG_ARE_DISPATCHER_OWNED` says plugin code never reaches the privileged writer directly. The `_writer` field is documented `@internal` at `src/vault.ts:100` and not exported from `src/index.ts`. But `Vault` itself IS exported, and the `_writer` field is `readonly _writer: PrivilegedWriter` on the public interface shape ‚Äî any plugin holding a `Vault` reference can read `vault._writer` despite the JSDoc comment. The structural fence is the `PrivilegedWriter` type not being exported, which prevents typed consumption ‚Äî but `as` casts would bypass it.
- **Evidence:** `src/vault.ts:44‚Äì101` (Vault interface includes `_writer`); `src/workflows/project-ai-sdk.ts:22` (sole legitimate consumer); no test asserts the @internal boundary.
- **Recommended fix:** Add a lockstep test parallel to `bundle-deps.test.ts` that greps non-`src/workflows/project-ai-sdk.ts` files for `\._writer` reads and asserts zero matches.
- **Substrate artifact to add or update:** New `docs/wiki/linters/no-writer-leak.md` (semantic linter spec sibling to `no-retired-symbol-names`) + matching test.

### F2 (Low). `loadDeclarativeHooks` error path bypasses ValidationError shape

- **Severity:** Low
- **Category:** Spec drift (boundary-validation-via-zod gotcha)
- **Why it matters:** The Zod boundary-validation gotcha specifies each loader returns `Result<T, ValidationError>`. `parseDeclarativeHook` does (`src/hooks/yaml-loader.ts:157`), but the `loadDeclarativeHooks` orchestration loop at line 130 collapses the typed error back to a `string` via `result.error.message` for `onLoadError`. Callers lose the `kind: "validation"` discriminant; the error's structural shape doesn't propagate past one frame.
- **Evidence:** `src/hooks/yaml-loader.ts:130‚Äì133`; `src/vault.ts:232‚Äì241`.
- **Recommended fix:** Change `onLoadError` signature to `(file: string, error: ValidationError) => void`.
- **Substrate artifact to add or update:** Update `docs/wiki/gotchas/boundary-validation-via-zod.md` to name the orchestration boundary as the last unflattened frame.

## Structure findings

### F3 (Medium). Value-import cycle reintroduced between `vault.ts` and `vault-dispatcher.ts`

- **Severity:** Medium
- **Category:** Seam (same shape as the latent-ordering scar `vaultRef` was meant to retire)
- **Why it matters:** The whole point of the `vaultRef` setter pattern was to kill positional-ordering rules. The decomposition then opens a new structural risk: `vault.ts` imports `wireDispatcher` (value) from `vault-dispatcher.ts`, and `vault-dispatcher.ts` imports `appendCycleLogEntry` (value) from `vault.ts`. It works today because the cycle resolves at function-call time, but this is the *same shape* of latent-ordering scar ‚Äî load-order-dependent ‚Äî just moved one layer up.
- **Evidence:** `src/vault-dispatcher.ts:34` (`import { appendCycleLogEntry } from "./vault"`); `src/vault.ts:13` (`import { wireDispatcher ‚Ä¶ } from "./vault-dispatcher"`).
- **Recommended fix:** Move `appendCycleLogEntry` into `vault-dispatcher.ts` (its only caller). It is dispatcher-owned by function and dispatcher-owned by invariant (INDEX_AND_LOG_ARE_DISPATCHER_OWNED).
- **Substrate artifact to add or update:** A note in `sdk-surface.md` ¬ß"Composable construction" that the three helpers must depend in one direction (vault ‚Üí helpers, never back), and a lockstep test assertion in `vault-bootstrap-order.test.ts`.

### F4 (Low). Three vault helpers have one caller ‚Äî composable-construction framing ahead of evidence

- **Severity:** Low
- **Category:** Premature centralization (framing concern, not implementation concern)
- **Why it matters:** `loadVaultConfig`, `buildBuiltinHookRegistry`, `wireDispatcher` are consumed only by `openVault` and their own unit tests. The spec calls them "composable construction" for v1+ HTTP/voice surfaces; that's speculative until a second caller exists. The decomposition pays for itself today (kills the three ordering rules; makes bootstrap testable in isolation), so this is locality-positive ‚Äî but the public-API framing in the spec is ahead of evidence.
- **Recommended fix:** Keep the decomposition. Tone the spec's "composable construction" language down to "internal helpers; v1+ surfaces *may* assemble from these once stable; not exported from `src/index.ts`."
- **Substrate artifact to add or update:** `docs/wiki/specs/sdk-surface.md` ¬ß"Composable construction" framing tighten.

### F5 (Low ‚Äî informational positive). `checkFrontmatterType` absorbed 4 CHECKs because they share derived state

- **Severity:** Low (positive informational)
- **Why it matters:** CHECK 1, 4, 5, 8 all need `knownPluralDirs`, `declaredExtensionTypes`, and `perTypeFields` derived from `vault.pageTypes`. Splitting them would re-derive that state three times or invent a "page-type catalogue" helper that has one caller. The 138-line module pays its weight.
- **Recommended fix:** Update module docstring's first line from "Page-type / frontmatter integrity check" to "Page-type catalogue integrity (CHECKs 1, 4, 5, 8 share derived state)".

### F6 (Low ‚Äî informational positive). Three Zod schemas across three locations ‚Äî the inconsistency is fine

- **Severity:** Low (positive informational)
- **Why it matters:** `vault-schemas.ts` and `state-schemas.ts` exist as their own files because each is consumed by multiple call sites. `DeclarativeHookSchema` lives inline in `yaml-loader.ts` because the loader is the only consumer. The asymmetry tracks "is there a second consumer?" ‚Äî the right signal.
