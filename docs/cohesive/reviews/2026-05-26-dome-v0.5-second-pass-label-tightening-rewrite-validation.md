## Verdict: Issues Found

The structural split (`AbstractSurface` in core + `renderMcp` in `@dome/sdk/mcp`) is clean and a future contributor can read `sdk-surface.md` §"Consumer surfaces" and reproduce the layering — `vault.tools` ⊆ `AbstractSurface.tools` ⊆ `renderMcp.tools` is a single sealed staircase. ## Important issues

### I1. `bindEntry` vs `bindTools` naming split across two surfaces

**Severity:** Medium
**Category:** Domain model

**Why it matters:** `sdk-surface.md:45` says `"\`bindTools\` (the source for \`vault.tools\`) and \`bindAiSdkTools\` ... both consume the helper"`. `HOOK_DISPATCH_IS_VAULT_BOUND.md:25` says `"\`bindEntry\` (the bindTools internal that populates \`vault.tools\`) consumes it"`. These don't contradict — `bindEntry` is presumably called by `bindTools` — but the reader has to infer the relationship. The ledger §"Implementation work" entry 4 (`code-bind-entry-consume-wrap`) talks about rewriting `bindEntry`; entry 3 (`code-wrap-mutating-invoke`) names the helper. The matrix at `tool-invariant-enforcement.md:55` says `"the projection-construction layer (\`bindTools\` / \`bindAiSdkTools\`)"`. Three surfaces, two function names, no canonical mapping.

**Evidence:** `sdk-surface.md:45`, `HOOK_DISPATCH_IS_VAULT_BOUND.md:25`, `tool-invariant-enforcement.md:55`, ledger lines 124 and 125.

**Recommended fix:** Either (a) define the `bindTools` / `bindEntry` relationship in one place (`sdk-surface.md` §"Hook dispatch is intrinsic" is the natural home — name them as "`bindTools` is the public; `bindEntry` is the per-entry helper inside it"), or (b) use only `bindTools` across all four surfaces (preferred — `bindEntry` is an implementation detail).

**Substrate artifact to add or update:** `wiki/specs/sdk-surface.md` §"Hook dispatch is intrinsic".

### I2. Enforcement story for HOOK_DISPATCH_IS_VAULT_BOUND uses "should catch" hedging

**Severity:** Medium
**Category:** Enforcement

**Why it matters:** The invariant's structural-enforcement story relies on (a) the single-source helper (real structural seam) + (b) two integration tests (real). But the prose at `HOOK_DISPATCH_IS_VAULT_BOUND.md:25` hedges: *"A future contributor copying the dispatch logic inline produces a duplicate that the review process should catch — and that the tests below would catch at runtime if the duplicate diverges."* And `:31` says: *"the contributor's pull request would surface the missing `renderHttp.test.ts` analog during review."* Both shift load to "review process" — which is exactly what the invariant exists to remove. The tests *would* catch divergent behavior but not a hand-inlined duplicate that happens to be byte-equivalent until `wrapMutatingInvoke` is next changed. The ledger acknowledges this — line 23 says a future code-graph linter is v0.5.1+ — but the invariant's enforcement-story prose doesn't name that gap.

**Evidence:** `HOOK_DISPATCH_IS_VAULT_BOUND.md:25`, `:31`; ledger line 23.

**Recommended fix:** Rewrite the structural-enforcement bullet to honestly name the gap: "Structural enforcement is the single-source helper + the two integration tests. The tests catch behavioral divergence; they do not catch a hand-inlined byte-equivalent duplicate (the v0.5.1+ code-graph linter would close that gap)." Don't write "the review process should catch" — that's the failure mode the substrate model exists to prevent.

**Substrate artifact to add or update:** `wiki/invariants/HOOK_DISPATCH_IS_VAULT_BOUND.md` §"Structural enforcement".

Lower-stakes than other findings; flagging because two entries away from the file paragraph this matters for downstream auditing.

**Substrate artifact to add or update:** Ledger §"Implementation work" entries 4-5.
