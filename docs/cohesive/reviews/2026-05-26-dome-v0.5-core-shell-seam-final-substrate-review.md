# Final Substrate Alignment Review — Dome v0.5 core-shell seam (Phase B)

**Date:** 2026-05-26

**Worktree:** `/Users/mark.toda/dev/dome/.claude/worktrees/design+dome-v0.5-core-shell-seam`
**Substrate compared against:** sdk-surface.md, mcp-surface.md, CORE_HAS_NO_LLM_OR_MCP_DEPENDENCY.md, tool-invariant-enforcement.md, consumer-surface.md, page-schema.md, the 11 gotcha docs, and the 13-invariant `INVARIANTS` const in `src/types.ts`.

## Verdict: Risky

The four-entrypoint package shape is structurally true and CORE_HAS_NO_LLM_OR_MCP_DEPENDENCY holds at the static-import-graph layer the bundle-deps test asserts. The lockstep tests (matrix-coverage, gotcha-coverage, invariant-coverage, public-surface-shape) are cleanly wired to substrate conventions. Vault interface matches the new sdk-surface.md spec; `close()` lives at the right seam; `INVARIANTS` carries all 13 entries. However, **the entrypoint split introduced a silent hook-dispatch regression at the MCP and AI-SDK seams** (Finding B1): post-Phase-B, MCP-routed and `projectAiSdk`-bound Tool mutations no longer flow through the Vault's hook dispatcher because each entrypoint rebuilds `bindTools` without `wrapMutation`. This is the behavioral seam pre-Phase-B's single `bindTools(partial, privilegedWriter, wrapMutation)` call established; no test asserts it post-split. One spec/code shape mismatch on the `DomeMcpServer` constructor (Finding I1). One forbidden-package name divergence between docs and `package.json` (Finding I2). Otherwise alignment is high.

## Findings

### B1 — MCP and AI-SDK Tool routes bypass the Vault hook dispatcher

**Severity:** Blocker (correctness regression, not caught by any test)
**Category:** Under-enforced invariant / Spec drift

**Why it matters:** Pre-Phase-B, `openVault` called `bindTools(partial, privilegedWriter, wrapMutation)` exactly **once**, producing `{ tools, aiTools, parsers }` where every mutating function — whether consumed via `vault.tools.X`, `vault.aiTools.X`, or `vault.toolParsers.X` — was wrapped with the dispatcher-firing `wrapMutation`. The previous `registry.ts` comment explicitly named this: "Apply the same mutating wrapper to parseAndInvoke so MCP-routed calls also flow through the hook dispatcher; otherwise the SDK consumer (Vault.tools.*) and the MCP consumer diverge on hook firing." Post-Phase-B, `vault.ts:245` calls `bindTools(partial, privilegedWriter, wrapMutation)` and destructures `{ tools }` only — discarding the wrap-aware `parsers`. The MCP adapter at `src/mcp/tool-adapters.ts:52-54` then calls `bindTools(vault, writer)` with no third argument (defaulting to `identityWrap`), as does `src/mcp/project-mcp.ts:21-24`. `src/workflows/project-ai-sdk.ts:16-23` likewise calls `bindAiSdkTools(vault, writer)` directly (the AI-SDK binding never threads a wrap). Net result: a Claude Code session writing via `dome.write_document` does the on-disk write correctly but **does not fire** `document.written.wiki.*` events on the vault's hook dispatcher — `auto-update-index`, `auto-cross-reference`, and every declarative-YAML intake hook silently miss MCP-routed writes. Same for any future `runWorkflow` consumer that drives Tools via the AI-SDK `ToolSet` path.

This contradicts `sdk-surface.md §"Vault surface"` ("projectMcp(vault) … sharing the hook-dispatch wrap") and `tool-adapters.ts:38-41`'s own internal claim ("delegates the call to the per-Tool parser from `bindTools(vault, writer)`, which … invokes the same Vault-bound function `vault.tools[name]` exposes (**sharing the hook-dispatch wrap**)").

**Evidence:**
- `src/vault.ts:245` — `const { tools } = bindTools(partial, privilegedWriter, wrapMutation);` — drops the wrap-aware `parsers`.
- `src/mcp/tool-adapters.ts:52-54` — `function buildToolAdapters(vault: Vault): ToolAdapter[] { const writer = makePrivilegedWriter(vault.path); const { parsers } = bindTools(vault, writer); ...` — no `wrapMutating` argument; parsers are bare.
- `src/mcp/project-mcp.ts:21-24` — same shape.
- `src/workflows/project-ai-sdk.ts:16-23` — `bindAiSdkTools(vault, writer)` has no wrap parameter; AI-SDK Tools call `entry.invoke(vault, writer, compacted)` directly.
- `src/tools/ai-sdk-binding.ts:20-35` — `bindAiSdkTools` does not accept or thread a mutating wrapper.
- Pre-Phase-B `src/tools/registry.ts` (git show main) wrapped `parseAndInvoke` for mutating Tools in `bindTools` itself, so even an MCP-side `bindTools(vault, writer)` call would have been wrap-aware via the wrapper passed at vault-open time. The current `bindTools` requires the caller to pass the wrap — and the MCP/workflows callers do not.
- No integration test in the diff asserts "MCP-routed writeDocument fires auto-update-index"; `tests/integration/consumer-surface.test.ts` and `tests/integration/mcp-surface.test.ts` cover surface shape, not hook-dispatch propagation.

**Recommended fix:** Move the hook-dispatch wrap to the substrate seam where every entrypoint sees it. Two viable shapes:

1. **Wrap inside `entry.invoke` via a dispatcher held by the Vault.** Add an internal `vault._dispatch(events)` method (or expose `dispatchEvents` as the seam — it already exists at `vault.ts:218-225`). Change `bindTools` so the parser path calls `await vault.dispatchEvents(projectEffectsToEvents(out.effects))` after each mutating invoke unconditionally. The wrap is no longer a parameter; it's a property of the Vault the parser closes over. `projectMcp(vault)` and `projectAiSdk(vault)` then inherit dispatch for free.

2. **Cache the wrap-aware parsers on Vault.** Re-introduce a private `vault._parsers` field (not part of the public surface — the substrate already says "tool projections live OFF Vault" for the type-system seam, but a private cache is fine). `buildToolAdapters` and `projectMcp` consume `vault._parsers` directly. The public API still says "no aiTools / toolParsers fields"; the hook-dispatch contract is preserved by the private cache.

Either way: **add an integration test that asserts an MCP-routed `dome.write_document` fires the `document.written.wiki.*` hook chain** (creates a wiki entity via the MCP adapter, then asserts `auto-update-index` updated `index.md`). That test would have caught this gap and is the load-bearing structural fence going forward.

**Substrate artifact to add or update:** Update `sdk-surface.md §"Vault surface"` to name the hook-dispatch contract explicitly (currently glosses it as "sharing the hook-dispatch wrap" without specifying the mechanism). Add an integration test `tests/integration/mcp-hook-dispatch.test.ts` as the AC3-shape structural fence. Consider promoting this to a named invariant `HOOK_DISPATCH_IS_VAULT_BOUND` since it is the structural fence behind `HOOKS_CANNOT_BYPASS_TOOLS`'s symmetric "all-Tool-routes-fire-hooks" guarantee.

### I1 — `DomeMcpServer` constructor shape diverges from spec

**Severity:** High
**Category:** Spec drift

**Why it matters:** `mcp-surface.md §"Construction"` and `sdk-surface.md §"Consumer surfaces"` both show `new DomeMcpServer({ surface })` — surface-only construction. The whole point of the ConsumerSurface concept named in repair pass 2 is that "DomeMcpServer is a thin protocol adapter over ConsumerSurface." The implementation at `src/mcp/server.ts:18-29` actually requires `{ surface, vault }` because `buildToolAdapters(vault)` (called in the constructor) needs the Vault to bind tool parsers. A future contributor following the spec will write `new DomeMcpServer({ surface })`, get a TypeScript error, and discover the spec is wrong — or worse, the spec gets believed and the future HTTP adapter is designed against a surface-only construct that doesn't match what MCP actually does.

The root cause is that `buildToolAdapters` belongs *inside* the ConsumerSurface (the tools field) rather than being computed in the server constructor — which would let the constructor honestly take `{ surface }` only. As designed, the seam leaks Vault into the constructor.

**Evidence:**
- `docs/wiki/specs/mcp-surface.md:21-25` — code block: `const server = new DomeMcpServer({ surface });`
- `docs/wiki/specs/sdk-surface.md:303-307` — same shape.
- `docs/wiki/specs/mcp-surface.md:29` — prose: "DomeMcpServer({ surface }) adapts that surface to the MCP wire protocol."
- `src/mcp/server.ts:18-29` — `DomeMcpServerOpts` has both `surface: ConsumerSurface` and `vault: Vault`, with a comment acknowledging the divergence ("Vault is still passed because buildToolAdapters consumes it … The surface itself doesn't include tool-adapter shapes").
- `tests/integration/consumer-surface.test.ts:43` — `new DomeMcpServer({ surface, vault })` — test agrees with code, not with spec.
- `tests/integration/mcp-surface.test.ts:34` — same.

**Recommended fix:** Pick one of:

1. **Hoist tool adapters into ConsumerSurface.** Add a `toolAdapters: ToolAdapter[]` field to `ConsumerSurface` (computed by `buildConsumerSurface`). The constructor honestly takes `{ surface }` only. Cost: `ToolAdapter` is MCP-specific, so this conceptually leaks MCP shape into the ConsumerSurface type. Mitigation: rename to a protocol-agnostic shape like `RawInputAdapter` (the same shape an HTTP adapter would consume).
2. **Update the specs to match code.** Change every `new DomeMcpServer({ surface })` instance to `new DomeMcpServer({ surface, vault })` in `mcp-surface.md` and `sdk-surface.md`. Document the Vault parameter's role in `DomeMcpServerOpts` and explain why the surface alone is insufficient.

Option 1 is the substrate-aligned answer (it makes the "thin protocol adapter over ConsumerSurface" claim structurally true); option 2 is the cheap-and-honest answer. Either is acceptable; the current ambiguity is not.

**Substrate artifact to add or update:** `mcp-surface.md §"Construction"`, `sdk-surface.md §"Consumer surfaces"`. If option 1 is chosen, also update `consumer-surface.md` matrix's "MCP server" row Tools cell to read `mcp` only (no `core +` compound) since the adapter shape now lives inside the surface.

### I2 — Forbidden-package list in normative docs names a package not in `package.json`

**Severity:** Medium
**Category:** Spec drift

**Why it matters:** `CORE_HAS_NO_LLM_OR_MCP_DEPENDENCY.md:13`, `sdk-surface.md:335-356`, `mcp-surface.md:12`, and `consumer-surface.md:17,40` all name `@anthropic-ai/sdk` as the package the core must not transitively depend on. Actual `package.json:34` declares `@ai-sdk/anthropic` (a Vercel-AI-SDK provider adapter for Anthropic), not `@anthropic-ai/sdk` (Anthropic's first-party SDK). The `bundle-deps.test.ts:29-43` correctly catches both (forbids the exact match `@anthropic-ai/sdk` AND the prefix `@ai-sdk/`), so the structural enforcement holds. But the docs name a package the project doesn't use. A v1+ contributor reading the invariant doc, opening `package.json`, and grepping for `@anthropic-ai/sdk` finds nothing — and may conclude either the invariant is dead-letter or the docs are stale.

**Evidence:**
- `package.json:34` — `"@ai-sdk/anthropic": "^3.0.79"` (Vercel adapter)
- No `@anthropic-ai/sdk` in `dependencies` or transitive (via `bun pm ls` if run; the package name doesn't appear in the project).
- `src/workflows/agent-loop.ts:9` — `import { anthropic } from "@ai-sdk/anthropic";` — confirms the project uses the AI-SDK provider, not Anthropic's first-party SDK.
- `docs/wiki/invariants/CORE_HAS_NO_LLM_OR_MCP_DEPENDENCY.md:13,21,27` — repeatedly names `@anthropic-ai/sdk`.
- `docs/wiki/specs/sdk-surface.md:356` — Dependencies table row: `| @anthropic-ai/sdk | LLM client | Anthropic's official TS SDK | workflows |` — claims it's a bundled dep, but it isn't bundled.
- `docs/wiki/specs/harnesses.md:56` — "This loop uses `@anthropic-ai/sdk` directly" — wrong; it uses `@ai-sdk/anthropic`.

**Recommended fix:** Replace `@anthropic-ai/sdk` with `@ai-sdk/anthropic` throughout the normative docs, OR add `@anthropic-ai/sdk` as the forbidden name with a note explaining it's the historical name preserved for forward-compat (in case the project ever drops the AI-SDK adapter and bundles Anthropic's first-party SDK directly). The honest version is option A: name what's actually there.

**Substrate artifact to add or update:** `CORE_HAS_NO_LLM_OR_MCP_DEPENDENCY.md`, `sdk-surface.md §"Distribution"` and §"Dependencies", `mcp-surface.md`, `consumer-surface.md` legend.

### I3 — `tool-adapters.ts` comment claims `bindTools` "transitively imports `ai`"; it does not

**Severity:** Low (comment drift; no functional bug)
**Category:** Spec drift (internal-doc form)

**Why it matters:** Future contributors trying to understand why `buildToolAdapters` is in `@dome/sdk/mcp` (versus core) will read the comment at `src/mcp/tool-adapters.ts:45-50` and conclude `bindTools` pulls `ai`. But the T8 commit (`4b2f2bc feat(tools): split registry`) explicitly removed the `ai` import from `registry.ts` — `bindTools` is now AI-SDK-free, by design, so `CORE_HAS_NO_LLM_OR_MCP_DEPENDENCY` holds. The comment is leftover from the pre-split shape and now misleads. The real reason `buildToolAdapters` belongs to `@dome/sdk/mcp` is that `ToolAdapter` is MCP-shaped (snake_case `dome.*` names) — not because of transitive deps.

**Evidence:**
- `src/mcp/tool-adapters.ts:45-50` — `Note: this function calls bindTools directly (which transitively imports ai for the AI-SDK Tool<> inputSchema). That's allowed inside @dome/sdk/mcp because the mcp entrypoint's dep scope already includes ai...`
- `src/tools/registry.ts:1-12` — `Phase B split: this module is part of @dome/sdk core and contains NO dependency on ai (Vercel AI SDK)` — direct contradiction.
- `tests/integration/bundle-deps.test.ts` passes — proves `src/tools/registry.ts` does not transitively import `ai`.

**Recommended fix:** Rewrite the comment to name the real reason: `ToolAdapter` is MCP-protocol-specific (carries `dome.*` snake_case names per MCP convention), so it belongs in the MCP entrypoint by surface-shape, not by transitive dep.

**Substrate artifact to add or update:** Update the file comment only. No spec change required.

### I4 — `vault.close()` is implemented as `drainHooks` alias; spec promises additional resource release

**Severity:** Low
**Category:** Under-enforced invariant

**Why it matters:** `sdk-surface.md §"Vault lifecycle"` `close()` step (b) names "release Vault-owned resources — the `p-queue` instance held by `HookDispatcher` (no more events accepted) and any file handles held by the quarantine-store factory." Implementation at `src/vault.ts:286` is `close: drainHooks` — literally an alias for the drain function. There is no explicit release-step that prevents further events from being accepted or that closes file handles. In v0.5 the spec explicitly says "calling vault.tools.X after close() is undefined behavior — v0.5 doesn't enforce a post-close guard," so the absence of a guard is consistent with the spec. But the resource-release step (b) is named in the spec and not implemented; `close()` and `drainHooks()` are now indistinguishable. The lifecycle test at `tests/invariants/vault-lifecycle.test.ts:31-53` asserts only that `close()` returns without throwing.

This is consistent at the contract level (a no-op release IS a release in v0.5) but the spec language ("release Vault-owned resources") promises more than the code delivers. A v1+ mobile shell that calls `vault.close()` expecting file-handle release will be surprised when the p-queue stays open and the quarantine-store file handles linger.

**Evidence:**
- `src/vault.ts:282-287` — `close: drainHooks` (literal alias).
- `docs/wiki/specs/sdk-surface.md:55-60` — spec describes a multi-step close: (a) drain, (b) release resources, (c) optionally invalidate.
- `tests/invariants/vault-lifecycle.test.ts:31-53` — `close()` test asserts only that it returns; doesn't assert resource release.

**Recommended fix:** Either (a) update the spec to say `close()` and `drainHooks()` are identical in v0.5 with a footnote that v1+ may add resource-release; or (b) introduce a one-line resource-release step (e.g., set an internal `_closed = true` flag the dispatcher consults to refuse new events) and add a test asserting `vault.dispatchEvents(...)` after `close()` is a no-op. Option (a) is cheaper; option (b) is the load-bearing seam for v1+ mobile/desktop shells.

**Substrate artifact to add or update:** `sdk-surface.md §"Vault lifecycle"` step (b), or `vault.ts:282-287` close() implementation + a vault-lifecycle test.

## Substrate alignment that holds

The structural fence the axiom names is in place.

- **Four-entrypoint package shape.** `package.json:8-25` declares the four exports (`.`, `./workflows`, `./mcp`, `./cli`); each entrypoint barrel file (`src/index.ts`, `src/workflows/index.ts`, `src/mcp/index.ts`, `src/cli/index.ts`) exists with the expected surface.

- **Matrix-coverage lockstep.** Every bolded enforcement cell in `tool-invariant-enforcement.md` maps to a test name pattern in `tests/invariants/<slug>.test.ts` — verified by the matrix-coverage test wiring.
