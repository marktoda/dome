## Executive judgment

The rewrite delivers on its premise: labels catch up to the post-pass-2 structure (every `ConsumerSurface` reference renames cleanly to `AbstractSurface`), the three "Adding a new X" recipes parallel the canonical "Adding an 8th Tool" shape, the `openVault` composable-construction shape with `vaultRef: { current: Vault | null }` is a real structural fence against the three positional-ordering scars, and the off-matrix lockstep convention is now substrate rather than reviewer memory. A future contributor can pick up the SDK and predict the right answer for every common change shape. Three issues block merge: a count-drift in the ledger preamble itself (the very class of bug `substrate-count-drift.md` documents), a `no-retired-symbol-names` linter spec that grep-matches its own allow-list table, and an `enforced_at:` target on `substrate-count-drift.md` that violates the page-schema spec's definition.

## Blocking issues

### B1. Ledger preamble miscounts specs rewritten (substrate-count-drift in the count-drift ledger itself)

- **Severity:** High
- **Category:** Spec drift
- **Why it matters:** The preamble is what `validate-rewrite` quotes verbatim into the validation review and what downstream readers (including `implement-cohesively`) use as the change index. A miscount here teaches future agents the wrong "what was actually touched" and is the exact bug class `gotchas/substrate-count-drift.md` exists to prevent — landing in the same ledger that introduces the count-drift linter.
- **Evidence:** `delta-ledgers/2026-05-26-dome-v0.5-to-v1-tightening.md:13` — "13 substrate files rewritten (**5 specs**, 2 matrices, 1 invariant, 1 catalog index, 1 handoff archival banner, 3 supporting docs)". The §"Per-file changes" body enumerates **6** spec rewrites: `sdk-surface.md`, `cli.md`, `hooks.md`, `prompts-and-workflows.md`, `harnesses.md`, `page-schema.md`. The "3 supporting docs" slot is also unaccounted for in the body — only `transitive-llm-dependency.md` (one rewrite) and the eleven-off-matrix-gotchas frontmatter pass (one entry) are present.
- **Recommended fix:** Update the preamble to "6 specs, 2 matrices, 1 invariant, 1 catalog index, 1 handoff archival banner, 2 supporting docs" (or "13 substrate files rewritten" with a categorical reference rather than inline tally). Per `substrate-count-drift.md` the link-to-canonical-surface convention should apply to the ledger preamble too.
- **Substrate artifact to add or update:** Delta ledger preamble.

### B2. `no-retired-symbol-names` linter spec lacks self-exemption from its own scan target

- **Severity:** High
- **Category:** Enforcement
- **Why it matters:** The linter scans `docs/wiki/**/*.md` and asserts no normative doc names a retired symbol. The linter spec itself sits at `docs/wiki/linters/no-retired-symbol-names.md` and enumerates `ConsumerSurface`, `buildConsumerSurface`, `projectMcp`, `McpProjection`, `McpToolName` in a literal table as the allow-list. When the v0.5.1 lockstep ships, it will grep-match itself and either fail on first run or require an unannounced exclusion the spec never named. Either outcome breaks the trust-the-spec contract for the next contributor wiring up the test.
- **Evidence:** `linters/no-retired-symbol-names.md:18-28` lists the retired names as a markdown table inside `docs/wiki/`. §"Programmatic detection" lines 32-34 enumerate excluded paths (reviews / brainstorms / delta-ledgers / substrate-discovery / archival-marked handoffs) but **not** `docs/wiki/linters/`. `docs/index.md:67` also enumerates `ConsumerSurface`, `buildConsumerSurface`, `projectMcp`, `McpToolName`, `SENSITIVE_GOES_TO_INBOX` in normative scope.
- **Recommended fix:** Either (a) add `docs/wiki/linters/no-retired-symbol-names.md` and the `docs/index.md:67` line to the exclusion list explicitly, or (b) move the allow-list out of the doc surface entirely so the typed const at `src/types.ts` is the only home and the doc cites it categorically. Option (b) is the closer parallel to `substrate-count-drift.md`'s link-to-canonical-surface convention.
- **Substrate artifact to add or update:** `linters/no-retired-symbol-names.md` §"Programmatic detection"; possibly a typed const in `src/types.ts`.

### B3. `enforced_at:` on `substrate-count-drift.md` violates the page-schema spec's definition

- **Severity:** High
- **Category:** Spec drift
- **Why it matters:** `page-schema.md:130` defines `enforced_at:` as either "the canonical test file" or "the source file the scar lives in." The gotcha-coverage lockstep (per ledger §F5) reads the field and asserts "the named path exists." `substrate-count-drift.md:8` points at `docs/wiki/linters/no-retired-symbol-names.md` — a linter spec doc that explicitly says "lockstep test ships in v0.5.1" and is otherwise convention-with-grep. The page exists, so the lockstep test passes; but the *semantic* claim ("this scar has structural mitigation at the named path") is false: the linter hasn't shipped. The check satisfies its own assertion while documenting nothing.
- **Evidence:** `page-schema.md:130` defines the allowable shapes. `substrate-count-drift.md:8` — `enforced_at: docs/wiki/linters/no-retired-symbol-names.md`. The linter spec at line 12: "**Status:** Lockstep test ships in v0.5.1 ... The convention until that lockstep ships is reviewer attention." Reviewer attention is not structural mitigation per `cohesion-rubric.md` §"Invariant enforcement."
- **Recommended fix:** Either (a) point `enforced_at:` at the (future) test path `tests/integration/no-retired-symbol-names-in-specs.test.ts` and accept that the doctor check will warn until the test ships, or (b) extend `page-schema.md` §"Extension types" to add a `deferred` semantic for `enforced_at:` (mirroring the existing `deferred` value for `coverage:`). Option (a) is cleaner — it makes the deferred-mitigation state visible rather than papering over it with a doc reference.
- **Substrate artifact to add or update:** `gotchas/substrate-count-drift.md` frontmatter, or `specs/page-schema.md` §"Extension types".

## Important issues

### I1. `buildInstructions(vault)` referenced as load-bearing but never enumerated in the SDK distribution

- **Severity:** Medium
- **Category:** Domain model
- **Why it matters:** A future contributor reading `sdk-surface.md` §"Distribution" to find the canonical exports of `@dome/sdk` core sees `buildAbstractSurface`, `PromptDescriptor`, `ResourceDescriptor` enumerated. They do not see `buildInstructions`. Yet `prompts-and-workflows.md:163, 169` and `WORKFLOWS_KNOW_VAULT_CONTEXT.md:15` treat `buildInstructions(vault)` as a normative, named entry point that composes the `instructions` payload. The consumer-surface matrix doesn't have a column for it either. Implicit declaration of a load-bearing symbol drifts the entrypoint-export contract.
- **Evidence:** `sdk-surface.md:453` enumerates core exports without `buildInstructions`. `prompts-and-workflows.md:163` and `:169` reference it as if it were a known core export. `matrices/consumer-surface.md:22` legend lists what `core` carries — `buildInstructions` is absent.
- **Recommended fix:** Add `buildInstructions(vault): Promise<string>` to `sdk-surface.md` §"Distribution" core line and to the `AbstractSurface` column of `matrices/consumer-surface.md` (the symbol family is "Instructions composer" or fold into the `AbstractSurface` column with a parenthetical).
- **Substrate artifact to add or update:** `specs/sdk-surface.md` §"Distribution"; `matrices/consumer-surface.md`.

### I2. `boundary-validation-via-zod.md` `enforced_at:` anchors at the precedent rather than the scar

- **Severity:** Medium
- **Category:** Spec drift
- **Why it matters:** Page-schema permits `enforced_at:` to name "the source file the scar lives in" for documented scars. The scar this gotcha names lives in the *absence* of Zod schemas at `src/hooks/yaml-loader.ts`, `src/vault.ts`, `src/reconcile.ts`, and `src/quarantine-store.ts` — but `enforced_at:` points at `src/tools/schemas.ts`, which the gotcha body explicitly calls "the precedent the boundary loaders converge on" (not where the scar lives). The lockstep test passes because the path exists, but readers tracing the scar will land at the wrong place.
- **Evidence:** `gotchas/boundary-validation-via-zod.md:8` — `enforced_at: src/tools/schemas.ts`. The gotcha body §"Root cause" names the four scar sites; §"Structural mitigation" names `src/tools/schemas.ts` as the precedent the future schemas should converge on.
- **Recommended fix:** Either (a) point `enforced_at:` at one of the four scar sites (e.g., `src/hooks/yaml-loader.ts` — the loader most likely to be the first implementation target per the F6 implementation list), or (b) accept that this gotcha is genuinely "deferred-mitigation" and use `coverage: deferred` instead of `coverage: off-matrix`. Option (a) matches `transitive-llm-dependency.md`'s anchoring shape (pointing at the test file that pins the contract).
- **Substrate artifact to add or update:** `gotchas/boundary-validation-via-zod.md` frontmatter.

### I3. `openVault` return-shape inconsistency between sections

- **Severity:** Low
- **Category:** Domain model
- **Why it matters:** §"Vault" describes the Vault as if `openVault` returns it directly (line 16: "Constructed by `openVault(path: string)`"). §"Vault lifecycle" diagram (line 54) shows `openVault(path) → Vault`. The new §"Composable construction" code at line 74 wraps the return in `Promise<Result<Vault, ToolError>>`. The composable example at line 416 (the `DomeMcpServer` usage) shows the unwrap (`if (!vaultR.ok) throw vaultR.error`). The Result-wrap is the truth; the earlier-section prose is shorthand. A first-time reader of §"Vault" will write code against the wrong signature.
- **Evidence:** `sdk-surface.md:16` "Constructed by `openVault(path: string)`"; `:54` diagram; `:74` `async function openVault(root: string): Promise<Result<Vault, ToolError>>`; `:416` `if (!vaultR.ok) throw vaultR.error`.
- **Recommended fix:** Update §"Vault" intro and the §"Vault lifecycle" diagram to show the `Result` wrap explicitly, or add a one-sentence pin: "`openVault` returns `Promise<Result<Vault, ToolError>>`; the diagrams below elide the unwrap step for clarity."
- **Substrate artifact to add or update:** `specs/sdk-surface.md` §"Vault" and §"Vault lifecycle".

## What looked right

- The `openVault` composable-construction shape with `vaultRef: { current: Vault | null }` is a real structural fence — the three positional-ordering rules earlier carried as inline comments collapse into one setter; the bootstrap-order test pins the construction against the next contributor's "tidying up" instinct.
- The off-matrix lockstep convention (delegating-stub via `await import(...)` rather than `expect(true).toBe(true)`) is the right kind of substrate move: it converts a reviewer-memory rule into a structural fence the AC3 meta-check enforces.
- The three "Adding a new X" recipes (CLI command, hook, invariant) all parallel the existing "Adding an 8th Tool" shape — same enumerate-the-edits surgical form, same closes-the-class lockstep tests named. Symmetry without ceremony.
- The one-`PromptLoader`-per-Vault contract in §"Prompt loading lifecycle" correctly names the v1+ shape (`Vault.prompts` projection) it anticipates without committing to it, and preserves the bundling-axiom split.

## Recommended repairs (ranked)

1. Fix the ledger preamble spec count (B1) — one-line edit; closes the substrate-count-drift in the count-drift ledger.
2. Add `docs/wiki/linters/` and `docs/index.md` to `no-retired-symbol-names`'s exclusion list, or move the allow-list to `src/types.ts` (B2).
3. Re-anchor `enforced_at:` on `substrate-count-drift.md` and `boundary-validation-via-zod.md` to test-or-scar paths, not convention paths (B3 + I2).
4. Add `buildInstructions(vault)` to the `sdk-surface.md` §"Distribution" enumeration and the consumer-surface matrix (I1).
5. Pin `openVault`'s return shape consistently across §"Vault", §"Vault lifecycle", and §"Composable construction" (I3).
