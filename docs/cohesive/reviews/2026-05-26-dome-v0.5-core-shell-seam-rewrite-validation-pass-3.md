# Rewrite Validation Review — Dome v0.5 core-shell seam (Phase B pass-3)

**Verdict:** Approved

## Architectural reflection

The core/shell seam is structurally honest. Pulling `aiTools`/`toolParsers` off the Vault interface and naming `ConsumerSurface` as a runtime construct means a future mobile/voice/web shell can import `openVault` with no LLM tax — the conceptual seal and the bundling seal now agree. The axiom + entrypoint split + matrix triangle is mutually load-bearing; removing any one would unravel the others.

- **Easier downstream:** Adding a v1+ HTTP/voice/mobile shell becomes a row-add on the consumer-surface matrix plus a single entrypoint import — the dep-cost is automatic.
- **Harder downstream:** Any future shell that wants `ConsumerSurface` today must reach through `@dome/sdk/mcp` and pay the `@modelcontextprotocol/sdk` dep. The matrix's Future HTTP row commits to this; the deferred `@dome/sdk/http`/`buildHttpSurface(vault)` split is the rewrite's acknowledged escape hatch.
- **Load-bearing on memory:** Two enumerations remain bound to reviewer attention until lockstep tests land — the axiom count in `sdk-surface.md §"Tiered feature model"` (I1; closed in this same pass after the review per the rubric) and per-cell entrypoint claims in `consumer-surface.md`. `bundle-deps.test.ts` will pin the axiom; no test pins the matrix cells.
