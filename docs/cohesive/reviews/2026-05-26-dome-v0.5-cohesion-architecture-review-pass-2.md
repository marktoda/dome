# Architecture Review — Dome v0.5 (pass 2)

**Date:** 2026-05-26
**Scope:** whole repository (`/Users/mark.toda/dev/dome`)
**Branch:** `design/dome-v0.5-core-shell-seam`
**Pass-1 review:** [`2026-05-26-dome-v0.5-cohesion-architecture-review.md`](2026-05-26-dome-v0.5-cohesion-architecture-review.md)
**Future pressure threaded:** v0.5 usability end-to-end; v1+ extensibility with multiple surfaces (CLI as one consumer); concept simplification/collapse; *labels on the new core/shell seam*.

**Verdict (internal):** Cohesive but under-enforced.
**Verdict (user-facing):** The core/shell seal is real now, the substrate is denser than it was last pass, and most pass-1 findings closed structurally. What remains is tightening *labels on the new seam* — three places where the docs promise a property the code only half-enforces. Closing them is a single small rewrite-specs pass and makes v1's next consumer surface (TUI, web, daemon, voice) a ~200-line job.

## Thesis

The pass-1 architecture review made the case that the core/shell seal was half-built. The implementation closed that gap structurally: `src/index.ts` no longer re-exports LLM or MCP machinery, `tests/integration/bundle-deps.test.ts` walks the static graph and pins the axiom, `ConsumerSurface` names the four-kind aggregation, `vault.close()` exists with a flag the dispatcher consults, and lockstep tests now bind matrix+gotcha docs to test files. The substrate's pass-1 question — *"can a v1 contributor predict the right answer for most changes"* — is now genuinely yes for the structural shape of the codebase.

What this pass surfaces is that **every Phase B closure shipped a thin layer of label-ahead-of-structure drift** — three places where the spec claims a property and the code's enforcement of it is partial or split. The new `WORKFLOWS_KNOW_VAULT_CONTEXT` axiom is in the substrate index, has a doc, has runtime tests, but is *not in the `INVARIANTS` const* — so the AC3 lockstep silently skips it. `ConsumerSurface` is named the "protocol-agnostic four-kind aggregation" but its `tools` field is MCP-rendered (`dome.*` snake_case, MCP handler signature), and the matrix already admits one ConsumerSurface per protocol. The intrinsic hook-dispatch wrap is declared at `sdk-surface.md:46` as "a property of the Vault, not a per-call-site decorator" — and is implemented at *two* call sites (`registry.ts:230-243` and `ai-sdk-binding.ts:34-41`) with no shared helper. These three are convergent — substrate-alignment, structure, library-native, and agent-readiness reviewers each surfaced one of them from a different angle, and they share the same shape.

The highest-leverage forward move is *aligning the labels to the structure* before the next consumer surface lands. The implementation is small (one const entry, one helper extraction, one type split or rename) and unlocks the v1 question: a TUI/web/voice/daemon shell becomes a thin protocol-renderer over a protocol-agnostic `AbstractSurface`, not yet-another bespoke aggregation. Defer this and v1 mobile/desktop/voice each pay a tax — duplicated bindTools wraps, duplicated PromptLoader walks, duplicated ToolAdapter projections, plus an axiom that drifts every release because the lockstep doesn't catch it. The system can scale development without founder memory; what it can't yet do is *prove* its own promises structurally at the new seam.

## Highest-leverage findings

### F1. `WORKFLOWS_KNOW_VAULT_CONTEXT` shipped without its lockstep slot — four enforcement layers bypass it

**Severity:** High
**Category:** Spec drift / Named-invariant enforcement gap

**Why it matters:** The 14th axiom is in `docs/index.md:35` and has `docs/wiki/invariants/WORKFLOWS_KNOW_VAULT_CONTEXT.md` declaring it non-disable-able. But four structural enforcement layers all silently skip it:

1. **`INVARIANTS` const omits it.** `src/types.ts:68-82` carries 13 entries; no `WORKFLOWS_KNOW_VAULT_CONTEXT`. Verified by grep.
2. **AC3 lockstep skips it.** `tests/integration/invariant-coverage.test.ts:13-22` iterates `Object.entries(INVARIANTS)` and requires `tests/invariants/<slug>.test.ts`. Because the const omits the entry, no test is required; no `tests/invariants/workflows-know-vault-context.test.ts` exists. Verified by `ls tests/invariants/`.
3. **`InvariantName` type union excludes it.** `InvariantName = typeof INVARIANTS[keyof typeof INVARIANTS]` at `src/types.ts:84` cannot ever carry `"WORKFLOWS_KNOW_VAULT_CONTEXT"`. A future `ToolError` of kind `invariant-violated` for the axiom cannot be expressed type-safely.
4. **The preamble pipeline lacks an order or extension contract test.** `tests/workflows/agent-loop.test.ts:175-195` asserts the composer joins with blank lines and produces ≥2 sections, but does not pin order (vault-identity before rendering-surface) or that empty-returning preambles drop.
5. **The migrate-bootstrap convention is reviewer-memory, not lockstep.** `WORKFLOWS_KNOW_VAULT_CONTEXT.md:40` says "an analogous assertion should be added to every future workflow-driven CLI command" — no test enforces this for `lint`, `export-context`, or future shims.

A future contributor adding a 15th invariant by following the lockstep pattern lands in a confusing state: the 14th invariant they thought existed isn't actually in the const. A future agent moving the preamble order silently regresses the axiom.

**Evidence:** `/Users/mark.toda/dev/dome/src/types.ts:68-82`; `/Users/mark.toda/dev/dome/docs/index.md:35`; `/Users/mark.toda/dev/dome/docs/wiki/invariants/WORKFLOWS_KNOW_VAULT_CONTEXT.md:9,40`; `/Users/mark.toda/dev/dome/tests/integration/invariant-coverage.test.ts:13-22`; `/Users/mark.toda/dev/dome/src/workflows/agent-loop.ts:188-191,238-253`; `/Users/mark.toda/dev/dome/tests/workflows/agent-loop.test.ts:175-195`.

**Recommended fix:** (a) Add `WORKFLOWS_KNOW_VAULT_CONTEXT: "WORKFLOWS_KNOW_VAULT_CONTEXT"` to `INVARIANTS` const. (b) Create `tests/invariants/workflows-know-vault-context.test.ts` — either as a lockstep pointer (re-importing the agent-loop assertions, parallel to `core-has-no-llm-or-mcp-dependency.test.ts:1-20`), or by relocating the preamble tests from `tests/workflows/agent-loop.test.ts`. (c) Add two assertions to that test: preamble order (`composed.indexOf("# Current vault") < composed.indexOf("# Rendering surface")`) and empty-output drop (inject an empty-returning preamble into a test copy, assert it doesn't introduce a blank section). (d) If the migrate-bootstrap convention is real, add `tests/integration/workflow-cli-preamble-coverage.test.ts` enumerating workflow-driven CLI commands and asserting each has the bootstrap test; if not real, downgrade the doc claim.

**Substrate artifact to add or update:** `src/types.ts` const + new `tests/invariants/workflows-know-vault-context.test.ts` + extension assertions; optional new `tests/integration/workflow-cli-preamble-coverage.test.ts`; possible edit to `WORKFLOWS_KNOW_VAULT_CONTEXT.md:13-18` if the extension model is "SDK-internal in v0.5" (the registry shape is private-module per agent-readiness F2).

### F2. `ConsumerSurface` is named protocol-agnostic, but `tools` is MCP-rendered — the v1 HTTP seam already broke

**Severity:** High
**Category:** Architectural seam / Naming-vs-implementation drift

**Why it matters:** `src/mcp/consumer-surface.ts:20-25` declares `tools: ReadonlyArray<ToolAdapter>`. `ToolAdapter` is *protocol-rendered* — `dome.*` snake_case names, the MCP handler signature `(input) => Result`. `src/mcp/tool-adapters.ts:48-54` literally says so in the comment: *"the `ToolAdapter` shape is MCP-protocol-shaped."* The spec at `wiki/specs/sdk-surface.md §"Consumer surfaces"` describes `ConsumerSurface` as the "protocol-agnostic four-kind aggregation." The matrix at `wiki/matrices/consumer-surface.md:33` already encodes the contradiction: Future HTTP / ConsumerSurface cell reads "a future `@dome/sdk/http` could expose its own `buildHttpSurface(vault)`" — i.e., one `ConsumerSurface` per protocol. The name promises agnostic; the type is bound.

This is the v1-extensibility tax the pass-1 review tried to head off. A v1 web shell that wants to expose Dome over HTTP cannot consume the current `ConsumerSurface` — it must define `HttpConsumerSurface` and reproduce the aggregation. A v1 desktop shell wanting to expose Dome to a local IPC bridge must do the same. The duplication Phase B was structurally preventing reappears at the protocol-adapter layer.

The pass-1 architecture review's F2 named "MCP-style aggregation needs a name." Phase B gave it the right name and the wrong type signature.

**Evidence:** `/Users/mark.toda/dev/dome/src/mcp/consumer-surface.ts:20-25`; `/Users/mark.toda/dev/dome/src/mcp/tool-adapters.ts:48-54` (the comment admitting MCP-shape); `/Users/mark.toda/dev/dome/docs/wiki/specs/sdk-surface.md §"Consumer surfaces"`; `/Users/mark.toda/dev/dome/docs/wiki/matrices/consumer-surface.md:33` (Future HTTP cell admitting protocol-per-surface).

**Recommended fix:** Two options, pick by taste.

- **Option A (cheap, honest):** Rename `ConsumerSurface` → `McpConsumerSurface` and admit it lives fully in `@dome/sdk/mcp`. Update `sdk-surface.md §"Consumer surfaces"` to name it as the MCP-protocol surface. Future HTTP/voice surfaces define siblings.
- **Option B (structural, future-cheap):** Split into `AbstractSurface = { tools: BoundToolSurface, prompts: PromptDescriptor[], resources: ResourceDescriptor[], instructions: string }` (lives in `@dome/sdk` core; tools as the canonical Vault-bound surface, not protocol-rendered) and `McpSurface = renderMcp(AbstractSurface)` (lives in `@dome/sdk/mcp`; applies `dome.*` snake_case + MCP handler signature). A future `renderHttp(AbstractSurface)` is a 50-line file.

Option B is what the spec promises; option A is what the code does. Option B costs ~200 lines of refactor and prevents the v1 surface tax.

**Substrate artifact to add or update:** `wiki/specs/sdk-surface.md §"Consumer surfaces"` (rewrite to match the chosen option); `wiki/matrices/consumer-surface.md` (rewrite the Future HTTP / Future voice rows to consume the abstract layer); if option B, new `wiki/specs/sdk-surface.md §"Surface rendering"` subsection; if option A, rename across spec/matrix/code in lockstep.

### F3. The intrinsic hook-dispatch wrap has two implementations — drift is one PR away

**Severity:** High
**Category:** Pattern propagation / Convergent reviewer signal

**Why it matters:** `src/tools/registry.ts:230-243` (in `bindEntry`, called by `bindTools`) and `src/tools/ai-sdk-binding.ts:34-41` (in `bindAiSdkTools`) both implement the same post-invoke logic: *if mutating, await `vault.dispatchEvents(projectEffectsToEvents(out.effects))`*. The pass-2 final-substrate-review explicitly named "the hook-dispatch wrap is intrinsic to the Vault-bound Tool registry, not a per-call-site decorator" as the substrate-load-bearing seam (`final-substrate-review-pass-2.md §"New substrate seam introduced"`). The spec at `sdk-surface.md:46` repeats this: *"the wrap is a property of the Vault, not a per-call-site decorator."* Structurally, there are two implementations.

`tests/integration/mcp-hook-dispatch.test.ts` pins the MCP path; no symmetric test pins the AI-SDK path. A future change to one wrap (adding causation metadata, a backpressure gate, a `closed`-flag pre-check à la F7-vault-close) must be made in two places, or silently regresses. Worse: a future projection (`projectHttp(vault)`, `projectGrpc(vault)`) will copy whichever implementation the contributor sees first; depending on which, the new path either inherits or skips the future enhancement.

This is the most predictable wrong move the next six months: an agent will add a metadata field to one wrap site and not the other.

**Evidence:** `/Users/mark.toda/dev/dome/src/tools/registry.ts:230-243`; `/Users/mark.toda/dev/dome/src/tools/ai-sdk-binding.ts:34-41`; `/Users/mark.toda/dev/dome/src/mcp/tool-adapters.ts:56-68` (the correct delegation: MCP path uses `bindTools(...).parsers`); `/Users/mark.toda/dev/dome/docs/cohesive/reviews/2026-05-26-dome-v0.5-core-shell-seam-final-substrate-review-pass-2.md §"New substrate seam introduced"`.

**Recommended fix:** Extract `wrapMutatingInvoke(vault, entry, writer): (input) => Promise<ToolReturn<unknown>>` (or `withHookDispatch(vault, invoke)`) to `src/tools/registry.ts` exported internally. `bindEntry` consumes it; `bindAiSdkTools` consumes it; future `projectHttp`/`projectGrpc` consume it. Add a test parallel to `mcp-hook-dispatch.test.ts` for the AI-SDK path. Optionally promote `HOOK_DISPATCH_IS_VAULT_BOUND` to a named invariant — the pass-2 review suggested this and deferred it; the textual single-source-helper is the structural fence the invariant needs to become real.

**Substrate artifact to add or update:** Code extraction + new test. Optional: `docs/wiki/invariants/HOOK_DISPATCH_IS_VAULT_BOUND.md` (axiom-tier) with bundle-test-style enforcement (a code-graph check that mutating Tool invocations across the codebase route through `wrapMutatingInvoke`).

### F4. `projectMcp` and `buildToolAdapters` are the same concept exported twice from `@dome/sdk/mcp`

**Severity:** High
**Category:** Concept overlap / Duplicate substrate

**Why it matters:** `src/mcp/project-mcp.ts:21-25` exports `projectMcp(vault): McpProjection` returning `{ parsers }`. `src/mcp/tool-adapters.ts:56-69` exports `buildToolAdapters(vault): ToolAdapter[]` which builds the same parsers and wraps them with `dome.*` name + description + schema. **Evidence:** `src/mcp/project-mcp.ts:21-25`; `src/mcp/tool-adapters.ts:56-69`; `src/mcp/consumer-surface.ts:41`; `src/mcp/index.ts:13,16`. Pick (a) unless F2 option B is chosen.

**Substrate artifact to add or update:** Code deletion (option a) or refactor (option b); update `wiki/specs/sdk-surface.md §"Vault surface"` `projectMcp` line; update `wiki/matrices/consumer-surface.md:25` Tools-column citation.

### F7. `vault.close()` doesn't guard `vault.tools.X` — long-running consumers will assume it does

**Severity:** Medium
**Category:** Misleading documentation / v1-consumer trap

**Why it matters:** `wiki/specs/sdk-surface.md §"Vault lifecycle"` (post-N2 closure) says: *"calls become silent no-ops — the dispatcher accepts no new event work after close."* A v1 desktop/Electron contributor reading "close the vault" assumes Tool calls fail after close — that's the meaning of "close" in every database / file-handle / connection-pool they've seen. The current shape silently writes the file (Tool succeeds) while no-oping the hooks (no index update, no log entry, no event projection). For a desktop shell that opens/closes vaults across user sessions, this is a worst-case failure mode: state diverges from index without anyone noticing.

The consumer-surface matrix at `consumer-surface.md` lists Future desktop and Future mobile rows without warning. The spec admits the gap as a v0.5 limitation, but doesn't promote the warning to the matrix or a gotcha. Pass-2 N2 closed the *observable no-op* test gap; the *Tool-call guard* gap is structurally analogous but unaddressed.

**Evidence:** `/Users/mark.toda/dev/dome/src/vault.ts:226-286` (close-flag set; tool surface unchanged); `/Users/mark.toda/dev/dome/docs/wiki/specs/sdk-surface.md §"Vault lifecycle"`; `/Users/mark.toda/dev/dome/docs/wiki/matrices/consumer-surface.md` (desktop + mobile rows).

**Recommended fix:** Two options. (a) Guard `vault.tools.X` after close: each parser checks the `closed` flag and returns `ToolError` of kind `vault-closed`. ~15 lines. (b) Promote to gotcha `docs/wiki/gotchas/post-close-tool-call.md` and reference it from `consumer-surface.md` for long-running-shell rows. Option (a) closes the gap honestly; option (b) admits it.

**Substrate artifact to add or update:** Either code guard + new `ToolError` kind + test, or new gotcha + matrix cross-reference.

### F8. Smaller concept collapses + remaining substrate gaps

**Severity:** Low-Medium (clustered)
**Category:** Pattern propagation / Missing seams

**Why it matters:** Six smaller findings that share a shape — each is below blocking severity but together they show *where the Phase B refactor accreted instead of collapsed*:

- **`PromptLoader` re-instantiation cascade.** `new PromptLoader(vault)` runs in `WorkflowRegistry` constructor, `buildPromptAdapters`, and `buildInstructions` (`src/prompts/registry.ts:17`, `src/mcp/prompt-adapters.ts:33`, `src/mcp/instructions-builder.ts:16`). Each independently walks `<vault>/.dome/prompts/` + `BUILTIN_DIR`. Structurally the same shape as the pre-Phase-B `vault.aiTools` problem. **Fix:** Have `buildConsumerSurface` construct one `PromptLoader` and pass it to both helpers; OR add `vault.prompts` projection.

- **`projectAiSdk` vs `bindAiSdkTools` vs `buildAiSdkTools` — three names, one transformation chain.** All exported from `@dome/sdk/workflows`. **Fix:** Keep `projectAiSdk` as the public surface; make `bindAiSdkTools` non-exported (internal to `ai-sdk-binding.ts`); drop `buildAiSdkTools` (an alias).

- **`WorkflowName` const-object vs `TOOL_NAMES` const-tuple — two patterns for "named-string-union plus iteration."** v1 contributors reading both files in one sitting have no rule. **Fix:** Pick one (probably the `TOOL_NAMES` tuple pattern), migrate `WorkflowName`, or write a `wiki/gotchas/canonical-string-enum-shape.md` documenting both as deliberate.

- **`SYSTEM_PREAMBLES` is misnamed — it's a pipeline, not a registry.** `agent-loop.ts:80-85` calls it a "preamble registry"; the shape is a const-array of pure functions composed in order. Calling it a registry oversells it. **Fix:** Rename to `SYSTEM_PREAMBLE_PIPELINE` or update the comment to "ordered pipeline of preamble functions."

- **`SYSTEM_PREAMBLES` is private-module — `WORKFLOWS_KNOW_VAULT_CONTEXT.md:13-18,47` implies plugin/vault extension that the code doesn't offer.** A v1 voice-shell contributor wanting to inject "today's date" or "active speaker" finds no `registerPreamble()` seam and either edits the core file or monkey-patches downstream. **Fix:** Either export `registerPreamble()` from `@dome/sdk/workflows`, OR be explicit in the invariant doc: "preambles are SDK-internal in v0.5; plugin/vault preamble registration deferred to v0.5.1+."

- **Vault-bootstrap temporal-dead-zone scar.** `src/vault.ts:227-235` builds `dispatchEvents` before `tools` is assigned via `bindTools` at line 249. Works only because the closure captures the `const` binding lazily. A v1 contributor refactoring `openVault` for clarity will trip on this. Pass-2 noted the scar; no gotcha doc landed. **Fix:** Add `docs/wiki/gotchas/vault-bootstrap-temporal-dead-zone.md` OR restructure `bindTools` to accept a `dispatchEventsRef` setter so the wiring is explicit.

- **`instructions-builder.ts` is the cold-start contract — no spec names the four required sections.** A v1 voice contributor building `buildVoiceConsumerSurface(vault)` has no canonical spec for "the four things that go in `instructions`." Pass-1 F8 recommended `wiki/specs/cold-start.md`; it didn't ship. **Fix:** Either write the spec, OR make `buildInstructions` produce a structured object (`{ systemBase, invariants, pageTypes, vaultNotes }`) so future surfaces project rather than re-parse markdown.

- **Repo-root entry point missing.** No top-level `README.md`, `CLAUDE.md`, or `AGENTS.md` at the worktree root.
