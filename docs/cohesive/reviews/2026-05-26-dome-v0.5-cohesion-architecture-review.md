# Architecture Review — Dome v0.5

**Date:** 2026-05-26
**Scope:** whole repository (`/Users/mark.toda/dev/dome`)
**Future pressure threaded:** v0.5 end-to-end usability; v1+ extensibility with multiple surfaces (CLI as one consumer, not the system); concept simplification/collapse; core/surface boundary clarity.

**Verdict (internal):** Cohesive but under-enforced.
**Verdict (user-facing):** The architecture is sound, the specs and code mostly agree, and the substrate is unusually rich for a v0.5 — but the boundary between "core SDK" and "consumer shell" is half-built. Pinning a few more rules structurally now will make v1's multi-surface work cheap; deferring it will make every new consumer pay a tax.

## Thesis

Dome v0.5 has a sealed four-concept core (Vault, Document, Tool, Hook) that the implementation respects: the seven Tools live in one registry; every named invariant has a test file; the privileged writer is correctly partitioned by `hook.source === "sdk"`; the per-workflow atomic commit lines up `log.md` with `git log` at byte-identical subject lines. The substrate consolidation work of the last 15 commits — single-source Tool registry (36c5e1b), unified `BoundToolSurface` (4ebd070), `@dome/sdk` vs `@dome/sdk/cli` split (9d7ca1f), one-source shipped defaults (463aae8) — has materially reduced the conceptual surface a future contributor must hold in their head to make a correct change.

The highest-leverage risk is that this consolidation **stopped one layer short of the v1-extensibility seam**. `Vault.aiTools` is built eagerly on every `openVault`; `src/index.ts` re-exports `runWorkflow` and `DomeMcpServer` directly; `import { openVault } from "@dome/sdk"` transitively pulls `@anthropic-ai/sdk` and `@modelcontextprotocol/sdk` into every consumer's bundle. A v1 web shell that wants only typed-markdown reads, a mobile shell that holds a long-lived Vault across user turns, a voice client that writes to `inbox/voice/`, and a plugin SDK author who only needs the seven Tools all hit the same wall — and there is no named invariant, matrix, or `dome doctor` check that would catch a future regression that pulls more into core.

The system can scale development without founder memory: the substrate is dense enough that a v1 contributor can predict the right answer for most changes. The exceptions are the seven findings below; each names a place where convention substitutes for structural enforcement.

## Highest-leverage findings

### 1. The core/shell seal is half-built; `@dome/sdk` still transitively bundles Anthropic + MCP

- **Severity:** High
- **Category:** Architecture seam / Locality
- **Why it matters:** `wiki/specs/sdk-surface.md:265-269` claims "a future mobile / web / voice shell consumes only `@dome/sdk` and never pulls Commander or the seven `dome <cmd>` implementations into its bundle." That's structurally true for Commander (the recent split worked). It's structurally **false** for the AI SDK and the MCP SDK: `src/tools/registry.ts:16` imports `ai` at the top level; `src/index.ts:54-93` re-exports `runWorkflow`, `WorkflowRegistry`, `DEFAULT_MODEL`, `DomeMcpServer`, `buildToolAdapters`, `buildPromptAdapters`, `ResourceAdapter` from the core surface; `src/vault.ts:51` declares `aiTools: import("ai").ToolSet` and `bindTools` at `src/tools/registry.ts:264` builds it eagerly inside `openVault`. The result: `import { openVault } from "@dome/sdk"` reaches `@anthropic-ai/sdk` + `@modelcontextprotocol/sdk` transitively. A v1 web shell that wants only the read-side (search, link-resolve, read) ships Anthropic and MCP regardless.
- **Evidence:** `/Users/mark.toda/dev/dome/src/tools/registry.ts:16`; `/Users/mark.toda/dev/dome/src/vault.ts:51,197`; `/Users/mark.toda/dev/dome/src/index.ts:54-93`; the seal claim at `/Users/mark.toda/dev/dome/docs/wiki/specs/sdk-surface.md:265-269`.
- **Recommended fix:** Apply the same split that worked for the CLI. Introduce `@dome/sdk/workflows` (carries `runWorkflow`, `WorkflowRegistry`, `PromptLoader`, `aiTools`, `@ai-sdk/anthropic`, `ai`) and `@dome/sdk/mcp` (carries `DomeMcpServer`, adapters, `@modelcontextprotocol/sdk`). Keep `@dome/sdk` to Vault / Document / Tool / Hook + the seven Tools + privileged-writer + reconcile + watcher. Lazy-build `aiTools` / `toolParsers` via getters on `Vault` so a consumer that never reads `vault.aiTools` doesn't pay the `ai` import cost. The `runWorkflowAtPath` primitive at `src/cli/run-workflow-at-path.ts` belongs in `@dome/sdk/workflows`, not in `cli/`.
- **Substrate artifact to add or update:** New invariant `docs/wiki/invariants/CORE_HAS_NO_LLM_OR_MCP_DEPENDENCY.md` (axiom-tier). New matrix `docs/wiki/matrices/consumer-surface.md` (rows = consumer shells, columns = exported symbol families). New gotcha `docs/wiki/gotchas/transitive-llm-dependency.md`. New regression test `tests/integration/bundle-deps.test.ts` that asserts the `@dome/sdk` core entrypoint does not transitively depend on `ai`, `@ai-sdk/anthropic`, or `@modelcontextprotocol/sdk`.

### 2. No named seam for "MCP-style aggregation" — every v1 surface will reinvent it

- **Severity:** High
- **Category:** Missing concept
- **Why it matters:** `src/mcp/server.ts:12-37` aggregates four things for one consumer: Tools, prompts (`PromptAdapter[]`), resources (`ResourceAdapter`), instructions (`buildInstructions(vault)`). This pattern — "an SDK surface that bundles the four kinds the SDK exposes for one consumer shell" — has no name. v1 mobile, desktop, voice, and web will each need the same aggregation (tools to call + prompts to invoke + resources to fetch + cold-start instructions). Without a named `ConsumerSurface` concept, each new shell duplicates the wiring; MCP-specific concerns (the prompt-adapter shape, the `dome://` URI scheme for resources) leak into shells that don't speak MCP. The substrate-discovery report independently named this as "the 'surface' pattern that does not yet have a name."
- **Evidence:** `/Users/mark.toda/dev/dome/src/mcp/server.ts:12-45`; `/Users/mark.toda/dev/dome/src/mcp/instructions-builder.ts`; the v1+ surface enumeration at `/Users/mark.toda/dev/dome/docs/wiki/specs/harnesses.md:62-69`.
- **Recommended fix:** Promote the aggregation to a named protocol-agnostic concept. `ConsumerSurface = { tools, prompts, resources, instructions }` built once per Vault. `DomeMcpServer` becomes a thin protocol adapter over `ConsumerSurface`. Future HTTP / mobile surfaces consume the same `ConsumerSurface`. The `aiTools` + `toolParsers` projections that today live on `Vault` (Finding 1) can also become projections on `ConsumerSurface`.
- **Substrate artifact to add or update:** New section `wiki/specs/sdk-surface.md` §"Consumer surfaces" naming the concept. Update `wiki/specs/harnesses.md` §"Future-harness pressure" to declare that v1 surfaces compose against `ConsumerSurface`, not by re-wiring directly against `Vault`.

### 3. `dome doctor` silently no-ops 6 documented flags; `INBOX_IS_EPHEMERAL` is structurally undefended

- **Severity:** High
- **Category:** Spec drift
- **Why it matters:** `wiki/specs/cli.md:135-145` describes `--drain-hooks`, `--recent-activity`, `--reset-quarantined-hooks`, `--show review-queue`, `--show raw-citations`, `--show recent-hook-cycles` as functional diagnostic surfaces. `src/cli/commands/doctor.ts:254-259` accepts each and writes `info.push("--<flag>: no-op in v0.5")` then exits clean. At least `--drain-hooks` and `--recent-activity` are cheaply implementable (`vault.drainHooks()` exists at `vault.ts:67,220`; `log.md` has parseable timestamps already walked at `doctor.ts:217`). Separately, `wiki/invariants/INBOX_IS_EPHEMERAL.md:23` declares "`dome doctor` reports any vault where `inbox/<bucket>/` contains files older than 24 hours" and `wiki/matrices/tool-invariant-enforcement.md:15,40` repeats this as the structural check (since the invariant is workflow-prompt-enforced, the doctor check is the only structural fallback). `doctor.ts` contains zero inbox-age logic; `tests/invariants/inbox-is-ephemeral.test.ts` only asserts intake prompts contain `"inbox"` and `"deleteDocument"` literals — the doctor-check claim is not exercised.
- **Evidence:** `/Users/mark.toda/dev/dome/docs/wiki/specs/cli.md:135-145` vs `/Users/mark.toda/dev/dome/src/cli/commands/doctor.ts:254-259`; `/Users/mark.toda/dev/dome/docs/wiki/invariants/INBOX_IS_EPHEMERAL.md:23`; `/Users/mark.toda/dev/dome/docs/wiki/matrices/tool-invariant-enforcement.md:15,40`; `/Users/mark.toda/dev/dome/tests/invariants/inbox-is-ephemeral.test.ts:13-37`.
- **Recommended fix:** (a) Wire the two cheap flags: `--drain-hooks` → `vault.drainHooks()`; `--recent-activity` → walk `log.md` for the last N entries. (b) Demote the other four to a `## Not yet shipped (v0.5)` section in the CLI spec, OR make the CLI refuse them with a usage error. (c) Add an inbox-age walk to `domeDoctor` (threshold from `.dome/config.yaml`, default 24h) emitting a violation per stale file; extend the invariant test to backdate a fixture inbox file and assert the violation appears.
- **Substrate artifact to add or update:** Update `docs/wiki/specs/cli.md` §"`dome doctor` Flags" to partition shipped vs `v0.5.1+` flags. Update `docs/wiki/invariants/INBOX_IS_EPHEMERAL.md` §"Structural enforcement" to either remove the doctor-check claim or reference the new code. Update `tests/invariants/inbox-is-ephemeral.test.ts`.

### 4. Substrate-count drift is now a pattern — needs structural defense

- **Severity:** Medium
- **Category:** Spec drift
- **Why it matters:** Three live count-drift instances now: (a) `docs/cohesive/delta-ledgers/2026-05-25-dome-v0.5-foundation.md:189` records `Invariants ... — split 5 axiom + 3 default + 2 opt-in` (11 total), while live state is 6 + 4 + 2 = 12 invariants per `src/types.ts:68-81` and `tests/integration/public-surface.test.ts`. (b) `docs/wiki/syntheses/v0.5-build-plan.md:27` reads `CLI: 5 commands (init, migrate, serve, lint, doctor, export-context)` — six items, claimed as five, missing `reconcile` (which is in `cli.md` and `cli.ts`); live state is 7. (c) `docs/wiki/syntheses/v0.5-build-plan.md:25` reads "8 workflow prompts" but live state is 9 per `src/workflows/workflow-name.ts` and the AC3 workflow-count test. Each instance is low individually; collectively they show a missing structural check. - **Evidence:** `/Users/mark.toda/dev/dome/docs/cohesive/delta-ledgers/2026-05-25-dome-v0.5-foundation.md:189`; `/Users/mark.toda/dev/dome/docs/wiki/syntheses/v0.5-build-plan.md:25,27`; canonical sources `/Users/mark.toda/dev/dome/src/types.ts:68-81` (INVARIANTS) and `/Users/mark.toda/dev/dome/src/workflows/workflow-name.ts` (WORKFLOW_NAMES).

(b) Add a `dome doctor` semantic-linter rule (or a dedicated `dome doctor --check-substrate-counts` mode) that diffs documented counts against the const arrays they cite. (c) Promote the lesson to a gotcha doc so future contributors writing synthesis prose know to link rather than inline.
- **Substrate artifact to add or update:** New gotcha `docs/wiki/gotchas/substrate-count-drift.md`. New `dome doctor` mode `--check-substrate-counts`. Edits to the two synthesis docs + the delta ledger to align counts.

### 5. AI-SDK `Tool<>` generic cast at `registry.ts:128` hides a real version-coupling fragility

- **Severity:** Medium
- **Category:** Type circumvention
- **Why it matters:** The registry's load-bearing claim (`src/tools/registry.ts:9-13`) is "we use AI SDK's native `Tool<INPUT, OUTPUT>` so the SDK names the right fields." But `tool({...})` is double-cast through `unknown` at line 128 because the generic doesn't line up with the SDK's actual `Tool` signature in v6 — the SDK now infers from `inputSchema` rather than from the explicit generic. The cast pairs with `satisfies Record<ToolName, ToolRegistryEntry<any, any>>` at line 215 (acknowledging the variance workaround in the comment at 210-214). Both are working around the same problem. If AI SDK v6 narrows or renames `execute` / `inputSchema` again, the casts keep compiling while breaking at runtime. The `compactX` helpers at `src/tools/schemas.ts:68-121` are six functions whose entire job is stripping `undefined` keys to satisfy `exactOptionalPropertyTypes` — also a workaround for the same boundary friction.
- **Evidence:** `/Users/mark.toda/dev/dome/src/tools/registry.ts:128,210-215`; `/Users/mark.toda/dev/dome/src/tools/schemas.ts:68-121`.
- **Recommended fix:** Either drop the explicit `<TInput, TOutput>` generic on `entry()` and let AI SDK infer (return `ReturnType<typeof tool>` instead of the cast), or pin a single `InferToolInput<S>` / `InferToolOutput<S>` alias derived from the Zod schema so the generics agree end-to-end. The `satisfies Record<ToolName, ToolRegistryEntry<any, any>>` then becomes `satisfies Record<ToolName, ToolRegistryEntry>` with no `any`. Plan a Zod v4 migration alongside the next AI SDK bump (v6 exports `Tool<v3>` and `Tool<v4>` shapes) — that's the trigger for collapsing the `compactX` helpers.
- **Substrate artifact to add or update:** New gotcha `docs/wiki/gotchas/ai-sdk-tool-variance.md` documenting why the cast exists today (load-bearing scar) and the migration trigger (next AI SDK major bump). A line in `wiki/specs/sdk-surface.md` §"Dependencies" naming the Zod major version Dome commits to and the migration trigger.

### 6. No lockstep test for matrices or gotchas — only invariants are enforced

- **Severity:** Medium
- **Category:** Pattern propagation
- **Why it matters:** `tests/integration/invariant-coverage.test.ts` (the AC3 lockstep) is exactly the right shape: every entry in the `INVARIANTS` const must have a `tests/invariants/<name>.test.ts` file. There is no analogous test for matrices (every bolded cell of `tool-invariant-enforcement.md` should map to a test) or gotchas (only `tests/gotchas/concurrent-harness-write.test.ts` exists; seven other gotchas have no regression test). Additionally, `wiki/invariants/INDEX_AND_LOG_ARE_DISPATCHER_OWNED.md:29` claims four test guarantees but `tests/invariants/index-and-log-are-dispatcher-owned.test.ts` covers only two of them ((1) and (3); (2) "appendLog is the only public mutator of log.md" and (4) "`vault.rebuildIndex()` regenerates correctly" are absent). A v1 contributor retiring a gotcha, expanding a matrix, or refactoring `appendLog` would not learn from a failing test that they've broken the substrate's claims.
- **Evidence:** `/Users/mark.toda/dev/dome/tests/gotchas/` (1 file vs 8 gotcha docs); `/Users/mark.toda/dev/dome/tests/integration/invariant-coverage.test.ts` (the AC3 precedent); `/Users/mark.toda/dev/dome/tests/invariants/index-and-log-are-dispatcher-owned.test.ts` (covers 2 of 4 claimed guarantees per `docs/wiki/invariants/INDEX_AND_LOG_ARE_DISPATCHER_OWNED.md:29`).
- **Recommended fix:** (a) Add `tests/integration/gotcha-coverage.test.ts` asserting every `docs/wiki/gotchas/*.md` has a corresponding `tests/gotchas/<slug>.test.ts`. (b) Add `tests/integration/matrix-coverage.test.ts` parsing bolded cells of `tool-invariant-enforcement.md` and asserting each maps to a test name pattern. (c) Add two missing test cases for `INDEX_AND_LOG_ARE_DISPATCHER_OWNED`: assert `appendLog` is the only Tool whose effects array contains `kind: "appended-log"` across the seven Tools; assert `vault.rebuildIndex()` produces an `index.md` whose entry set equals the wiki tree.
- **Substrate artifact to add or update:** Two new lockstep tests parallel to AC3; two test additions for `INDEX_AND_LOG_ARE_DISPATCHER_OWNED`.

### 7. `Vault` has no lifecycle spec; bare-event expansion is a hidden rule; smaller concept collapses

- **Severity:** Medium
- **Category:** Hidden rule / Missing concept / Concept overlap
- **Why it matters:** Three smaller findings clustered into one:
  - **Vault lifecycle is implementation, not contract.** `Vault` exposes `drainHooks` and `dispatchEvents` but no `close()`. `domeServe` stops the watcher externally but doesn't tear down the Vault. `drainHooks` is implementation-idempotent (`hook-dispatcher.ts:152-154`) but no spec says so. A v1 mobile/desktop process holding a long-lived Vault across user turns will need this and has nowhere to align against.
  - **Bare-event-pattern expansion is hidden.** `src/hooks/yaml-loader.ts:117-126` silently rewrites declarative `event: document.written` to `document.written.*`. A contributor writing a unit test that registers `document.written` directly via the programmatic form will observe their hook never fires. The rule is in a code comment, not in `hooks.md` §"Declarative" or the event-types matrix.
  - **Small concept collapses.** `McpToolName` (PascalCase keyed object at `src/mcp/tool-names.ts`) is a "backwards-compat" alias for `MCP_TOOL_NAMES` (`src/tools/registry.ts:60-68`) with no shipped consumers to protect (`version: 0.0.1`). `WorkflowName` enum + `WORKFLOW_NAMES` const at `src/workflows/workflow-name.ts` is a parallel surface to the cleaner `TOOL_NAMES` `as const` tuple pattern. `remark` + `unified` are declared in `package.json:33-35` per `sdk-surface.md:282` but never imported across `src/` and `tests/` — `wikilinks.ts` and `frontmatter.ts` use regex + `gray-matter`.
- **Evidence:** `/Users/mark.toda/dev/dome/src/vault.ts:39-87` (no `close`); `/Users/mark.toda/dev/dome/src/hooks/yaml-loader.ts:117-126`; `/Users/mark.toda/dev/dome/src/mcp/tool-names.ts`; `/Users/mark.toda/dev/dome/src/workflows/workflow-name.ts`; `/Users/mark.toda/dev/dome/package.json:33-35` vs `grep -r "from \"remark\"" src/` (zero hits).
- **Recommended fix:** (a) Add `vault.close(): Promise<void>` to the surface; spec the lifecycle in `wiki/specs/sdk-surface.md` §"Vault lifecycle" (open / drain / close + idempotency declaration). (b) Document bare-event expansion in `wiki/specs/hooks.md` §"Declarative" and add a row to `wiki/matrices/event-types-and-payloads.md`. (c) Delete `mcp/tool-names.ts` and `McpToolName`; collapse `WorkflowName` enum to a const-tuple matching the `TOOL_NAMES` shape; drop `remark` + `unified` from `package.json` and amend `sdk-surface.md:282` (or wire them in if the regex parser is genuinely missing CommonMark edge cases).
- **Substrate artifact to add or update:** `wiki/specs/sdk-surface.md` §"Vault lifecycle" (new section). `wiki/specs/hooks.md` §"Declarative" expansion paragraph. Code deletions for `McpToolName`, `WorkflowName` enum, and `remark`/`unified` deps; spec edits to match.

### 8. `IMPLEMENTATION_HANDOFF.md` is residue; shipped `AGENTS.md` is too thin

- **Severity:** Low
- **Category:** Misleading documentation
- **Why it matters:** `docs/cohesive/IMPLEMENTATION_HANDOFF.md` reads "you are picking up where the design conversation left off" and recommends Stages 1-N for a build that has already shipped — a v1 contributor finding it via the file index will spend time on archeology. Separately, `src/cli/commands/init.ts:22-45`'s shipped `AGENTS.md` template is correct in its partition (vault-owned vs SDK-owned) but gives an agent that lands without MCP only a four-bullet fallback and no link to `wiki/invariants/`. If the harness is misconfigured, the agent will guess about `raw/` immutability, page-type-by-directory, or wikilink full-path — all of which are encoded as Tool errors but require the agent to know they exist *before* reading the error.
- **Evidence:** `/Users/mark.toda/dev/dome/docs/cohesive/IMPLEMENTATION_HANDOFF.md:1-60`; `/Users/mark.toda/dev/dome/src/cli/commands/init.ts:22-45`.
- **Recommended fix:** (a) Top-of-file banner on `IMPLEMENTATION_HANDOFF.md`: "This handoff was the v0.5 implementation kickoff and is now historical. For current onboarding see `docs/index.md` and `wiki/specs/sdk-surface.md`." Or move to `docs/cohesive/archive/`. (b) Add a 3-line "If MCP is unavailable, the canonical rules live in `docs/wiki/invariants/`" pointer to `SHIPPED_AGENTS_MD`. Or: ship a `wiki/specs/cold-start.md` and link from `AGENTS.md`.
- **Substrate artifact to add or update:** Archive marker on `IMPLEMENTATION_HANDOFF.md`. Edit to `SHIPPED_AGENTS_MD` constant in `init.ts` + optional new `wiki/specs/cold-start.md`.

## Phased roadmap

**Phase A — substrate repairs (1-2 days).** Land Findings 3, 4, 7, 8 as a single delta-ledger pass via `cohesive:rewrite-specs`. These are pure spec/test edits (plus a small `dome doctor` enhancement for substrate-count check and inbox-age check); no architectural restructuring. Outcome: the spec stops drifting; the structural defenses around `INBOX_IS_EPHEMERAL` and substrate counts are real; small concept collapses remove residue.

**Phase B — name the seams (3-5 days).** Land Findings 1, 2, 6 as a brainstorm-design + rewrite-specs + implement-cohesively chain. This is the v1-extensibility seam work: declare `CORE_HAS_NO_LLM_OR_MCP_DEPENDENCY` as an axiom, name the `ConsumerSurface` aggregation pattern, build the lockstep tests for matrices and gotchas, and ship the `@dome/sdk/workflows` and `@dome/sdk/mcp` entrypoint split. Outcome: a v1 contributor adding a mobile/desktop/voice/web shell has a structural template to follow and a bundle-deps test that fails when core leaks.

**Phase C — pin the type-system boundaries (deferred until next AI SDK or Zod major).** Finding 5 is a load-bearing scar that should be left in place until the next ecosystem bump forces a migration; the gotcha doc records the trigger.
