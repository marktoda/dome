## Architectural reflection

The pass-3 → pass-2-repair rewrite is now substrate-coherent. The substantive structural moves — `AbstractSurface` + per-protocol renderers, `wrapMutatingInvoke` single-source helper, the `vaultRef`-setter composable construction, one-`PromptLoader`-per-Vault, off-matrix delegating-stub lockstep, three "Adding a new X" recipes paralleling "Adding an 8th Tool" — are mutually consistent and trace cleanly through the spec set. A future contributor reading from `docs/index.md` can answer every common change-shape question with bounded context, anchored to a named structural fence (AC3 lockstep, bundle-deps test, matrix-coverage test, gotcha-coverage test, or one of the new v0.5.1 lockstep extensions the implementation work names).

- **Easier downstream:** Adding any of the seven common change shapes (Tool, CLI command, hook, invariant, gotcha, consumer surface, page type) is now a surgical N-file edit per a named recipe. A new consumer surface (HTTP, voice, future shells) ships as a ~50-line `renderXxx(surface)` projection over `AbstractSurface`. The `enforced_at_status: deferred` semantic lets v0.5 substrate honestly carry "mitigation is named but not yet shipped" without weakening the lockstep on the merged-shipped paths.
- **Harder downstream:** The multi-vault session shape is unnamed (deferred to a separate `brainstorm-design`); v1 mobile/desktop work that touches multi-Vault contexts will hit this question first. `asyncConcurrency: 1` is the deliberate v0.5 default; long-running shells with high-concurrency intake hooks may need to bump it, and the spec names the knob but doesn't carry guidance on when. The linter-self-exclusion pattern (allow-list lives in the linter spec; the lockstep skips that doc) requires three coupled-site updates when adding a new excluded path — substrate carries the coupling explicitly on `linters/no-retired-symbol-names.md:44`, but coupled-update sites are still a maintenance cost.
- **Load-bearing on memory:** The `vaultRef` setter pattern is convention-with-comment until `tests/integration/vault-bootstrap-order.test.ts` ships in the implementation pass — until then, a future contributor restructuring `openVault` for clarity could regress the bootstrap-ordering safety without any test failure. The "linter spec is canonical home of the allow-list" convention is reviewer-memory until the v0.5.1 lockstep arms. The `enforced_at_status: deferred` semantic itself is convention-with-grep until `tests/integration/gotcha-coverage.test.ts` is extended to read the field (named in F5 implementation work).

The deferrals above are deliberate v0.5 scope; none of them are structural problems with the chosen direction. The substrate is ready to lock and the implementation pass that follows is what makes its claims true in code.

## Important issues

### I1. Repo-root `README.md` absent despite pass-3 F8 naming three missing files

- **Severity:** Medium
- **Category:** Spec drift
- **Why it matters:** The pass-3 architecture review names "No `/Users/mark.toda/dev/dome/AGENTS.md`, `CLAUDE.md`, or `README.md`" as the F8 cluster item. The delta ledger §"Repo-root files added" claims to close that item but ships only AGENTS.md + CLAUDE.md. A `README.md` is the npm-package-page-facing artifact (visible on npmjs.com when `@dome/sdk` publishes); AGENTS.md serves agentic-contributor orientation, not the public-package surface. The substrate's claim about itself ("AGENTS.md closes the F8 cluster item") diverges from what the F8 cluster actually named.
- **Evidence:** Pass-3 review names three files; ledger §"Repo-root files added" ships only two; the §"Repair pass 2" prose cites the original F8 text. A first-time reader landing from GitHub or npm sees no README.
- **Recommended fix:** Either (a) add a minimal `/README.md` (npm-package-facing: one paragraph on what `@dome/sdk` is + pointer at `AGENTS.md` for contributors + pointer at `docs/index.md` for substrate readers), or (b) narrow the ledger's F8 closure claim to "AGENTS.md + CLAUDE.md ship; README.md deferred to the npm-publish PR" and add a remaining-ambiguity entry.
- **Substrate artifact to add or update:** Either `/README.md` (new file) or `delta-ledgers/2026-05-26-dome-v0.5-to-v1-tightening.md` §"Remaining ambiguity".

### I2. Linter spec asserts allow-list-as-typed-const before the const ships

- **Severity:** Low
- **Category:** Vague language
- **Why it matters:** `docs/wiki/linters/no-retired-symbol-names.md:18` declares "**The retired-names allow-list** is a typed const exported from `src/types.ts` alongside `INVARIANTS`. As of v0.5.1:" The const does not yet exist in v0.5; per the ledger's implementation work (`f1-retired-names-const`) it ships with the v0.5.1 lockstep. The doc's present-tense framing reads as if the const exists today, which a reader running `grep "RETIRED_SYMBOLS" src/types.ts` discovers is false.
- **Evidence:** `linters/no-retired-symbol-names.md:18` present-tense framing. `delta-ledgers/2026-05-26-dome-v0.5-to-v1-tightening.md` §F1 names `f1-retired-names-const` as deferred implementation work.
- **Recommended fix:** Reword line 18 to "**The retired-names allow-list** will be a typed const exported from `src/types.ts` alongside `INVARIANTS` when the v0.5.1 lockstep ships. Until then, the table below is the canonical home; the lockstep skips this doc per the exclusion set."
- **Substrate artifact to add or update:** `docs/wiki/linters/no-retired-symbol-names.md` §"The retired-names allow-list".

### I3. Linter doc carries `tier: shipped-default` while body declares v0.5.1 ship

- **Severity:** Low
- **Category:** Domain model
- **Why it matters:** `linters/no-retired-symbol-names.md` frontmatter declares `tier: shipped-default` and `target_version: v0.5.1`. A reader scanning frontmatter alone concludes the linter ships and is on-by-default in v0.5; reading the §"Status" body line clarifies "Lockstep test ships in v0.5.1." The tier-field semantic (per `sdk-surface.md` §"Tiered feature model") describes *active* enablement, not destination shipping. The sibling `wrap-mutating-invoke-consumption` linter carries the same ambiguity. `page-schema.md` §"Extension types" doesn't enumerate the `linter` page type, so the `tier:` field on linter docs is a v0.5 convention without a documented schema.
- **Evidence:** `linters/no-retired-symbol-names.md:6` `tier: shipped-default`; line 12 `**Status:** Lockstep test ships in v0.5.1`. `page-schema.md` §"Extension types" doesn't list `linter`.
- **Recommended fix:** Either (a) extend `page-schema.md` §"Extension types" to declare a `linter` extension type with frontmatter-schema fields (`tier`, `target_version`, `status: shipped | planned | deferred`), or (b) drop `tier:` from linter frontmatter until ship and let the §"Status" body be canonical. Option (a) closes the schema-coverage gap for linters categorically.
- **Substrate artifact to add or update:** `docs/wiki/specs/page-schema.md` §"Extension types" (preferred) OR `docs/wiki/linters/no-retired-symbol-names.md` frontmatter.
