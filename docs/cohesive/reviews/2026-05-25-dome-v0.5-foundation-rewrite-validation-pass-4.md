**Verdict:** Issues Found

## Executive judgment

The substrate is broadly implementable. Four repair passes have closed every previously-flagged Blocker and most High findings, and what's left is genuinely small — but one survives that a future contributor will trip over: the rewrite has not unified what happens when the `auto-update-index` hook writes `index.md`. `sdk-surface.md` and `cli.md` claim `writeDocument('index.md', ...)` is rejected unconditionally and that index writes use a "dispatcher's privileged path"; `hooks.md` and `vault-layout.md` claim the hook writes index.md via `writeDocument`. Both cannot be true at the same time, and the seam is load-bearing. Three smaller divergences ride on top: a matrix preamble that names a row that doesn't exist, a column header that miscategorizes itself, and a workflow tool-binding gap on `deleteDocument`.

## Blocking issues

### B1. `index.md` write path is contradicted across four docs
- **Severity:** High
- **Category:** Spec drift
- **Why it matters:** The seam between "dispatcher-owned files" and "Tool-callable files" is what the SDK implementor must build first. An implementor reading sdk-surface builds a Tool that rejects `writeDocument('index.md', ...)` unconditionally; an implementor reading hooks.md and vault-layout.md builds `auto-update-index` to call `writeDocument('index.md', ...)`. The first build crashes the second on first invocation. The cli.md `--rebuild-index` flag explicitly cites the dispatcher's privileged path — but the path is undefined.
- **Evidence:**
  - `sdk-surface.md:182` — "writes the affected index entries via the dispatcher's privileged path. `index.md` is dispatcher-owned, not Tool-callable: `writeDocument('index.md', ...)` and `writeDocument('log.md', ...)` are rejected unconditionally regardless of vault config."
  - `cli.md:137` — "The rebuild bypasses the normal Tool-rejection of `writeDocument('index.md', ...)` because dispatcher-owned files are writable only through the dispatcher's privileged path."
  - `hooks.md:72` — "The handler reads the modified Document, computes the index entry, and writes the updated `index.md` via `writeDocument(index.md, ...)`."
  - `vault-layout.md:114` — "`index.md` | Dome (via `writeDocument` invoked by the `auto-update-index` hook) | Mutated only through the auto-update-index shipped-default hook."
- **Recommended fix:** Pick one of two designs and propagate.
  - **Option A:** Introduce a named privileged-write affordance the dispatcher uses for `index.md` only — e.g., `dispatcher.writeIndex(entry)` is an internal mutation primitive accessible to shipped-default hooks; `writeDocument('index.md', ...)` rejects unconditionally; the seam is named and testable. Promote `INDEX_AND_LOG_ARE_DISPATCHER_OWNED` to a named invariant.
  - **Option B:** `writeDocument` accepts `index.md` from hook contexts only via a `HookContext`-only flag on the input shape; the seam is the call-site capability, not the path.
  - Whichever, update all four sites in one sweep + add a `dome doctor` check.
- **Substrate artifact:** sdk-surface (Tool catalog + Tool signatures); hooks.md; vault-layout.md; cli.md; new named invariant (Option A) or new field on `writeDocument` signature (Option B).

## Important issues

### I1. Matrix preamble names a row that doesn't exist; column header miscategorizes its own scope
- **Severity:** Medium
- **Category:** Spec drift / Domain model
- **Why it matters:** The matrix is the canonical surface for "which Tool enforces which invariant." A reader who follows the preamble to `(Workflow handlers)` finds no such row; the actual row is `(Hook handlers)`. The "Tool-boundary-enforced invariants" header is also misleading since `HOOKS_CANNOT_BYPASS_TOOLS` is type-system-enforced on handler signatures.
- **Evidence:**
  - `tool-invariant-enforcement.md:10` — preamble: "all Tool-boundary-enforced invariants"
  - `tool-invariant-enforcement.md:15` — "The matrix's `(Workflow handlers)` row notes this" — the row is actually named `(Hook handlers)` and the INBOX_IS_EPHEMERAL carve-out actually lives in a separate §"workflow-enforced (off-matrix)" subsection.
  - `tool-invariant-enforcement.md:36` — the actual row name.
- **Recommended fix:** Two coupled edits. (a) Replace preamble "all Tool-boundary-enforced invariants" with phrasing that covers the type-system-enforced HOOKS_CANNOT_BYPASS_TOOLS case. (b) Replace the "`(Workflow handlers)`" reference with "see §`INBOX_IS_EPHEMERAL` — workflow-enforced (off-matrix) below".
- **Substrate artifact:** `tool-invariant-enforcement.md`.

### I2. `migrate` workflow tool-binding omits `deleteDocument` despite sdk-surface saying migrate may delete
- **Severity:** Medium
- **Category:** Spec drift
- **Why it matters:** Workflow bindings are the structural enforcement of "what the agent can do." `migrate` is expected to delete superseded files per `sdk-surface.md:88`, but `deleteDocument` is not bound in either `prompts-and-workflows.md` or `intent-prompt-tools.md`.
- **Evidence:**
  - `sdk-surface.md:88` — "migrate may delete superseded files."
  - `prompts-and-workflows.md:62` — migrate binds readDocument, writeDocument, moveDocument, appendLog, searchIndex, wikilinkResolve — `deleteDocument` absent.
  - `intent-prompt-tools.md:24` — same.
- **Recommended fix:** Add `deleteDocument` to migrate's bound tools in both files. (User decision possible: alternatively scope migrate to propose-only and have the user manually `git rm`; but `migrate --apply` already executes in cli.md, so adding deleteDocument is the consistent path.)
- **Substrate artifact:** `prompts-and-workflows.md`, `intent-prompt-tools.md`.

### I3. `appendLog × EVERY_WRITE_IS_LOGGED` matrix cell is semantically opaque
- **Severity:** Low
- **Category:** Domain model / Vague language
- **Why it matters:** "*(the enforcement target)*" inverts cause and effect. `appendLog` doesn't enforce the invariant; it's the primitive other Tools call to satisfy it.
- **Evidence:** `tool-invariant-enforcement.md:31`.
- **Recommended fix:** Replace with "*(the primitive; other mutating Tools call this to satisfy the invariant)*" or leave the cell blank.
- **Substrate artifact:** `tool-invariant-enforcement.md` cell.

### I4. Normative "should" in `prompts-and-workflows.md`
- **Severity:** Low
- **Category:** Vague language
- **Why it matters:** Either every workflow prompt is required to include `system-base.md` (requirement) or it's a recommendation (convention). A plugin author can't tell whether a workflow without the include is broken or merely unconventional.
- **Evidence:** `prompts-and-workflows.md:85` — "Every workflow prompt should start with `{{include: system-base.md}}`."
- **Recommended fix:** Tighten to "Every workflow prompt starts with `{{include: system-base.md}}`. The SDK warns at workflow-load time if the include is absent."
- **Substrate artifact:** `prompts-and-workflows.md`.
