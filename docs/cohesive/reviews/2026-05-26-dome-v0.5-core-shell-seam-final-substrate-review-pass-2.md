## Findings

### N1 — `transitive-llm-dependency.md:15` is internally circular after the I2 rename

**Severity:** Low
**Category:** Spec drift (introduced by repair commit `f134b73`)

**Why it matters:** The bulk-rename pass replaced `@anthropic-ai/sdk` → `@ai-sdk/anthropic` everywhere. The pre-rename sentence in the gotcha doc described the real transitive chain: `ai → @ai-sdk/anthropic → @anthropic-ai/sdk` (the Vercel adapter wrapping the first-party SDK). After the bulk rename the sentence reads `ai → @ai-sdk/anthropic → @ai-sdk/anthropic` — the trailing link is now identical to its predecessor, so the chain explanation is broken. A reader trying to understand "why does importing one package pull all three of these" finds a tautology. The gotcha doc is the canonical scar story for the axiom; readers will land here from the invariant page's `**Related**` block.

**Evidence:**
- `docs/wiki/gotchas/transitive-llm-dependency.md:15` — "...reached `tools/registry.ts`, which imports `ai`, which transitively imports `@ai-sdk/anthropic`, which imports `@ai-sdk/anthropic`. The chain ran for every consumer..."
- `package.json:34` — `@ai-sdk/anthropic` is the Vercel adapter, not Anthropic's first-party SDK.

**Recommended fix:** Rewrite the chain explanation to name what `@ai-sdk/anthropic` actually wraps (e.g., "...imports `ai`, which is the Vercel AI SDK; `@ai-sdk/anthropic` is the provider adapter that bridges `ai` to Anthropic's HTTP API."). Or simplify: "...the chain pulls `ai` + `@ai-sdk/anthropic` + their transitives." The point of the symptom paragraph is the bloat, not the exact chain.

**Substrate artifact to add or update:** `docs/wiki/gotchas/transitive-llm-dependency.md` §"Severity" paragraph.

### N2 — `dispatchEvents after close()` test asserts no-throw, not no-op

**Severity:** Low
**Category:** Test guarantee gap (introduced by repair commit `1019359`)

**Why it matters:** The I4 repair's load-bearing claim is that `close()` flips a flag and `dispatchEvents` consults it. The spec at `sdk-surface.md:64` says: "calls become silent no-ops — the dispatcher accepts no new event work after close." The test at `tests/invariants/vault-lifecycle.test.ts:55-89` invokes `vault.dispatchEvents([...])` after `close()` and asserts only `expect(true).toBe(true)` — i.e., no-throw. A future regression that removes the `if (closed) return;` guard at `src/vault.ts:228` would still pass this test (dispatchEvents would re-queue the event into the now-released p-queue, which probably still functions enough not to throw, and the file-on-disk side effects would actually fire). The test inline comment acknowledges this: "We assert no-throw here; the structural pin is that the closed flag stops the dispatcher from queueing work into resources that may have been released in a future minor." The structural pin the comment describes is the code; the test does not pin it.

**Evidence:**
- `tests/invariants/vault-lifecycle.test.ts:79-85` — dispatches a fake `document.written.wiki.entity` event after close, then drainHooks, then expects `true === true`. No assertion that auto-update-index didn't fire; no observable used to detect the no-op.
- `docs/wiki/specs/sdk-surface.md:64` — "calls become silent no-ops — the dispatcher accepts no new event work after close."
- `src/vault.ts:228` — `if (closed) return;` is the load-bearing line.

**Recommended fix:** Before close, dispatch a write through `vault.tools.writeDocument` and `await drainHooks()` to capture the baseline `index.md` body. After close, dispatch an event referencing a new wiki path that — if auto-update-index fired — would change `index.md`. Re-read `index.md` and assert it equals the baseline. This pins the spec's "silent no-op" claim observably. Cheap (~15 lines).

**Substrate artifact to add or update:** Extend `tests/invariants/vault-lifecycle.test.ts` test 3.

## New substrate seam introduced (worth naming)

The B1 repair introduces an unnamed but load-bearing substrate property: **"the hook-dispatch wrap is intrinsic to the Vault-bound Tool registry, not a per-call-site decorator."** This is structurally what makes the MCP/AI-SDK/SDK-direct symmetry work, and `tests/integration/mcp-hook-dispatch.test.ts` is its structural fence. The spec subsection §"Hook dispatch is intrinsic" at `sdk-surface.md:44-46` names it in prose but does not promote it to a named invariant.

**Recommendation (non-blocking):** Consider whether `HOOK_DISPATCH_IS_VAULT_BOUND` deserves invariant-tier promotion in a follow-up. The pass-1 review suggested this; the repair encoded the behavior but didn't elevate the rule to named-invariant status. The matrix-coverage test would gain a new cell; bundle-deps would gain a sibling test. Defer to taste — the spec subsection plus the integration test is sufficient enforcement for v0.5.
