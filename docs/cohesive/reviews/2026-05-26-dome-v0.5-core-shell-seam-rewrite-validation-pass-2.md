## Blocking issues

### B1. Preamble + body undercount the I3-closure rewrite footprint

- **Severity:** High
- **Category:** Spec drift
- **Why it matters:** The closure of pass-1 I3 modified `page-schema.md` and added frontmatter to 8 pre-existing gotcha docs. Neither the preamble's `5 substrate files rewritten (3 specs, 1 matrix, 1 catalog)` nor the body's `### Substrate rewritten (5 files)` enumeration records these edits. The ledger's audit-trail purpose is to let a fresh reader see the rewrite as a delta; right now the reader sees ~14 modified files as 5.
- **Evidence:** Preamble Files line; body §"Substrate rewritten (5 files)"; the §"Repair pass 2" I3 row names page-schema.md edits and "All 11 gotcha docs gain the field" but those aren't reflected in the rewritten-files counts.
- **Recommended fix:** Update the preamble line to `5 substrate files rewritten (3 specs, 1 matrix, 1 catalog) plus page-schema.md (spec) and 8 pre-existing gotchas (frontmatter-only edits)` — or restructure as "normative rewrites: 5; frontmatter-only edits: 9." Mirror in the body §"Substrate rewritten" with a sub-section for frontmatter-only edits.
- **Substrate artifact to add or update:** Ledger preamble + §"Substrate rewritten" body section.

## Important issues

### I1. Future-HTTP Tools cell mis-cites `projectAiSdk`

- **Severity:** Medium
- **Category:** Domain model
- **Why it matters:** The matrix's Future HTTP row Tools cell reads `core + workflows` and justifies it as "via `projectAiSdk(vault)`." But `projectAiSdk` is the AI-SDK ToolSet projection consumed by `runWorkflow`/`generateText` — not a generic tool-projection for a web-app HTTP backend. A reader using this matrix to choose entrypoints for a future HTTP shell would conclude the HTTP backend needs `@anthropic-ai/sdk` + `ai`, which is conceptually wrong: an HTTP server exposing Tools to a web frontend would parallel MCP (raw parsers per Tool), not AI-SDK.
- **Evidence:** `consumer-surface.md` Future HTTP row Tools cell.
- **Recommended fix:** Change the Future HTTP Tools cell to `core` (with the note "a future `projectHttp(vault)` would extend `mcp`-shape parsers to HTTP envelopes"). Move the `core + workflows` only into the Workflow-runner column where it already lives.
- **Substrate artifact to add or update:** `consumer-surface.md` Future HTTP row.

### I2. "## Structural mitigation section" claim doesn't match gotcha-doc shape

- **Severity:** Low
- **Category:** Spec drift
- **Why it matters:** The pass-2 I3 closure row says off-matrix gotchas carry "structural-mitigation rationale named in each gotcha's `## Structural mitigation` section." None of the 11 gotcha docs have a `## Structural mitigation` H2 header. Several use a `**Structural mitigation:**` bold-lead in prose; several have no labeled mitigation block at all.
- **Evidence:** Ledger §"Repair pass 2" I3 row prose vs gotcha file header scan.
- **Recommended fix:** Either rephrase the I3 closure to "structural-mitigation rationale named inline in each gotcha doc" (matching observed shape), or actually add `## Structural mitigation` H2 sections to all 11 gotchas. Rephrase is the cheaper choice.
- **Substrate artifact to add or update:** Ledger §"Repair pass 2" I3 row prose.
