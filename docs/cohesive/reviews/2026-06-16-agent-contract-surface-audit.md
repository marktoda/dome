# Contract-surface audit — is Dome a good agent backend?

**Date:** 2026-06-16
**Scope:** the agent-facing contract — `src/surface/` collectors, the `dome mcp` tools, the `src/http/` API, and the per-vault `AGENTS.md` orientation. Lens: the [[wiki/concepts/client-model]] thesis that *the product is the contract clients consume*, and the sharpened question — can a voice-first phone agent do the read → search → synthesize → write → capture loop, and is `AGENTS.md` the cross-client quality multiplier?
**Method:** three parallel read-only audits of the live code + a real vault (`~/vaults/work`).

## TL;DR

**Verdict:** Strong at *read*, deliberately incapable of *authoring* for any client without a git checkout. Recall beats grep; writing a page needs a filesystem. That asymmetry is the whole story.

**Thesis:** Dome's recall surface genuinely earns "the product is the contract" — `query`/`export-context` return ranked matches **plus** facts, claims, diagnostics, open questions, and `sourceRefs`, which is materially better than raw grep. But the *only* mutations any client gets across CLI/MCP/HTTP are `capture` (raw note → `inbox/raw/`) and `resolve` (answer an existing question). Page authoring happens only as a `git commit` adopted by the daemon, so it is bound to a co-located checkout — i.e. desktop Claude Code. And the orientation that should make *any* agent write good pages (`AGENTS.md`) teaches workflow well but omits the markdown conventions (page types, claims, task anchors, generated blocks, source-backing), so it is ~a 50–60% multiplier.

## Findings

### 1. Recall is the contract's strength — keep this posture

`query` (`dome.search.query/v1`) and `export-context` (`dome.search.export-context/v1`)
return what filesystem access can't: FTS ranking, related `facts`, `claims`,
content `diagnostics`, open `questions`, and `sourceRefs` into specific adopted
commits; `export-context` is a curated, source-backed packet an agent can reason
over directly. This is a real reason a client wants Dome underneath rather than
just reading files. No change needed; this is the model working.

### 2. The structural finding — read + capture, no authoring

The agent-facing contract exposes exactly two mutations, identical across CLI,
the 8 `dome mcp` tools, and HTTP:

- `capture` → append a raw note to `inbox/raw/` (one file, one commit)
- `resolve` → answer a question the engine already raised

There is **no** `createPage` / `editDocument` / `propose-patch`. `src/vault.ts`
states it: *"there is no write method, no Proposal constructor, no
submitProposal."* Enforced by [[wiki/invariants/PROPOSALS_ARE_THE_ONLY_WRITE_PATH]].

**Consequence:** authoring is bound to a co-located git checkout. Claude Code on
the desktop has one and writes freely; a phone app / remote MCP client / Claude
iOS app has none and can only read + capture. The client-model's
read/synthesize/**write**/capture loop is the desktop story only. (Captured in
[[wiki/concepts/client-model]] §"The authoring boundary".)

### 3. No agent runs inside Dome

HTTP and MCP are pure data APIs — no LLM/agent is hosted anywhere (`src/http/`
imports no model SDK; there is no `/ask`/`/chat`/synthesis route). "Ask my brain
and get an answer" therefore needs an agent running *somewhere*: desktop Claude
Code today; for the phone, the intended answer is the Claude iOS app over remote
MCP — **unbuilt** — and even then it cannot author. So both phone "ask" and
phone authoring converge on the same deferred dependency: **a reachable hosted
agent on an always-on host.**

### 4. Mapping to the phone app the owner actually wants

The wanted app (cockpit + recents + voice capture + ask) is lighter than remote
authoring, so most of it is reachable:

| App element | Contract today | Gap |
|---|---|---|
| Voice capture → `POST /capture` | ✅ works (idempotent via `captureId`) | none |
| Cockpit brief → `GET /today` / `/tasks` | ✅ tasks/loops/questions/calendar, 15s poll, tap-to-resolve | none |
| Recents history | ⚠️ `activity` collector exists (git × run-ledger join, powers `dome log`) but is **not exposed over MCP/HTTP**, and is an *activity* log, not a "viewed/touched" feed | add a route + a viewed-history notion |
| "Ask" with a synthesized answer | ❌ no hosted agent; HTTP/MCP are data-only | reachable hosted agent (always-on host) |
| Multi-device | ❌ single shared bearer token; per-device issuance/rotation never delivered | per-device tokens |

### 5. `AGENTS.md` is a workflow primer, not an authoring teacher (~50–60% multiplier)

Generated by `renderAgentsMd(...)` (`src/cli/commands/init-templates.ts`),
first-write-only with a `--refresh-instructions` path that preserves user prose
via `<!-- BEGIN/END user-prose -->` delimiters (pinned by
[[wiki/invariants/AGENTS_MD_IS_ORIENTATION_SURFACE]]). Harness-neutral (good);
`CLAUDE.md` is just the Claude-Code load shim.

**Teaches well:** the daily commit-boundary loop, recall-first discipline
(query/export-context before grep), the `resolve` flow (automation_policy), the
vault layout, the load-bearing rules.

**Omits the conventions an agent needs to *author* correctly** — and that
knowledge lives in `docs/wiki/specs/` the agent will never read at session
start:

- **Page-type schema** — entity/concept/source/synthesis + required frontmatter,
  `description:`, `sources:`. Without it, untyped pages trip adoption lint.
- **Claims grammar** — `**Key:** value *(as of …)* ^c…`; agents must not hand-edit `^c` anchors.
- **Task anchors** — `^t…` move-stable identity; agents treat them as typos and corrupt task reconciliation.
- **Generated-block awareness** — `<!-- dome.* -->` markers; edits inside vanish on next sync.
- **Source-backing discipline** — the thing that makes the vault trustworthy.

Since the entire value is "the agent writes good pages," this is the cheapest,
highest-leverage fix in the audit — pure template content, ships to every vault
via `--refresh-instructions`, lifts every client at once.

## Prioritized punch list

1. **Fill `AGENTS.md`'s authoring conventions** (page types, claims, task
   anchors, generated blocks, source-backing). Content-only; multiplies every
   client. **Do first.**
2. **Decide the hosted-agent + remote-reach question** (always-on host running a
   Claude-Code-class agent + daemon + checkout, reachable from the phone). This
   single decision unblocks both phone "ask" and phone authoring; everything
   else mobile waits on it. Reopens the laptop-first deferral in [[v1]] WS3.
3. **Expose a recents route + per-device tokens.** Foundations exist
   (`src/surface/activity.ts`); needed for the app specifically.
4. **Name the authoring fork** (done in [[wiki/concepts/client-model]]):
   authoring is co-located-only by design; phone-side authoring is a deliberate
   contract change (`propose-patch`) or a hosted co-located agent — not a small
   route.
