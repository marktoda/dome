## Thesis

The pass-2 architectural moves landed structurally: `AbstractSurface` ships in core with `renderMcp` as the first protocol projection, `wrapMutatingInvoke` is the single-source hook-dispatch helper, `HOOK_DISPATCH_IS_VAULT_BOUND` joined the named axioms, and the compiler-reframe (`dome serve` as compiler daemon, AGENTS.md as orientation surface) gives v1 its consumer chassis. The 16-invariant + AC3-lockstep substrate is genuinely working; the user has been daily-driving v0.5 inside Claude Code with the substrate doing its job.

What four parallel reviewers converged on is a single class of gap repeated across four surfaces: **labels lag structure on the new seams**. Three normative docs still describe the pre-`AbstractSurface` shape (`ConsumerSurface`, `buildConsumerSurface`). `domeStats` ships and has a spec but isn't re-exported from `src/cli/index.ts`. The AGENTS.md user-prose delimiters are load-bearing string literals duplicated across 3 files with no single source. Eleven of twelve gotchas declare `coverage: off-matrix` with no AC3-style fence checking the elsewhere. The most-trafficked v1 change shape (adding a CLI command) is also the least-documented and least-supported: a 482-line `doctor.ts` mixing ten concerns, a drifted `DoctorFlag` enum, no "Adding a new command" recipe paralleling the spec's surgical "two files to add a Tool" guidance.

The v1 lift is small in lines and large in leverage. A single `rewrite-specs` pass aligns the seven specs/invariants/gotchas still describing the pre-pass-2 shape; a lockstep extension closes the off-matrix loophole on consumer-surface and intent-prompt-tools matrices; a CLI-recipe + `src/cli/index.ts` symmetry pass closes the change-shape gap. After that, the next consumer surface inherits a substrate that proves its own promises instead of carrying them in folklore ‚Äî exactly what pass-1 said it wasn't yet doing, and exactly what the v1 multi-surface bet depends on.

## Highest-leverage findings

### F1. Seven live normative documents still name retired `ConsumerSurface` / `buildConsumerSurface` after the pass-2 ‚Üí `AbstractSurface` migration

**Severity:** High
**Category:** Spec drift / Naming-vs-implementation drift

**Why it matters:** A v1 contributor reading `wiki/specs/harnesses.md` is told to "construct against `ConsumerSurface`" ‚Äî a type that no longer exists. The live shape is `buildAbstractSurface(vault) ‚Üí AbstractSurface` + `renderMcp(surface) ‚Üí McpSurface`. This is the exact failure mode `docs/wiki/gotchas/substrate-count-drift.md` was supposed to prevent ‚Äî it caught counts, but not type-name renames.

Pass-2 introduced the same shape of finding at the prior seam (`ConsumerSurface` named protocol-agnostic, type signature was MCP-bound); option B (split + rename) shipped *in code*, but the spec sweep didn't propagate through every load-bearing surface.

### F3. `vault.ts` openVault is a 348-line god-factory with three hidden ordering rules

**Severity:** High
**Category:** Locality / Hidden invariant / Agent-readiness

**Why it matters:** `src/vault.ts:129-348` inlines six concerns (config load + page-types load + privileged-writer construction + hook-registry construction with built-in registrations + hook-dispatcher wiring with cycle listener + `dispatchEvents` closure + `bindTools` invocation + `rebuildIndex` closure + `drainHooks` closure + `close` closure + declarative-hook YAML load) plus three load-bearing ordering rules that live only in inline comments:

1. **Temporal dead zone scar.** `bindTools(partial, ...)` at line 282 receives a `partial` Vault where `dispatchEvents` closes lazily over `tools` declared below it. The 13-line comment at lines 268-281 documents the workaround; the gotcha doc never landed (pass-2 F8 carry-over).
2. **`loadDeclarativeHooks` MUST run LAST** (line 336) because handlers close over `vault`. Inline comment at lines 332-335 only.
3. **`hookDispatcher.onCycleDetected` MUST be wired AFTER `privilegedWriter` exists but BEFORE any event can fire** (lines 238-240). No test fails if reordered.

A v1 surface that wants a Vault-without-watcher, a Vault-with-different-built-in-hook-set, or an in-memory test Vault must either accept all of openVault's choices or fork the function. The flag `connectStdio` (`src/cli/commands/serve.ts:30-34`) and `skipCommit` (`src/workflows/agent-loop.ts:33`) already show the "let-callers-opt-out" tax appearing at adjacent layers. An agent compacting `openVault` to a more natural construct order silently breaks cycle logging.

**Evidence:** `/Users/mark.toda/dev/dome/src/vault.ts:129-348`; `:268-281` (TDZ comment); `:332-335` (loadDeclarativeHooks LAST comment); `:238-240` (cycle-listener order). No gotcha doc at `docs/wiki/gotchas/vault-bootstrap-temporal-dead-zone.md`.

**Recommended fix:** Two-part fix, both partial fixes are useful on their own.

- **Part A ‚Äî structural decomposition.** Extract three named helpers without changing the public `openVault` signature: `loadVaultConfig(root) ‚Üí {config, pageTypes}`, `buildBuiltinHookRegistry(root, config) ‚Üí HookRegistry`, `wireDispatcher(registry, writer, opts) ‚Üí {dispatcher, dispatchEvents, close, drainHooks}`. `openVault` becomes a ~30-line composition. A v1 surface that wants a custom subset assembles from these.
- **Part B ‚Äî kill the orderings.** Accept a `vaultRef: { current: Vault | null }` setter, build the dispatcher / registry / cycle-listener against that, then assign `vaultRef.current = vault` at the end. The TDZ disappears, "loadDeclarativeHooks LAST" becomes a method on the constructed Vault not a positional rule, and the cycle-listener wiring can be in any order. Add a regression test that scrambles bootstrap section order and asserts hook-cycle logging still works.

If Part B isn't taken, ship the gotcha doc pass-2 recommended (`docs/wiki/gotchas/vault-bootstrap-temporal-dead-zone.md`) as the honest minimum.

**Substrate artifact to add or update:** Code refactor in `src/vault.ts`. If Part B taken, regression test extension; if not, new gotcha doc. Optionally `sdk-surface.md ¬ß"Vault"` picks up a "Composable construction" subsection if a v1 surface needs it.

### F4. `PromptLoader` triple-walks the prompts filesystem ‚Äî `WorkflowRegistry` ignores the loader its caller already built

**Severity:** High
**Category:** Duplication signaling missing concept / Pattern propagation

**Why it matters:** `WorkflowRegistry` constructor (`src/prompts/registry.ts:17`) builds its own `PromptLoader`. `buildAbstractSurface` (`src/abstract-surface.ts:94-101`) builds *both* a `PromptLoader` and a `WorkflowRegistry` ‚Äî the latter silently throws away the loader the caller already built and walks `.dome/prompts/` + `BUILTIN_DIR` a second time. `agent-loop.runWorkflow` (`src/workflows/agent-loop.ts:69`) builds a *third* `WorkflowRegistry`+loader per workflow invocation. A v1 web shell doing `buildAbstractSurface(vault)` + `runWorkflow(vault, ...)` does at minimum three filesystem walks per request. Pass-2 F8 named this; the fix didn't ship.

The missing concept is **PromptCatalog**: one walker per Vault, shared by every consumer. Same shape as the pre-Phase-B `vault.aiTools` problem that the AbstractSurface split closed at the macro level ‚Äî the micro-level repeat is still live.

**Evidence:** `/Users/mark.toda/dev/dome/src/prompts/registry.ts:13-18`; `/Users/mark.toda/dev/dome/src/abstract-surface.ts:94-101`; `/Users/mark.toda/dev/dome/src/workflows/agent-loop.ts:69`. Also `src/workflows/project-ai-sdk.ts:21` reconstructs `PrivilegedWriter` per call ‚Äî the parallel "rebuild what the Vault already has" pattern; dead-construction, since the actual mutating path runs through `wrapMutatingInvoke` ‚Üí `vault.dispatchEvents` ‚Üí built-in hooks ‚Üí `ctxFactory.privilegedWriter`.

**Recommended fix:** `WorkflowRegistry` constructor accepts an optional `PromptLoader` parameter (existing or constructs one if absent ‚Äî backward-compatible). `abstract-surface.ts:96` passes its already-built `loader`. `agent-loop.runWorkflow` accepts an optional `registry: WorkflowRegistry`; the long-running `domeServe` path builds one and reuses it across requests. Optional v1 follow-up: attach the loader to `Vault.prompts` lazily and let it be the canonical one. For `projectAiSdk`'s writer reconstruction: either (a) `Vault` carries a module-private `_writer` field that `projectAiSdk(vault)` reads, or (b) `bindAiSdkTools(vault)` (no writer arg) ‚Äî either way the dead reconstruction goes.

**Substrate artifact to add or update:** Code refactor (`src/prompts/registry.ts`, `src/abstract-surface.ts`, `src/workflows/agent-loop.ts`, `src/workflows/project-ai-sdk.ts`); no new spec needed. Consider a regression test that mocks the filesystem-walk count and asserts ‚â§1 walk per Vault per surface build.

### F5. Off-matrix invariant + gotcha lockstep is reviewer-memory, not structure ‚Äî AC3 silently passes on stubs

**Severity:** Medium
**Category:** Invariant enforcement / Test guarantee

**Why it matters:** Multiple substrate artifacts declare "enforced elsewhere" without a structural fence verifying the elsewhere actually exists. The pattern works today by reviewer attention; it doesn't survive contact with a v1 contributor or agent who doesn't share the reviewers' memory.

- **AC3 lockstep on `core-has-no-llm-or-mcp-dependency`** at `tests/invariants/core-has-no-llm-or-mcp-dependency.test.ts:14-19` is `expect(true).toBe(true)`. The AC3 gotcha-coverage test at `tests/integration/gotcha-coverage.test.ts:24-49` accepts `off-matrix` as a valid coverage value with no further check. A regression that drops the watcher's chokidar config (`out-of-band-vault-edits`), or that retries hook handlers without idempotency-checking the effects (`hook-non-idempotent`), would not fail any test. The bar of "named gotcha ‚Üí AC3-lockstep test" is currently aspirational for everything but the one matrix-tagged gotcha.

**Evidence:** `/Users/mark.toda/dev/dome/tests/invariants/core-has-no-llm-or-mcp-dependency.test.ts:14-19`; `/Users/mark.toda/dev/dome/tests/invariants/hook-dispatch-is-vault-bound.test.ts`; `/Users/mark.toda/dev/dome/tests/integration/gotcha-coverage.test.ts:24-49`; `/Users/mark.toda/dev/dome/tests/integration/matrix-coverage.test.ts:16`; `/Users/mark.toda/dev/dome/docs/wiki/matrices/tool-invariant-enforcement.md:13-18,65`.

Then change AC3 meta-check at `invariant-coverage.test.ts` to require the file to contain a `describe()` block whose body imports or runs the enforcement, not just `expect(true).toBe(true)`. Document the convention in `docs/wiki/conventions/off-matrix-lockstep.md`.
2. **Gotcha coverage.** Extend `coverage:` frontmatter from a single string to `coverage: off-matrix` + `enforced_at: tests/integration/<file>.test.ts`. AC3 check requires the named file to exist. Gotchas with no extant enforcement become candidates for the v0.5.1 backlog.

### F6. YAML and JSON ingestion hand-rolls validation in three places where Zod is the pattern

**Severity:** Medium
**Category:** Library misuse / Type-system circumvention / Reinvented primitive

**Why it matters:** The SDK already uses Zod pervasively (`src/tools/schemas.ts`, `src/prompts/workflow-frontmatter.ts`, every MCP descriptor via `zod-to-json-schema`) and the discipline of `Result<T, E>` at boundaries holds across the Tool surface. Three persistence/loader boundaries break the pattern:

- `src/hooks/yaml-loader.ts:106-136` ‚Äî parses YAML, then hand-validates with `typeof raw.event !== "string"` / `typeof raw.workflow !== "string"` / `typeof raw.path_pattern !== "string"`, each throwing string errors. Yields stringly-typed errors that throw rather than returning a `Result<DeclarativeHook, ValidationError>`.
- `src/vault.ts:141-153` ‚Äî `parseYaml(cfgText) as Partial<VaultConfig>` then a spread merge. The cast is unverified; a `config.yaml` shaped `{invariants: "string-instead-of-object"}` slips past TypeScript and explodes inside the spread.
- `src/reconcile.ts:121` ‚Äî `JSON.parse(...) as Record<string, ScheduledEntry>` for `.dome/state/scheduled.json`. `src/quarantine-store.ts:33-34` ‚Äî `JSON.parse(text); Array.isArray(parsed) ? parsed.filter(x => typeof x === "string") : []` for `.dome/state/quarantined.json`. Both hand-defensive; both silently empty-fallback on corruption.

For v1 surfaces hot-reloading on-disk state, silent shape-corruption becomes a debug nightmare.

**Evidence:** `/Users/mark.toda/dev/dome/src/hooks/yaml-loader.ts:106-136`; `/Users/mark.toda/dev/dome/src/vault.ts:141,153`; `/Users/mark.toda/dev/dome/src/reconcile.ts:121`; `/Users/mark.toda/dev/dome/src/quarantine-store.ts:29-39`.
