# Rewrite Validation Review — Dome v0.5 core-shell seam (Phase B pass-1)

**Verdict:** Issues Found

## Executive judgment

The architectural direction is sound and well-realized. The four-entrypoint shape genuinely solves F1 (the package-bundling boundary is now structural, not just conceptual); `ConsumerSurface` lands as a real runtime construct with a coherent three-doc story (sdk-surface defines it, mcp-surface consumes it, harnesses points future shells at it); and the new axiom is properly off-matrix with a parallel structural-enforcement story to `INBOX_IS_EPHEMERAL`. **However,** the rewrite ships with three concrete, fresh-eyes-discoverable defects: an internal contradiction in the Vault lifecycle spec about who owns watcher cleanup, four wrong cells in the `consumer-surface` matrix that contradict the very axiom the matrix is supposed to pin, and the ledger's `## Delta at a glance` preamble overcounts files vs. the body.

## Blocking issues

### B1. Vault lifecycle contradicts itself on watcher ownership

- **Severity:** High
- **Category:** Spec drift
- **Why it matters:** `sdk-surface.md §Vault lifecycle` describes `close()` step (b) as "release any held file handles (e.g., chokidar watchers started by `VaultWatcher`)" — and the very next paragraph says "`vault.close()` does NOT stop watchers it didn't create; callers that started a watcher hold that responsibility." These cannot both be true. An implementer landing `code-vault-close-impl` (ledger entry 4) cannot decide whether `vault.close()` releases watcher resources or refuses to.
- **Evidence:** `sdk-surface.md:60` ("(b) release any held file handles (e.g., chokidar watchers started by `VaultWatcher`)") vs `sdk-surface.md:62` ("`vault.close()` does NOT stop watchers it didn't create").
- **Recommended fix:** Remove the chokidar example from step (b) and replace with a concrete Vault-owned resource (e.g., the `p-queue` instance; the file handle on `.dome/state/quarantined.json`). Or: if `VaultWatcher` is always caller-owned, drop step (b) entirely and have `close()` be `drainHooks()` + invalidation only.
- **Substrate artifact to add or update:** Spec (`sdk-surface.md §Vault lifecycle`).

### B2. Consumer-surface matrix has cells that contradict the entrypoint legend

- **Severity:** High
- **Category:** Spec drift / Invariant
- **Why it matters:** The matrix is the substrate-shape pin for `CORE_HAS_NO_LLM_OR_MCP_DEPENDENCY`. Several cells today contradict the invariant directly:
  - **MCP server row, Tools cell** reads `core (via vault.tools + projectMcp(vault))`. `projectMcp` lives in `@dome/sdk/mcp` — the cell should read `core + mcp`, not `core`.
  - **Future HTTP row, Tools cell** reads `core (via projectAiSdk(vault) or future projectHttp(vault))`. `projectAiSdk` lives in `@dome/sdk/workflows` — labeling that reach as `core` directly contradicts the invariant.
  - **Future HTTP row, ConsumerSurface cell** reads `core-extended (or a future buildHttpSurface...)`. `buildConsumerSurface` lives in `@dome/sdk/mcp` today; "`core`-extended" is not in the legend.
- **Evidence:** `consumer-surface.md:27` (MCP server row), `:32` (Future HTTP row), and the legend at `:14-20` enumerating exactly five legal cell values.
- **Recommended fix:** Re-cell those rows: MCP server Tools → `core + mcp`; Future HTTP Tools → `core + workflows`; Future HTTP ConsumerSurface → `mcp` today, future-annotation only. Add an explicit "compound cell" form to the legend if cells legitimately need to name two entrypoints.
- **Substrate artifact to add or update:** Behavior matrix (`consumer-surface.md`).

### B3. Ledger preamble counts diverge from the body

- **Severity:** High
- **Category:** Spec drift
- **Why it matters:** The preamble is what consumers (this review; future `implement-cohesively` Step 1) quote verbatim at decision time. A preamble that miscounts files makes the consumer's render misleading.
- **Evidence:**
  - Preamble (`ledger:13`): "5 substrate files rewritten (4 specs, 1 matrix); 1 substrate catalog updated."
  - Body (§"Substrate rewritten (5 files)"): lists `sdk-surface.md`, `mcp-surface.md`, `harnesses.md`, `tool-invariant-enforcement.md`, `docs/index.md` — that's **3 specs + 1 matrix + 1 catalog = 5 files**, not "4 specs + 1 matrix + 1 catalog = 6."
  - Preamble "7 substrate files added... 1 new test file is implementation-deferred" — the test file should not be in the added-files count.
- **Recommended fix:** Update the preamble's Files line to: "6 substrate files added (1 invariant, 3 gotchas, 1 matrix, 1 new delta ledger); 5 substrate files rewritten (3 specs, 1 matrix, 1 catalog)." Re-title the body §"Substrate added (7 files)" to "(6 files)."
- **Substrate artifact to add or update:** Delta ledger.

## Important issues

### I1. `projectMcp` shape is underspecified

- **Severity:** Medium
- **Category:** Domain model
- **Why it matters:** `sdk-surface.md` declares `projectMcp(vault): { parsers, ... }` — the `...` is the only thing telling the implementer the return shape isn't just `{ parsers }`. Compare with `projectAiSdk(vault): ai.ToolSet` which is fully shape-typed.
- **Evidence:** `sdk-surface.md` Vault surface section; `mcp-surface.md §Construction` does not name `projectMcp`.
- **Recommended fix:** Either replace the `...` with the concrete return type, or move `projectMcp` to a private helper that `buildConsumerSurface` calls internally.
- **Substrate artifact to add or update:** Spec (`sdk-surface.md §Vault surface`).

### I2. `buildConsumerSurface` async vs sync

- **Severity:** Medium
- **Category:** Domain model
- **Why it matters:** `sdk-surface.md` declares `function buildConsumerSurface(vault: Vault): Promise<ConsumerSurface>;` but the spec doesn't name what's async. If there's filesystem work (plugin/vault-local prompt discovery), say so; if not, drop `Promise<>`.
- **Evidence:** `sdk-surface.md §Consumer surfaces`; `mcp-surface.md:22-24` uses `await`.
- **Recommended fix:** Add one sentence naming the async work, or drop the `Promise<>` wrapper.
- **Substrate artifact to add or update:** Spec (`sdk-surface.md §Consumer surfaces`).

### I3. Gotcha-coverage lockstep test needs a frontmatter convention

- **Severity:** Medium
- **Category:** Enforcement
- **Why it matters:** The ledger proposes `gotcha-coverage.test.ts` (every gotcha doc has a test OR `coverage: off-matrix` frontmatter). Three new gotchas (`transitive-llm-dependency`, `substrate-count-drift`, `ai-sdk-tool-variance`) explicitly have no per-gotcha test intended — yet none carry a `coverage:` frontmatter field, which the proposed test requires.
- **Evidence:** All three new gotcha frontmatters are `type / created / updated / sources / severity / first_observed` only; ledger references `coverage: off-matrix` but no doc has the field.
- **Recommended fix:** Either (a) name the frontmatter field shape (`coverage: matrix | off-matrix | deferred`) and add it to the three new gotchas, or (b) drop the off-matrix escape and write per-gotcha tests (each can assert the structural mitigation is in place).
- **Substrate artifact to add or update:** Convention + frontmatter for all three new gotchas; or rework the lockstep test spec.

### I4. MCP Tool catalog omits `expected_mtime`

- **Severity:** Low
- **Category:** Spec drift
- **Why it matters:** `mcp-surface.md §Tool catalog` shows the three mutating tools' input shapes without `expected_mtime`. The canonical SDK signatures at `sdk-surface.md §Tool signatures` include `expected_mtime?: string` on `writeDocument`, `moveDocument`, `deleteDocument` for optimistic locking.
- **Evidence:** `mcp-surface.md` Tool catalog table rows for `dome.write_document`, `dome.move_document`, `dome.delete_document`.
- **Recommended fix:** Add `expected_mtime?` to the three mutating MCP tool input rows with a one-line link to `sdk-surface.md §Concurrency`. Or, if MCP intentionally omits optimistic locking, name why.
- **Substrate artifact to add or update:** Spec (`mcp-surface.md §Tool catalog`).

## Substrate gaps

- **No `vault.close()` test guarantee named.** The lifecycle is normative, has one-shot semantics, must drain hooks idempotently. None of the ledger's proposed tests pin this. A `tests/invariants/vault-lifecycle.test.ts` or equivalent should land alongside `code-vault-close-impl` (ledger entry 4).
- **No regression test for "deleted from core public surface."** Ledger entry 9 (`code-core-index-prune`) removes 14 named symbols from `src/index.ts`. The bundle-deps test catches the transitive-dep outcome but not the symbol-removal itself.

## Locality concerns

- **`ConsumerSurface` lives in `@dome/sdk/mcp` but is named in three docs.** The choice is defensible (MCP is the first protocol consumer). But the matrix's Future-HTTP row labels the ConsumerSurface cell `core-extended` — implying a future migration. Either resolve via B2's fix or name the migration explicitly.

## Enforcement concerns

- **`CORE_HAS_NO_LLM_OR_MCP_DEPENDENCY` enforcement is well-pinned.** `bundle-deps.test.ts` is named, the introspection mechanism is described, the failure mode is concrete. This is the model the rest of the rewrite should match.
- **`substrate-count-drift` has no enforcement in v0.5.** The gotcha doc says `dome doctor --check-substrate-counts` is deferred to v0.5.1+. Honest deferral; acceptable.

## Behavior knowable outside implementation?

The four-entrypoint shape, Vault surface (post-shrink), Vault lifecycle (modulo B1), and `ConsumerSurface` are reproducible from the docs alone — except for B1 (watcher ownership), I1 (`projectMcp` shape), and I3 (frontmatter convention).

## Vague language to tighten

Nothing in normative sections. The "may" / "should" instances I checked all live in non-normative sections (future-pressure, anti-concept list, "cells that may grow") or are grammatically necessary.

## Recommended repairs (ranked)

1. **Resolve B1** by picking one watcher-ownership story.
2. **Re-cell the wrong cells in `consumer-surface.md`** (B2). Add compound-cell legend if needed.
3. **Tighten the ledger preamble counts** (B3).
4. **Specify `projectMcp`'s return type** (I1) or move it to a private helper.
5. **Disambiguate `buildConsumerSurface`'s async signature** (I2).
6. **Decide on the `coverage:` frontmatter convention** (I3).
7. **Add `expected_mtime?` to the MCP Tool catalog table** (I4) — or document its absence.

## What looked right

- **`CORE_HAS_NO_LLM_OR_MCP_DEPENDENCY` is a textbook axiom rewrite.** Statement, why, structural enforcement, off-matrix justification, counter-example, test guarantee, related links — every section concretely filled.
- **The three-doc `ConsumerSurface` story holds together.** `sdk-surface.md` defines; `mcp-surface.md` consumes; `harnesses.md` points future shells at it. Audience separation done right (modulo B2).
- **The `transitive-llm-dependency` gotcha tells a complete story.** Symptom → severity → root cause → two-layer mitigation → four specific scenarios → operational notes.
- **Deferred work is named and bounded.** The `## Deferred` block in the ledger isn't a TODO list — it's five concrete items with justifications.
