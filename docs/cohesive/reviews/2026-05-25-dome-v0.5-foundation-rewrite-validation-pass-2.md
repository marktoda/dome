## Blocking issues

### B1. Hook cycle detection has two contradictory semantics
- **Severity:** Blocker
- **Category:** Spec drift
- **Why it matters:** Two different dispatcher implementations would both claim to satisfy the spec. An auto-cross-reference hook that fans out 20 backlinks across 20 entity pages produces depth-20 chains. Under hooks.md semantics this triggers `hook.cycle-detected` (depth > 5) and quarantine. Under hook-cycle.md semantics the chain proceeds because no (same-hook, same-page) repetition occurs. The shipped default's behavior depends on which spec the implementer reads first.
- **Evidence:** `docs/wiki/specs/hooks.md:120` defines cycle prevention as "A causation chain depth limit (configurable, default 5)" — strict depth-limit semantics. `docs/wiki/gotchas/hook-cycle.md:18-22` defines it as "is depth > 5?" *but also* `:34` "depth tracker sees the chain doesn't include the *same* hook invoked twice on the *same* page, so it's allowed" — per-(hook, page) repetition semantics that explicitly contradicts a strict depth limit.
- **Recommended fix:** Pick one mechanism. The per-(hook, page) check is the right one for fan-out hooks like `auto-cross-reference`; rewrite hooks.md `§"Execution model"` and hook-cycle.md `§"Structural mitigation"` to agree. If keeping both signals, name them separately (e.g., "depth-limit safety net" + "per-key cycle detector") with explicit composition.
- **Substrate artifact to add or update:** `wiki/specs/hooks.md` §"Execution model"; `wiki/gotchas/hook-cycle.md` §"Structural mitigation".

### B2. `writeDocument` input shape is undefined in sdk-surface.md and contradicted across three specs
- **Severity:** High
- **Category:** Domain model
- **Why it matters:** `writeDocument` is the universal mutation entrypoint and gate for 6 invariants. The SDK spec carries no signature; downstream docs each invent a different one. An agent implementing the Tool from sdk-surface alone could not produce a callable function.
- **Evidence:** `docs/wiki/specs/sdk-surface.md:78-79` Tool catalog row gives `writeDocument` invariants enforced but no input shape. `docs/wiki/specs/mcp-surface.md:32` declares `{ path, body, frontmatter, reason?, sensitivity_classified? }`. `docs/wiki/invariants/PAGE_CREATION_REQUIRES_RECURRENCE.md:25` declares `writeDocument(path, body, frontmatter, { create: true, reason: ... })` — `create` and `reason` move into an options object. `docs/wiki/invariants/SENSITIVE_GOES_TO_INBOX.md:25` says "`writeDocument` accepts an optional `sensitivity_classified: 'normal' | 'sensitive'` input field" — flat shape.
- **Recommended fix:** Add a normative `writeDocument` signature block to `sdk-surface.md` §"Tool catalog" (or a dedicated subsection): `writeDocument(input: WriteDocumentInput)` with the full Zod schema, including which fields are gated by which opt-in invariants. Other specs cite this signature rather than restate it.
- **Substrate artifact to add or update:** `wiki/specs/sdk-surface.md` Tool catalog.

### B3. `sensitivity-classify` firing site is contradicted between hooks.md and intent-prompt-tools.md
- **Severity:** High
- **Category:** Spec drift
- **Why it matters:** The two specs describe two different runtime behaviors. As a hook on `document.written.inbox.raw`, the classifier runs *after* the file lands in inbox — too late to prevent a write to wiki/ (the invariant says sensitivity-classifier *gates* the write destination). As a sub-workflow inside `ingest`, the classifier runs *before* `writeDocument`, which is what the invariant requires.
- **Evidence:** `docs/wiki/specs/hooks.md:102` lists `sensitivity-classify-on-ingest` template as a hook with event `document.written.inbox.raw` invoking the `sensitivity-classify` workflow "after-ingest classification and routing." `docs/wiki/matrices/intent-prompt-tools.md:28` says `sensitivity-classify` is a "sub-workflow inside `ingest` when `SENSITIVE_GOES_TO_INBOX` is enabled, or pre-write hook." `docs/wiki/invariants/SENSITIVE_GOES_TO_INBOX.md:27` says "typically as a pre-write sync hook in personal vaults" — pre-write, not post-write.
- **Recommended fix:** Pick one. The pre-write sync-hook (or sub-workflow-inside-ingest) shape is the right one because of the invariant's gate semantics. Strike the `document.written.inbox.raw` post-write framing from hooks.md §"Opt-in intake patterns" and replace with the pre-write sync-hook shape.
- **Substrate artifact to add or update:** `wiki/specs/hooks.md` §"Opt-in intake patterns"; `wiki/matrices/intent-prompt-tools.md` row for `sensitivity-classify`.

### B4. The `(auto)` notation in the Tool catalog is undefined and semantically load-bearing
- **Severity:** High
- **Category:** Vague language
- **Why it matters:** Three Tool-catalog rows annotate `EVERY_WRITE_IS_LOGGED (auto)` without defining what `(auto)` means. The invariant doc says "Mutating Tools structurally include an `appendLog` step in their implementation" — meaning the Tool *automatically* emits the log entry on behalf of the caller. But that semantic is invisible in the catalog.
- **Evidence:** `docs/wiki/specs/sdk-surface.md:79`, `:83`, `:84` — three Tool rows use the `(auto)` notation. No definition appears in `§"Tool catalog"`, in the table legend, or in `§"Why this design"`. The matrix at `docs/wiki/matrices/tool-invariant-enforcement.md:30` spells the same semantic clearly: "emits appendLog effect when default enabled."
- **Recommended fix:** Either define `(auto)` once near the catalog (e.g., "`(auto)` means the Tool's effect array always includes the `appendLog` Effect — the caller does not separately invoke `appendLog`") or replace each occurrence with the matrix's phrasing.
- **Substrate artifact to add or update:** `wiki/specs/sdk-surface.md` §"Tool catalog" preamble.

## Important issues

### I1. `readDocument` matrix cell "*(parses links to detect)*" is opaque
- **Severity:** Medium
- **Category:** Domain model
- **Why it matters:** `readDocument` is documented as read-only (sdk-surface.md), yet the matrix annotates the cell under WIKILINKS_ARE_FULLPATH with "*(parses links to detect)*" — detect what, and detect for whom?
- **Evidence:** `docs/wiki/matrices/tool-invariant-enforcement.md:29` — the cell. No supporting prose in sdk-surface.md `readDocument` row.
- **Recommended fix:** Either delete the cell or expand to one clear phrase: "*(returns parsed wikilinks; does not reject short-form)*".
- **Substrate artifact to add or update:** `wiki/matrices/tool-invariant-enforcement.md`.

### I2. `index.md` ownership has no behavior spec when `auto-update-index` is disabled
- **Severity:** Medium
- **Category:** Locality
- **Why it matters:** `vault-layout.md` ownership table says index.md is "Mutated only through the auto-update-index shipped-default hook," and sdk-surface.md explicitly states the hook can be disabled per-vault. When disabled, index.md becomes a tracked file with no owner.
- **Evidence:** `docs/wiki/specs/vault-layout.md:114`; `docs/wiki/specs/sdk-surface.md:121`. No fallback rule in `docs/wiki/specs/cli.md` §"dome doctor" or §"dome reconcile".
- **Recommended fix:** Add a one-paragraph behavior note: "When `auto-update-index` is disabled, `index.md` is unmaintained — `dome doctor --rebuild-index` produces a fresh one on demand. `writeDocument('index.md', ...)` is rejected unconditionally (the index is dispatcher-owned, not Tool-callable)."
- **Substrate artifact to add or update:** `wiki/specs/vault-layout.md` ownership table; or `wiki/specs/sdk-surface.md` §"Hook" shipped-defaults subsection.

### I3. Eval suite is described as "(proposed v0.5 surface)" in a normative spec
- **Severity:** Medium
- **Category:** Vague language
- **Why it matters:** The eval suite is the *structural mitigation* for the highest-severity gotcha. "Proposed" is hedge language in a doc that is supposed to describe end state.
- **Evidence:** `docs/wiki/specs/prompts-and-workflows.md:109` — heading "## Eval suite (proposed v0.5 surface)".
- **Recommended fix:** Strike "(proposed v0.5 surface)" from the heading.
- **Substrate artifact to add or update:** `wiki/specs/prompts-and-workflows.md` §"Eval suite".

### I4. Tier vocabulary drift between invariant frontmatter and matrix legend
- **Severity:** Low
- **Category:** Spec drift
- **Why it matters:** Frontmatter uses `tier: shipped-default`; matrix legend uses "**Default**". Two vocabularies for the same tier.
- **Evidence:** `docs/wiki/invariants/EVERY_WRITE_IS_LOGGED.md:6` `tier: shipped-default`. `docs/wiki/matrices/tool-invariant-enforcement.md:22` legend "**Default** — shipped enabled".
- **Recommended fix:** Normalize the matrix legend to "**Shipped default**".
- **Substrate artifact to add or update:** `wiki/matrices/tool-invariant-enforcement.md` §"Tier legend".
