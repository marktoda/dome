# Rewrite Validation Review — Dome v0.5 spec drift repair (Phase A)

**Verdict:** Approved

## Architectural reflection

How it feels now: the spec stops drifting in the four places Phase A targeted, and it does so by hardening the substrate (a two-layer enforcement for `INBOX_IS_EPHEMERAL`, a derived-state-as-single-source convention, link-to-canonical-surface for inline counts) rather than partitioning honest from aspirational text. The rewrite is small enough that a future contributor can read it in one sitting and large enough to make `implement-cohesively` mechanical.

- **Easier downstream:** new shipped flags inherit a clean spec home (single `dome doctor` flag block in `cli.md`; quarantine-record explained in three load-bearing places — cli/hooks/vault-layout — that agree byte-for-byte).
- **Harder downstream:** the matrix cross-reference at `tool-invariant-enforcement.md:15` still names `moveDocument` as the workflow exit step, while the rewritten invariant doc commits to `deleteDocument`. Future readers who land at the matrix first will see the wrong primitive. Out of stated scope (the ledger explicitly lists matrices as "untouched"), but it's the one drift the rewrite created.
- **Load-bearing on memory:** `INBOX_IS_EPHEMERAL` is still primarily prompt-enforced; the doctor fallback is now spec'd, but the test guarantee assumes the implementer wires it. The two layers are clear, so the memory load is bounded.

## Executive judgment

The rewrite closes F3 (doctor-flag spec drift + `INBOX_IS_EPHEMERAL` undefended), F4 (count drift in synthesis + audit-trail in prior delta ledger), the F7 piece scoped to this pass (quarantined-state hidden rule), and the F8 piece scoped to this pass (shipped `AGENTS.md` shape now in the CLI spec). The four rewritten specs and the invariant agree with each other on quarantine, on the doctor's `INBOX_IS_EPHEMERAL` check, on the derived-state inventory, and on the AGENTS.md/CLAUDE.md shape. The "code catches up to spec" implementation handoff is concrete enough that an implementer can drive without re-deriving intent. One residual finding (matrix `moveDocument` vs invariant `deleteDocument`) lands as Medium — within Phase A's scope to either accept or close.

## Delta at a glance

**Classification:** **Mixed.** Design-layer changes describe an end state where (a) `INBOX_IS_EPHEMERAL` has a structural doctor-check fallback alongside the workflow-prompt enforcement, (b) `.dome/state/quarantined.json` joins `scheduled.json` and `last-reconciled-sha.txt` as canonical derived state, (c) inline counts in synthesis docs link to canonical const-array surfaces instead of inlining stale numbers, and (d) `dome init`'s shipped scaffolding (`AGENTS.md` + `CLAUDE.md` shim) is normatively documented in the CLI spec rather than implied by source code. Implementation-layer changes are deferred to `implement-cohesively`.

## Important issues

### I1. Workflow exit-step primitive disagrees between the matrix cross-reference and the rewritten invariant

- **Severity:** Medium
- **Category:** Spec drift
- **Why it matters:** A reader who lands at the cross-reference target before reading the invariant will internalize the wrong primitive.
- **Evidence:** `docs/wiki/invariants/INBOX_IS_EPHEMERAL.md:25` (`deleteDocument` as the v0.5 mechanism) vs `docs/wiki/matrices/tool-invariant-enforcement.md:15,40` (`moveDocument` as the exit step).
- **Recommended fix:** One-line edit to the matrix at both locations: `moveDocument` → `deleteDocument` in the workflow-enforcement description.
- **Substrate artifact to add or update:** Edit to `docs/wiki/matrices/tool-invariant-enforcement.md`.

### I2. `dome doctor` flag list is monolithic — implementer cannot tell shipped from "to-be-implemented this pass"

- **Severity:** Medium
- **Category:** Enforcement (handoff clarity)
- **Why it matters:** A fresh reader of `cli.md` alone cannot tell from the spec which flags need code work. The partition lives in the ledger §"Implementation handoff."
- **Evidence:** `cli.md:138-147` (nine flags, no shipped-state distinction) vs the ledger's Implementation handoff section.
- **Recommended fix:** Acceptable as-is. The implementer reads the ledger before the spec.
- **Substrate artifact to add or update:** None.

### I3. `INBOX_IS_EPHEMERAL` per-bucket `ephemeral: false` knob references a config shape `hooks.md` doesn't define

- **Severity:** Low
- **Category:** Spec drift (minor)
- **Why it matters:** A future contributor implementing per-bucket disable wouldn't know if the namespace is `inbox.buckets.*` (per the invariant) or `hooks.inbox.buckets.*`.
- **Evidence:** `INBOX_IS_EPHEMERAL.md:31-41` vs `hooks.md:148-157`.
- **Recommended fix:** Either add `inbox.buckets.<name>.ephemeral` to the `hooks.md` config table, or note in the invariant that the per-bucket knob is deferred and the threshold-bump is the v0.5 mechanism.
- **Substrate artifact to add or update:** Either `docs/wiki/specs/hooks.md` config table extension, or `docs/wiki/invariants/INBOX_IS_EPHEMERAL.md` deferring the per-bucket knob.
