# Dome v1 — self-sustaining, ambient, shareable-by-construction

Approved 2026-06-11. The v0.5→v1 plan that follows the wedge ([[wedge]] Phases 1–5 are
landed or in-flight; Phase 5 MCP shipped). Six workstreams over the sealed four-concept
core — every item below is a processor, an adapter, or a recipe. No new primitives.

## Scope decision

**v1 is the owner's polished daily driver, with shareability guardrails baked in.**
Not yet an installable product — but every design choice passes the test *"would this
decision block or embarrass a second-user install?"* Concretely:

- No owner-specific behavior hardcoded outside vault config.
- Security designed as if a token holder might not be the owner (per-device token
  issuance/rotation from day one).
- New surfaces built as protocol adapters over `src/surface/` collectors, per
  [[wiki/specs/mcp-surface]] precedent — never bespoke side daemons.
- A running "second-user blockers" ledger accumulates as WS1–5 land; burning it down
  is the v1 exit gate (WS6).

## Why this plan (review findings, 2026-06-11)

A four-track review (processor inventory, work-vault audit, external research, plan
inventory) established:

1. **The gardening works.** The feared append-itis is not present in the work vault.
   Entity pages are milestone-driven syntheses; the consolidation and sweep ledgers
   show real nightly tidies and integrations; the inbox drains; signal density is
   high. The engine co-authors the vault at 8–12 patches/day.
2. **The real rot is in Dome's own bookkeeping.** `index.md` (~42k tokens) exceeded
   safe tooling mutation (the chip-huyen index entry escalated to owner three runs
   running); `log.md` (619k chars) can no longer be appended by the consolidator;
   generated/system pages sit in indexing limbo. The gardener's artifacts need the
   lifecycle treatment the wiki pages already get.
3. **External convergence validates the architecture.** Letta Context Repositories
   (memory as git-repo files, sleep-time agents in worktrees), Anthropic's Claude
   Dreaming (background curation behind a human review gate), and the mem0 production
   audit (97.8% junk from unsupervised fact extraction with hallucination feedback
   loops) all confirm: files + git + offline consolidation + propose-not-auto is the
   evidenced design. LightMem (offline-staged consolidation, up to 117× token
   reduction) plus Batch API (50%) plus prompt caching bound garden costs to
   single-digit dollars/month.
4. **Ambient access is nearly free.** The HTTP surface (capture/query/tasks/
   questions/resolve, bearer token) and the MCP server (8 tools) already exist;
   mobile capture is a recipe, not a feature. Voice wearables are an acquisition
   graveyard (Limitless→Meta, Bee→Amazon); deliberate voice capture via the phone is
   the dependable bet.
5. **The "two parallel threads" problem is whitespace.** No product cleanly separates
   the task stream from the capture/yap stream. The scroll-up pain exists because the
   chat transcript doubles as the task surface; the durable task state already lives
   in the daily note. The fix is a cheap-to-summon rendered view, not a new store.

## Workstreams

### WS1 — The gardener maintains its own artifacts

Revised 2026-06-11 (same day, post-approval discussion): the original sharding and
rotation fixes treated symptoms. Both central registries were agents *maintaining*
what the engine should *project*; the revision makes that the rule.

- **Index = compiled projection, descriptions live with pages.** Each page carries a
  one-line `description:` in frontmatter (lint-enforced; ingest writes it as part of
  page creation — a tiny local edit). An adoption-phase deterministic processor
  projects `dome.page.description` facts; a renderer compiles `index.md` (root map:
  categories, counts, conventions) plus per-category shards as *generated* files,
  paginated automatically on a size budget. No model ever edits an index file again;
  the "too large for tooling" failure ceases to exist rather than being deferred.
  Sharding survives only as a rendering detail. Indexing becomes declarative
  (`index: false` / page-type rules), which also resolves the orphan-system-page
  limbo. Agents resolve pages via `dome query`; the index is the orientation and
  browsing surface, not the lookup mechanism. Migration is one-shot and nearly
  deterministic: parse existing `index.md` entries into per-page frontmatter.
- **Git is the native activity log; `log.md` is frozen.** A log.md entry summarizes
  a *run* — which maps onto the closure commit. The narrative the ingest/sweep/
  consolidate processors currently append to log.md moves into the closure-commit
  message body, making engine commit-message shape a spec'd contract (subject +
  narrative body + `Dome-*` trailers) — this also supplies the missing audit chain
  ("why was X merged into Y?"). A `dome log` view compiles recent narrative from
  git trailers + run ledger (`--since`, `--processor`, `--grep`). `log.md` freezes
  as `log-archive-through-2026-06.md`; no rotation machinery is built. Accepted
  regression: commit messages are not FTS-indexed (activity lookup is `dome log`,
  not `dome query`). Durability improves: narrative replicates with every vault
  clone, unlike gitignored runs.db.
- **Candidate invariant: NO_ACCRETING_REGISTRIES.** Both fixes instantiate the same
  rule — every central vault artifact is either source-of-truth markdown or a
  deterministic render from per-item sources; no file exists whose maintenance
  contract is "agents append entries forever." Pin it so the next bookkeeping file
  doesn't sneak in.
- **core.md lifecycle.** Preference promotion stays the single writer
  ([[wiki/specs/task-lifecycle]] M4/M5); add a scheduled pruning processor that
  *proposes* demotion of stale promoted items to `core-archive.md`. Caps the one
  always-loaded file.
- **Resilience trio.**
  - Brief-failure recovery: `dome.agent.brief` crash → escalated question +
    carry-forward falls back to prior-day context instead of a bare skeleton.
  - Sweep poison items: after 3 failures an item moves to `inbox/sweep-failed/` with
    a question — no indefinite requeue.
  - Consolidate scope config: `consolidate.targets` mirroring the sweeper's existing
    config pattern; the MAX_CHANGED_FILES cap stops being the only control.

### WS2 — Economics contract

Make the model provider batch-and-cache aware: nightly garden passes (consolidate,
sweep, warden) submit via the Batch API; per-note calls share a cached prompt prefix
(vault conventions + index map); triage/classification routes to a cheap model tier,
frontier models reserved for synthesis and merge proposals. The delta-ledger
architecture already does the hard part (process only drift). Target: bounded
single-digit $/month garden — a shareability feature as much as a cost one.

### WS3 — Ambient capture + Q&A

**Deployment topology (decided 2026-06-11):** the daemon runs on the owner's
always-on home server, on the Tailscale network — not the laptop. This is a
reliability upgrade (nightly gardens at 02:00/03:00/05:30 no longer depend on a
laptop lid being open) and gives phone recipes a stable endpoint. It requires the
daemon install path to be host-portable (launchd today; systemd if the server is
Linux) and raises the laptop-write-path question (see open questions).

- **Capture (recipes, zero engine code).** iOS Shortcut: Action Button / Apple Watch
  dictation → `POST /capture` to the server over the Tailscale trust domain
  ([[wiki/specs/http-surface]]). Ship as documented recipes plus a `dome recipe ios`
  helper that prints endpoint + token setup.
- **Q&A.** Expose `dome mcp` remotely (streamable-HTTP transport or `mcp-remote`
  bridge; bearer token; Tailscale trust domain) so the Claude iOS app — including
  voice mode — is the mobile front end, with the same 8 tools desktop sessions use.
  One brain, every device. The long-horizon "assistant in my ear" is AirPods +
  Claude voice mode + dome MCP; no hardware bet.
- Per-device token issuance/rotation designed now, single-user notwithstanding.

### WS4 — The cockpit

A live, self-refreshing render of today — tasks, open loops, fresh captures, pending
questions — built as the existing `dome.daily.today` view behind two thin adapters:

- `dome today --watch` (terminal),
- `GET /today` HTML page from the HTTP server (phone/second monitor; refreshes on
  adoption events).

Chat becomes pure yap-stream; settlement happens via checkbox, chat, or `resolve`,
and the cockpit reflects it within seconds. View processor + adapter routes only —
the [[wiki/specs/sdk-surface]] AbstractSurface direction, not a new engine feature.

### WS5 — Sources stop needing a courier

The owner is currently the integration layer (hand-feeding Slack and docs).
`dome.sources.fetch` exists and ships no feeds. Wire two:

1. **Calendar** — committed source files, already spec'd in [[wedge]] Phase 4; finish.
2. **Slack** — mentions/DMs/selected channels → `inbox/raw/` captures via the
   existing consent-gated fetch protocol. The trusted ingest→sweep pipeline does the
   rest.

### WS6 — Second-user gate (v1 exit criterion)

One person who is not the owner installs Dome (`dome init` → daemon → capture →
first morning brief) using only the docs. No onboarding UX is built; the
"second-user blockers" ledger accumulated during WS1–5 gets burned down at the end.

## Sequencing

1. **WS3-capture + WS4** — highest daily-life impact per unit work; both build on
   shipped surfaces.
2. **WS1** in parallel — mostly deterministic processors; cheap, testable.
3. **WS3-MCP-remote**, then **WS5** (calendar → Slack), then **WS2**.
4. **WS6** closes v1.

## Non-goals for v1

- Native iPhone app (v1.5 candidate once remote MCP proves the workflow).
- Embeddings (deferred-to-v1.1 decision stands; FTS + facts performing per audit).
- Passive/ambient always-on capture (wearable platform graveyard).
- Telegram or other bot channels (superseded by Claude-mobile-over-MCP; revisit only
  if a non-Apple channel becomes a requirement).
- Any new core primitive. Vault / Proposal / Processor / Effect stays sealed.

## Open questions carried into planning

- ~~Laptop write path once the vault lives on the server~~ — **decided 2026-06-11:
  SSH to the server.** Desktop deep-work sessions run Claude Code on the server
  (ssh/tmux or Claude Code remote); the vault stays single-residency, the daemon
  sees commits instantly, and no sync seam is built. Git-clone-push (the v1.5
  hosted-mode seam) stays deferred; remote-MCP-only remains blocked by design (MCP
  has no document-mutation tools). Accepted costs: sessions need connectivity;
  feeding in laptop-local files means scp or `POST /capture`.
- Daemon host portability: `dome install` is launchd-only; server deployment needs
  the systemd equivalent if the server is Linux. Decide in plan.
- Remote MCP transport: native streamable-HTTP server in `src/mcp/` vs. an
  `mcp-remote`-style bridge in front of the stdio entrypoint. Decide in plan.
- Engine commit-message contract: exact shape of the narrative body and how garden
  sub-proposal commits reference their run's closure narrative. Decide in plan.
- `GET /today` refresh mechanism: SSE push on adoption events vs. dumb polling.
  Dumb polling is acceptable for v1 if SSE complicates the adapter.
- Slack feed shape: which surface (mentions/DMs/channels) lands first, and the
  consent/config grammar in vault config. Decide when WS5 starts.
